### 1. 准备 warmup 数据（启动前执行，替代旧的 build_feature_store）

#### 推荐：从本地 parquet_data 加载（秒级完成）

```bash
cd /home/yin/trading/ml_trading_bot

# 从 data/parquet_data/ 读取已有 1min 聚合数据（推荐，秒级完成）
bash live/scripts/prepare_warmup_ticks.sh highcap 6 --from-local
```

#### 备选：从 Binance 下载（首次无本地数据时）

```bash
# 下载最近 6 个月 monthly + 最近 1 个月 daily aggTrades
bash live/scripts/prepare_warmup_ticks.sh highcap 6
```

> **说明（monthly/daily 含义）**：
> - 这里的 *monthly / daily* 仅指 **下载的原始 aggTrades ZIP 粒度**；
> - 无论是从 monthly 还是 daily ZIP 来的，都会先在 `prepare_warmup_ticks.py` 里转换成 **1min 聚合数据**，再按 **交易日** 拆分；
> - 所以在实盘目录中，`live/highcap/data/ticks/ADAUSDT/`、`bars/ADAUSDT/` 看到的文件名统一是 `YYYY-MM-DD.parquet`，即“每天一个 1min 聚合文件”，**不存在按月落盘的 ticks/bars 文件**。

#### 补全缺失数据（--from-local 后补上最近几天）
- **产出目录**：`live/highcap/data/ticks/` 和 `live/highcap/data/bars/`
- **两种模式对比**：

| 模式 | 命令 | 速度 | 数据源 |
|------|------|------|--------|
| `--from-local` | 从 `data/parquet_data/` 读取 | 秒级 | 研究 pipeline 已 convert 好的 1min 聚合 |
| `--fill-gap` | 只补缺失的 daily | 分钟级 | Binance daily aggTrades（自动检测缺口） |
| 默认（下载） | 从 Binance 下载全量 | 10-30 分钟 | Binance monthly + daily aggTrades |

> **推荐流程**：先 `--from-local` 加载已有数据，再 `--fill-gap` 补全最近缺失的天数。

- **可指定币种**：

```bash
python live/scripts/prepare_warmup_ticks.py --universe highcap --months 6 --symbols BTCUSDT,ETHUSDT
python live/scripts/prepare_warmup_ticks.py --universe highcap --months 6 --from-local
```

#### 数据格式：研究 vs 实盘

| 对比项 | 研究 `data/parquet_data/` | 实盘 `live/highcap/data/ticks/` |
|--------|--------------------------|----------------------------------|
| 文件名 | `{SYM}_{YYYY-MM}.parquet`（按月） | `{SYM}/{YYYY-MM-DD}.parquet`（按天） |
| 列 | `timestamp, price, volume, side, symbol` | `timestamp, price, volume, side` |
| 时区 | tz-naive（隐式 UTC） | tz-aware UTC |
| symbol 列 | 有（`DataConverter` 从文件名追加） | 无（目录结构隐含） |

> **说明**：tz-naive 与 tz-aware 数值完全一样，只差时区标签，`tz_localize("UTC")` 可互转。
> `--from-local` 模式在边界自动处理：去掉 `symbol` 列 + 加 UTC 时区。
> 研究 pipeline 保持 tz-naive 不动，避免全量改动风险。

### 3. 实盘前信号对齐验证（研究回测 vs 事件驱动模拟）

> **为什么需要验证？** 研究流程和实盘有两条不同的特征计算路径：
> - **研究路径**：`train_strategy_pipeline.py` 批量计算特征 → `predictions.parquet` → `backtest_execution_layer.py`（向量化执行层回测）
> - **实盘路径**：`IncrementalFeatureComputer`（流式）→ `BPCLiveStrategy.decide()`（事件驱动）
>
> `simulate_bpc_e2e.py` 用实盘代码路径处理历史数据，等价于 backtrader/nautilus 事件驱动回测。
> 实盘前必须确认两条路径产生相同的信号。

#### 两种回测的区别

| | 研究执行层回测 | 事件驱动模拟（实盘路径） |
|---|---|---|
| **脚本** | `backtest_execution_layer.py` | `simulate_bpc_e2e.py` |
| **特征计算** | 批量预计算（predictions.parquet） | IncrementalFeatureComputer 流式计算 |
| **决策链路** | Gate → Entry Filter → Evidence → Tier → 执行模拟 | BPCLiveStrategy.decide()（与实盘同一套代码） |
| **用途** | 研究参数优化、交易效果验证 | 实盘前信号对齐验证 |
| **适用新 archetype** | 通用（读取 YAML 配置） | 仿造 simulate_bpc_e2e.py 写新脚本 |

#### 验证步骤

选取 OOS 重叠时间段，两边都导出信号 CSV，然后打开对比：

```bash
cd /home/yin/trading/ml_trading_bot

# A. 研究路径：执行层回测导出信号
python scripts/backtest_execution_layer.py \
  --logs results/train_final_<timestamp>/bpc/predictions.parquet \
  --strategy bpc --tiers \
  --quantile-train-start 2025-02-01 --quantile-train-end 2025-08-01 \
  --export-signals /tmp/signals_research.csv

# B. 实盘路径：事件驱动模拟导出信号
python scripts/simulate_bpc_e2e.py \
  --symbol BTCUSDT --days 30 \
  --export-signals /tmp/signals_live_sim.csv

# 对比两个 CSV：打开看入场次数、方向、evidence_score、tier 是否一致
```

**导出 CSV 列说明**：
- `direction`: 原始 bpc_breakout_direction（gate/filter 前）
- `entry_direction`: 最终入场方向（0 = 被 gate/filter 拒绝）
- `evidence_score`: Evidence 评分（仅通过的 bar）
- `tier`: 分档名称

> **判断标准**：同一时段的入场次数、方向基本一致即可。evidence_score 允许微小数值差异（不同特征计算路径的浮点误差）。

### 5. 启动 highcap universe 实盘（带依赖自检 + 质量闸门）

```bash
cd /home/yin/trading/ml_trading_bot
./live/scripts/start_live.sh highcap
```

- **内部流程摘要（策略B）**：
  - 解析 `universe = highcap`，从 `live/highcap/universe.yaml` 读取币种列表；
  - **加载 `live/binance_mainnet.env` 中的 API 密钥**；
  - 设置 `PYTHONPATH` 与一系列 `MLBOT_LIVE_*` 环境变量；
  - 调用 `check_dependencies.py` 做依赖自检（失败则直接退出 OFFLINE）；
  - 调用 `scripts/run_live.py`：
    - 初始化 `BPCLiveStrategy` 与多币种 `MultiSymbolManager`；
    - 加载 `live/highcap/data/ticks/` 中的历史 1min ticks（由 `prepare_warmup_ticks.sh` 预先准备）；
    - 通过 `on_bar()` 逐条喂入 `IncrementalFeatureComputer`，重建流式特征状态；
    - **Evidence Quantiles 计算**：从 6 个月 warmup 数据计算 9 个 evidence 特征的分位数阈值（`_compute_initial_quantiles()`）；
    - 启动 WebSocket 接收实时 ticks，开始累积实时数据；
    - **基于实时数据累积量判定模式**（不依赖 Feature Store）：
      - 阈值：≥240 bars (4h) → NORMAL，120-240 bars (2-4h) → DEGRADED，<120 bars → OFFLINE
    - NORMAL → 正常交易（约启动后 4h）；
    - DEGRADED → 仅观察不交易（2-4h 之间）；
    - OFFLINE → 继续接收数据，等待累积（<2h）。
