# 批量计算架构重构总结

## 概述
完成了实盘特征计算架构从流式计算到批量计算的重大重构，解决了长 lookback 窗口、VPIN 自适应桶算法、以及与研发流程一致性等问题。

## 核心变更

### 1. IncrementalFeatureComputer 重构
- **新增 `compute_features_batch()` 方法**：从磁盘批量计算特征，与研发流程完全一致
  - 读取 90+ 天 1min bars（支持 atr_percentile=540 等长窗口）
  - 读取 7 天 1min ticks（支持 VPIN 7 天滚动窗口自适应桶）
  - 重采样为 4h 时间框架进行特征计算
  - 返回最后一行特征作为当前特征

- **简化 `on_tick()` 方法**：
  - 仅维护 CVD 累积器和 tick 缓冲区
  - 不再触发任何特征计算

- **简化 `on_bar()` 方法**：
  - 仅维护 bar_buffer 和 CVD 指标
  - 不再调用 `_update_timeframe_features()` 进行流式特征计算

### 2. OrderFlowListener 重构
- **重写 `_compute_and_save_15min_features()`**：
  - 从磁盘读取历史数据（100天 1min bars + 8天 1min ticks）
  - 调用 `feature_computer.compute_features_batch()` 进行批量计算
  - 保存特征并传递给决策引擎

- **简化 `_restore_state()`**：
  - 只恢复 memory_window 和时间戳
  - 不再回放 ticks/bars 到 feature_computer
  - 避免 O(n²) 的数据回放性能问题

### 3. 配置文件调整
- **恢复 `run_live.py`**：MLBOT_LIVE_WARMUP_DAYS 设为 7 天（足够恢复内存窗口）
- **恢复 `prepare_warmup_ticks.sh`**：默认下载 6 个月数据（支持长 lookback 特征）

## 架构优势

### 1. 解决长 Lookback 问题
- **原问题**：流式计算无法支持 atr_percentile(window=540 bars × 4h = 90天) 等长窗口特征
- **解决方案**：批量计算直接从磁盘读取完整历史窗口，支持任意长度 lookback

### 2. 确保算法一致性
- **原问题**：实盘 VPIN 自适应桶算法与研发不一致
- **解决方案**：批量计算使用相同的 7 天滚动窗口算法，确保完全一致

### 3. 提升性能
- **原问题**：流式计算 O(n²) 复杂度，每次新数据重新计算全部特征
- **解决方案**：批量计算 O(n) 复杂度，充分利用向量化计算

### 4. 支持订单流特征
- **VPIN**：使用 7 天滚动平均成交量 × K 倍数的自适应桶算法
- **Trade Clustering**：基于批量数据进行聚类分析
- **Footprint**：基于批量数据生成足迹图表特征

## 数据流变化

### 重构前（流式计算）
```
Real-time Tick/Bar → Incremental Updates → Feature Calculation → Decision
```

### 重构后（批量计算 v1 - 纯磁盘）
```
Disk Data (90+ days bars + 7 days ticks) → Batch Computation → Features → Decision
Real-time Tick/Bar → Buffer Maintenance Only
```

### 优化后（批量计算 v2 - 磁盘+Buffer融合）
```
┌──────────────────────────────────────────┐
│ 磁盘数据 (历史主体)                        │
│  - bars_disk:  100天 1min bars           │
│  - ticks_disk: 8天 1min ticks            │
└──────────────────────────────────────────┘
               │
               ├─── Merge + Dedup ───┐
               │                       │
┌──────────────────────────────────────────┐     │
│ 内存Buffer (最新补充)                      │     │
│  - memory_window: 最近4h bars (已完成)    │ ────┤
│  - tick_buffer:   最近N分钟 ticks        │     │
└──────────────────────────────────────────┘     │
                                               ▼
                                    Batch Computation
                                               │
                                               ▼
                                          Features
                                               │
                                               ▼
                                           Decision

Real-time Tick/Bar → Buffer Maintenance + Merge Source
```

**关键优化点**:
1. 磁盘数据可能有1-2分钟延迟（最新bars还在内存未落盘）
2. 合并内存buffer确保计算用到最新数据
3. 按timestamp去重（bars keep='last'，ticks保留全部）
4. 不改变批量计算核心架构，只优化数据源

## 性能改进

1. **消除 O(n²) 回放**：warmup 不再回放大量历史数据
2. **减少内存占用**：只维护必要的缓冲区，不存储完整历史
3. **提高计算效率**：批量计算充分利用向量化操作
4. **确保一致性**：实盘与研发使用完全相同的算法
