## 一、为什么盈利归因 ≠ 失败归因（核心逻辑）
盈利归因回答的是：

“我靠什么市场失衡赚钱？”

这是结构性的，而且通常是互斥的：

换手 → 再启动（BPC）

不换手 → 持仓推动（SC-shallow）

加速 → 追逐失衡（ME）

失败 → 反向失衡（Reversal）

👉 这天然适合作为 Archetype 定义轴。

失败归因回答的是：

“这笔钱为什么没赚到 / 亏了？”

而在趋势系统里，失败原因往往高度重复：

流动性枯竭

反向力量突然介入

宏观事件 / regime 切换

假突破 / 动能耗尽

👉 不同 Archetype 往往死在同一件事上。

所以你直觉觉得：

“趋势的失败归因可能都一样”

这是完全正确的观察。

## 二、为什么 Archetype 不该按失败归因来拆？

如果你按失败归因拆 Archetype，会发生什么？

比如：

“假突破型”

“动能耗尽型”

“流动性反转型”

结果是：

同一笔行情

可能同时满足多个失败标签

Archetype 之间高度重叠

Router / risk netting 复杂度暴涨

👉 这是反工程的。

## 三、Reversal 为什么可以先合并？

你说：

reversal 目前三个合并一个也是失败归因

这在你现在阶段是完全正确的。

原因有三：

1️⃣ Reversal 的盈利归因是共通的

不管是：

趋势失败反转

假突破反打

动能衰竭反向

它们的核心都是：

“原有失衡失效，反向失衡接管”

盈利机制是一致的。

2️⃣ Reversal 的失败边界更重要

反转最难的不是“进”，而是：

什么时候 不该反

什么时候 必须快跑

这些规则：

在 Gate / Holding / PCM 层解决

不需要拆 Archetype

3️⃣ 工程复杂度最小化

你之前已经明确：

不想引入 router

不想拆太多 path

那 Reversal 合并是工程最优解。

我给你一个可落地的 blueprint，按你 pipeline 来。

① Archetype 层（只看盈利归因）

② Gate 层（主要是失败归因）

Gate 回答的永远是：

“在这种失败条件下，

比如：

流动性枯竭 → deny SC / ME

无换手 → deny BPC

HTF 未失效 → deny Reversal

👉 失败归因主要落在 Gate。

③ Feature 层（中性描述）

Feature 只描述市场状态，不带结论：

价格行为

成交结构

波动 / 流动性

HTF context

同一组 feature：

可以被多个 Archetype 使用

但解读方式不同

④ Evidence 层（盈利归因的“证据”）

Evidence 是：

“为什么我相信这次盈利机制会成立？”

BPC 的证据：换手 + 重置

SC 的证据：持仓不卖 + 推进

ME 的证据：扩散 + 加速

👉 Evidence 是 Archetype-specific 的。

⑤ Entry / Execution（技术实现）

尽量共享

不要在这里体现 Archetype 差异

差异应该在 Evidence 之前解决

⑥ Holding / PCM（失败归因落地）

这里是失败归因的第二主战场：

什么情况下：

快速止盈

直接认错

而这些规则：

往往可以跨 Archetype 复用
