## 一、当前架构概览

### 决策管线

```
Direction → Gate → Entry Filter → Evidence → Tier → Execution → PCM
```

- **Direction**: `bpc_breakout_direction` 结构性特征（+1/−1/0）
- **Gate**: 3 hard deny + 9 soft filter + 2 guardrail（规则 veto）
- **Entry Filter**: BB 压缩 OR 流动性沉默（OR 逻辑）
- **Evidence**: 9 特征 quantile mapping → 0~1 评分
- **Tier**: T1(≥0.55) / T2(≥0.40) / T3(≥0.25) / T4(<0.25)
- **Execution**: SL/TP/Size 按 Tier 分档 + Noise Penalty 动态调整
- **持仓管理 7 步**: Time Stop → Breakeven Lock → Water Mark → Activation Trailing → SL Hit → TP Hit → Execute Close

### 核心原则

- **纯规则决策，无 ML 模型依赖**
- **配置驱动**：所有参数 YAML 化
- **责任可分解**：每层只看各自 KPI
- **可回滚**：任何自由度可开关、可审计、可降级

## 二、为什么不用 NN 多头路径模型

### 核心结论

> **NN 四个头（dir/mfe/mae/mtt）已被更稳定的模块完全覆盖。**
> 在 Gate → Evidence → Tier → Noise Penalty 已成型的系统里，NN 不再是"必需品"，而是"复杂度噪声"。

### 逐头替代分析

| NN 头 | 原职责 | 替代者 | 替代方式 | 为什么替代者更好 |
|-------|--------|--------|----------|-----------------|
| **dir** | 方向预测 | **Gate** | `bpc_breakout_direction` 结构性特征 | Gate 已确认方向，NN 再学 dir = 用黑盒复述白盒确认过的事实 |
| **mfe** | 空间预测→仓位 | **Evidence + Tier** | `evidence_score → tier → size_multiplier` | MFE 的信息增量仅 5-10%，但需付出 NN 训练/漂移/监控成本 |
| **mae** | 路径风险→止损 | **Noise Penalty + SL** | 数学特征压缩 + execution.yaml 宪法 | 真正 MAE 大的单子已被 Gate/Evidence 杀掉，剩余方差极小 |
| **mtt** | 时间预测→持仓 | **Execution Budget** | 规则 `time_stop_bars`（FAST/NORMAL/SLOW） | MTT 需要的不是精度而是分档（6/16/40 bars），规则完全覆盖 |

### 详细理由

#### 1) dir 头：概念性错误

dir 头在当前系统里是"概念性错误"的存在：
- Gate 已经用 `bpc_breakout_direction` 解决了"方向是否成立"
- 再让 NN 学 dir = 用一个黑盒去复述一个白盒确认过的事实
- 如果 NN dir 和 Gate dir 冲突 → 要么 Gate 不该放，要么 NN 不该信

**结论：dir 头应当删除，而不是降权。**

#### 2) mfe 头：性价比极低

在结构 + 订单流 + regime 都齐全的 Evidence 下，MFE 的信息增量很小：
- 为了 5-10% 的执行微调
- 要付出：NN 训练 + 数据漂移监控 + 多头一致性检查 + 系统复杂度
- MFE 是"锦上添花"，不是"雪中送炭"

#### 3) mae 头：理论有意义，工程收益极低

MAE vs Noise 的区分在理论上正确（Noise = 市场状态，MAE = 单笔路径风险），但：
- 当前是 failure-first 系统，MAE 大的单子已在 Gate/Evidence 被杀掉
- 留下来的交易 MAE variance 已经很小
- 属于"理论上有意义，工程上收益极低"

#### 4) mtt 头：被 Execution Budget 击败

MTT 的真正语义不是"市场会走多久"，而是"我愿意等多久"：

```python
# 规则版完全覆盖 NN 的功能
if trend_regime and low_noise:
    mtt_tier = FAST    # 6 bars
elif mean_regime:
    mtt_tier = SLOW    # 40 bars
else:
    mtt_tier = NORMAL  # 16 bars
```

这段逻辑的含义是：**"我不预测市场，我约束我自己"** — 这是 NN 永远无法做得更好的事情。

### 为什么现在用 NN 反而是错的

1. **目标不是"更聪明"，而是"更稳"**：NN 带来的不确定性、漂移、debug 困难，对"行为一致性"目标是负贡献
2. **NN 在 execution 层是"非法公民"**：execution 层的铁律是"宁可粗糙可解释稳定，不要精细不稳漂移"
3. **NN 的边际贡献实际只有 5-15%**，主要体现在执行节奏而非 Sharpe 或胜率

### 什么时候 NN 才合法回归

三个条件缺一不可：
1. Evidence 无法解释 realized path（高 evidence → 经常小 mfe）
2. 样本规模 > 100k trades（不是 bars）
3. NN 只当 advisor（权重 < 20%，永不 override Gate / Execution 宪法）

## 三、为什么不用 Router

### 核心结论

> **在非平稳、对抗性的市场中，概率 Router 往往只是在"分摊错误"，
> 而基于语义正交的 Archetype + Gate 反证法，才是在"裁决博弈是否成立"。**

### Router 的致命问题

#### 1) 概率加权在"语义不唯一"时 ≈ 平均输钱

Router 在多 archetype 同时出现时输出概率（如 ME: 0.4 / BPC: 0.35 / Fade: 0.25），但：

- **市场状态不是可线性混合的**
  - ME 成立的世界里，BPC 是噪声
  - BPC 成立的世界里，Fade 是自杀
  - Fade 成立的世界里，ME 是最后一棒

- **概率 ≠ 信念强度**
  - Router 的 0.6 不是"我有 60% 把握"
  - 而是"在相似样本中，60% 属于这一类"
  - 系统真正需要的是"叙事有没有被市场否定"，这是逻辑判断，不是概率回归

- **多头 + 分仓 = 把错误延迟暴露**
  - 错误不会立刻爆炸，但会持续 bleed
  - Sharpe 还行，曲线平滑，但说不清在赚哪种钱

#### 2) Router 被三种方式"内化"

当前系统已用更稳定的方式覆盖了 Router 的全部功能：

| Router 原功能 | 替代方式 | 为什么更好 |
|--------------|---------|-----------|
| Archetype 分类 | **互斥规则**：趋势 vs 反转不共存 | 不允许线性混合，符合博弈本质 |
| 概率排序 | **人工优先级**：ME > BPC > FBF > LSR > AER | 基于市场机制的强先验，比 ML 更稳 |
| Trade-off 仲裁 | **Gate 反证法**：有一个关键反证就不做 | hard veto 比加权平均更稳 |

#### 3) 规则 Router 的替代逻辑

```
规则 Router 逻辑:
  趋势族（ME + BPC）可共存 → 优先 ME
  趋势 vs 反转 不可共存 → 不交易
  单 archetype 通过 Gate → 直接交易
  多 archetype 冲突 → 降仓 or 不交易
```

Gate 是 hard veto（反证法），不是 soft score。在噪声高、非平稳的市场里，**强 veto 比加权平均更稳**。

### 什么时候需要重新引入 Router

| 触发条件 | 说明 |
|---------|------|
| 同族 archetype 内部 RR 分布显著分化 | ME-A（低波动扩张）vs ME-B（高波动爆发），规则优先级不够用 |
| 跨市场复用，优先级不再稳定 | 不同币种的 archetype 表现分布不同 |
| 允许同时开多单 | "这个 bar 同时做 ME 和 SC 各 0.5R" |

## 四、为什么不用树模型直接做策略，而是做规则语义发现

### 核心结论

> **树模型 = 望远镜，不是方向盘。用树发现规律，不用树直接决策。**

### 树模型直接做策略的问题

#### 1) 特征太多，语义规则难以发现

量化系统面临的核心困境：
- 100+ 特征，人工无法穷举所有有意义的 if/else 组合
- 特征之间有复杂交互（如 `cvd_long <= -34895 AND bb_width <= 4.86`）
- 人类直觉能发现的规则数量有限（通常 < 20 条），但市场语义维度远不止这些

#### 2) 树模型（黑盒）在实盘的致命缺陷

```python
# 树模型直接做策略
# 实盘 Sharpe 从 1.5 降到 0.3
# 问题：不知道是哪个因素变了
可能原因:
  - 市场 regime 变化？
  - 某个特征失效？
  - 数据分布漂移？
→ 无法定位问题，只能"重新训练"
```

vs

```python
# 规则系统（可归因）
# 实盘 Sharpe 从 1.5 降到 0.3
# 分析每个规则的贡献
fbf_vp_skewness_confirms: Sharpe 0.8 → 0.2 ❌ 这个语义失效！
fbf_wick_exhaustion: Sharpe 0.5 → 0.4 ✅ 仍然 work
fbf_sr_distance: Sharpe 0.6 → 0.5 ✅ 基本稳定
→ 定位问题: "VP偏度"语义在当前市场失效
→ 解决方案: 暂时禁用或调整阈值
```

#### 3) 新标的迁移问题

- **树模型**：在 BTC 上训练的模型应用到 ETH → Sharpe 可能 1.5 变 −0.3，不知道哪些假设不成立
- **规则系统**：逐条检查 → `sr_distance` ✅ 通用 / `vp_skewness` ❌ ETH 不同 → 只调整 VP 规则阈值

### 正确用法：树模型做假设发现器

树模型的价值不在于直接预测，而在于**高效发现人类难以穷举的语义规则**：

```
树模型训练 → 导出高频分裂规则 → 人工审核语义 → 写入 Gate/Evidence YAML
```

具体流程：
1. **训练 Return Tree**（在 Gate 过滤后的样本上学习）
2. **导出 `tree_rules.md`**（高频分裂特征 + 阈值）
3. **人工审核**：
   - 负面规则（树发现的"坏样本特征"）→ **Gate deny**
   - 正面规则（树发现的"好样本特征"）→ **Evidence 评分轴**
4. **阈值转换**：绝对阈值 → 分位数（跨 symbol 泛化）
5. **plateau 验证**：确认阈值的鲁棒性

> **树模型规则 ≠ Execution 触发器**
> **树模型规则 = 语义证据 → Gate 的 veto / Evidence 的评分轴**

## 五、为什么不用强化学习（RL）

### 核心结论

> **RL 不属于 v1 的主系统，只能作为"影子研究"。真正进入主系统只可能在 v4（甚至 v5）。**

### RL 与系统哲学的根本矛盾

#### v1 的哲学：限制自由度 → 换生存概率

```
Path → Gate → Evidence → Execution → PCM
各层只看各自 KPI
```

#### RL 的哲学：探索自由度 → 换期望收益

```
Execution/Routing ← reward ← PCM Sharpe
```

**这是正面冲突的。**

### RL 在 v1 会破坏什么

#### 1) 责任可分解性

RL 一旦进主链路，会立刻失去：
- ❌ Execution 的局部可解释性
- ❌ Gate 的阈值稳定性
- ❌ Evidence 的 veto 语义清晰度
- **你会不知道 Sharpe 是谁干的**

#### 2) 安全边界

RL 会天然倾向于：
- 在 Gate 边界试探
- 在 Gate 缝隙中 exploit
- 在 execution cost 上赌博
- **RL 在 v1 会和 Safety 打架**

#### 3) 可回滚性

v2 的任务是"删掉不必要的恐惧"，需要可解释 / 可删 / 可回滚。RL 不满足任何一条。

### RL 的正确定位

| 版本 | RL 的角色 |
|------|----------|
| **v1** | 只做影子研究：Counterfactual policy search / Router 阈值压力测试 |
| **v2** | 仍然不能进主系统 |
| **v3** | 结构冻结后开始试探 |
| **v4** | 才允许成为一等公民（此时 action space / veto 边界 / reward proxy 全部稳定） |

### 一句话定性

> **RL 不是用来赚钱的，是用来证明"哪些自由度不能给"的。
> 当 RL 学到的策略和你 v2 剩下的 Execution 几乎一致、只是参数微调时，RL 才"毕业"。**

## 六、为什么不用 Mamba / Transformer 序列模型

### 核心结论

> **Mamba/Transformer 在理论上有优势（线性复杂度 / 长程依赖），
> 但在当前系统中属于"技术上优雅，系统上不划算"。**

### Mamba/Transformer 的理论优势

- **状态记忆**：内置状态机制，匹配市场记忆效应
- **长期依赖**：O(n) 复杂度处理超长序列
- **实时推理**：恒定时间步进，适合流式数据

### 为什么在当前系统中不采用

#### 1) 黑盒性与系统精神冲突

当前系统的核心精神是 **"可观测、可归因、可回滚"**。Mamba/Transformer 的输出：
- 不可解释（attention weights 不等于因果关系）
- 不可归因（Sharpe 下降时无法定位是哪个时序模式失效）
- 不可回滚（只能重新训练）

#### 2) 序列 embedding 的工程问题

- **模型套模型**：`dl_seq_features → tree/rule` = 信息瓶颈 + 调参维度膨胀 + 可解释性下降
- **尺度漂移**：序列 embedding 的分布比结构特征更容易 drift，跨 symbol 可比性差
- **归一化依赖**：需要逐维 rolling z-score / L2 normalize，增加工程复杂度

#### 3) 信息增量有限

在 BPC 结构特征（breakout direction / pullback / score）+ 订单流特征（CVD / VPIN / taker ratio）已经覆盖核心语义的情况下：
- Mamba 额外捕捉的"微观模式"IC 不稳定（跨时间段、跨 seed 波动大）
- 多数情况下，这些微观模式已被结构特征间接编码
- 只有在 IC 稳定被证明后，才有资格进入候选池

#### 4) 维护成本极高

| 维度 | 规则系统 | Mamba/Transformer |
|------|---------|-------------------|
| 失效定位 | 逐规则检查 Sharpe 贡献 | 重新训练 |
| 新标的迁移 | 调整阈值 | 重新训练 |
| 版本管理 | YAML diff | 模型权重版本 |
| 实盘 debug | 看哪条规则挡了 | "attention 在看哪里" |

#### 5) 对当前数据量不经济

- 序列模型需要大量数据才能泛化（通常 > 100k 样本）
- 当前交易频率下，有效 trade 数量远不够
- 在小样本下训练序列模型 → 高方差 → 不稳定

### 什么时候才值得引入

- Trade 数量 > 100k 且结构特征无法解释的 pattern 稳定存在
- 有独立的 validation pipeline 证明 IC 跨时段/跨 seed 稳定
- 只作为 **feature 增强器**（输出喂给规则系统），不作为决策器

## 七、树模型的正确角色：规则语义发现器

### 核心定位

```
树模型 = 高效的"假设发现器"
规则系统 = 可维护的"因果假设库"
规则导出 ≠ 复制 Sharpe，而是为了可归因性
```

### 解决的核心问题：特征太多，语义规则难以发现

#### 问题描述

- 系统有 100+ 特征，人工无法穷举所有有效的阈值组合
- 特征间存在复杂的非线性交互
- 哪些特征组合能区分"好交易"和"坏交易"？人类直觉有限

#### 树模型的解法

```
训练树模型 (在 Gate 过滤后的样本上)
    ↓
自动发现高频分裂规则
    ↓
例如: if cvd_long <= -34895 AND bb_width <= 4.86 → 坏交易
      if wpt_exhaustion_score > 0.55 → 好交易
    ↓
人工审核语义（这个规则是否有物理/市场机制解释？）
    ↓
负面规则 → Gate (hard deny / soft filter)
正面规则 → Evidence (评分轴)
    ↓
阈值转换为分位数 (跨 symbol 泛化)
    ↓
Plateau 验证稳健性
```

### 树模型 vs 规则系统的角色分工

| 角色 | 树模型 | 规则系统 |
|------|--------|---------|
| **本质** | 假设发现器 | 因果假设库 |
| **输入** | 100+ 特征 | 树模型导出的候选规则 |
| **输出** | 高频分裂特征 + 阈值 | YAML 配置 (gate.yaml / evidence.yaml) |
| **生命周期** | 离线，可重复训练 | 在线，稳定运行 |
| **失效时** | 重新训练发现新规则 | 逐条检查/禁用/调阈值 |

### 实际流程

1. **训练 Return Tree**：
   ```bash
   mlbot train final --config config/strategies/bpc \
     --features config/strategies/bpc/features_evidence.yaml \
     --labels config/strategies/bpc/labels_return_tree.yaml
   ```

2. **检查 feature_importance**：
   - Top 10 中 ≥7 个是 Evidence 特征 → Gate 已足够
   - Top 10 中 ≥5 个是 Gate 特征 → Gate 需要加强

3. **导出规则**：`tree_rules.md` + `risk_gate_draft.yaml` + `evidence_candidates.yaml`

4. **人工审核语义后写入正式配置**：
   - 树的负面规则 → **Gate deny**（不要让 NN 学这个，直接硬编码）
   - 树的正面规则 → **Evidence 评分轴**

## 八、平坦高原优化：解决规则阈值稳定性

### 核心问题

规则系统的阈值如何选择？
- 过紧 → trade_rate → 0（不交易）
- 过松 → 噪声交易多，Sharpe 下降
- 最优点 → 可能是"尖峰"，微小扰动就失效

### 平坦高原的核心理念

> **Plateau 不是用来找"最优"，而是用来证明"不脆弱"。**

```python
# 平坦高原搜索
目标 = Robustness Score = min(Sharpe across all buckets)
     ≠ max(Sharpe)

# 找到的不是"最好的阈值"，而是"最不敏感的阈值区间"
```

### Plateau vs 其他优化方法

| 特性 | 平坦高原搜索 | Optuna |
|------|------------|--------|
| **优化目标** | Robustness（min Sharpe across buckets） | 平均性能（mean Sharpe） |
| **适用场景** | Gate/Evidence 单阈值优化 | 模型超参数联合优化 |
| **可解释性** | 高（完整阈值-Sharpe 曲线） | 中（采样点） |
| **鲁棒性** | 高（最差情况最好） | 中（平均情况最好） |
| **参数空间** | 低维（1-3 维/规则） | 高维（10+ 维） |

### 三阶段策略

#### Stage 0：物理层冻结

**不该进 plateau 搜索的参数**（结构参数）：
- 是否启用某条规则
- 是否拆 TE / TC
- 是否启用 extreme contraction

#### Stage 1：锚点参数粗扫后冻结

离散阈值（adx_min / ma200_pos / extreme_percentile），如果 ±10% 改动导致 trade_rate 变化 > 30% → 冻结。

#### Stage 2：真正找 plateau（2~3 维）

只剩连续可微的阈值参数（如 sr_distance / sqs_min），验证"是否存在对阈值不敏感的稳定区间"。

### 工程判断标准

> **如果某个参数的 ±10% 改动导致 trade_rate 或 regime_rate 变化 > 30%，
> 它不是 plateau 参数，是 gate 参数，应冻结。**

### 正确的优化流程

```
1. 启发式/Optuna → 找"能跑的 anchor"（trade_rate > 下限，各 regime 有样本）
2. 冻结 80% 参数（结构 + gate）
3. 对剩余 2~3 个连续阈值做 plateau 搜索
4. Plateau KPI 不看 pnl，只看：稳定性 / switch_rate / extreme_contraction
5. 推荐值偏宽松侧（bias = 0.2 if width < 0.25 else 0.1）
```

### 协议规则

> **Plateau 搜索不允许同时调节超过 3 个连续阈值参数。
> 结构性开关与语义层参数必须先冻结。
> 否则 plateau 结果不具工程意义。**

## 九、启发式规则主导的各层设计

### 9.1 Archetype 选择：先验规则 Router

当前不用 ML Router，而是用**启发式优先级 + 互斥规则**：

```
ME > BPC > FBF > LSR > AER
趋势族可共存 → 优先 ME
趋势 vs 反转 不可共存 → 不交易
单 archetype 通过 → 直接交易
多 archetype 冲突 → 降仓 / 不交易
```

**为什么启发式更好**：
- 基于市场机制（加速流 > 结构延续 > 失败叙事），比 ML 更稳
- 可解释、可审计：为什么选了这个、为什么拒绝了那个
- 不会因为训练数据分布变化而漂移

### 9.2 Execution 层：规则宪法

Execution 层完全由规则驱动，不依赖任何 ML 预测：

```yaml
# Tier 参数（按 Evidence Score 分档）
T1: {sl_atr: 1.5, tp_atr: 3.0, size: 1.0}
T2: {sl_atr: 1.8, tp_atr: 2.5, size: 0.7}
T3: {sl_atr: 2.0, tp_atr: 2.0, size: 0.5}
T4: {sl_atr: 2.5, tp_atr: 1.5, size: 0.3}

# 持仓管理 7 步（全规则）
1. Time Stop → 超时未达预期 → 平仓
2. Breakeven Lock → 浮盈达标 → 止损移到成本
3. Water Mark → 记录最高/最低浮盈
4. Activation Trailing → 浮盈达到激活线 → 启动追踪止损
5. SL Hit → 触发止损
6. TP Hit → 触发止盈
7. Execute Close → 执行平仓
```

**为什么不用 ML 做 execution**：
- Execution 层的铁律：宁可粗糙/可解释/稳定，不要精细/不稳/漂移
- 分档（离散）比连续函数更容易校准和回测
- 每档独立验证，问题定位清晰

### 9.3 PCM（Portfolio Capital Management）：固定规则

PCM 负责的是"是否允许"的决策，完全由规则驱动：

- **slot rotation**：max_slots = 2（硬编码）
- **add-on 许可**：是否允许加仓
- **portfolio 风险预算**：总风险限额

> **Execution 是单兵战术，PCM 是战役指挥。** PCM 不管 entry/exit 细节，只管生存。

### 9.4 为什么启发式规则比 ML 更适合这些层

| 维度 | 启发式规则 | ML 模型 |
|------|-----------|--------|
| **可观测性** | 每条规则的触发/否决可记录 | 输出一个数字，无法解释来源 |
| **失效模式** | 逐条规则可禁用/调阈值 | 只能重新训练 |
| **迭代成本** | 改一行 YAML，10 秒生效 | 训练 → 验证 → 部署，几小时到几天 |
| **跨标的迁移** | 逐条检查规则适用性 | 重新训练 |
| **极端行情** | 自动收缩（hard veto 生效） | 可能 OOD → 输出异常 |

## 十、总结：为什么纯规则架构是更高级的选择

### 系统成熟度的标志

> **不是"退回简单系统"，而是从"研究系统"毕业到了"交易系统"。**

研究系统崇尚：
- NN、连续输出、精细预测

交易系统崇尚：
- 宪法、预算、约束、可预期行为

### 各被淘汰方案的一句话总结

| 方案 | 淘汰原因 |
|------|---------|
| **NN 多头模型** | dir/mfe/mae/mtt 四个头全被更稳定的模块覆盖，NN 在 execution 层是"非法公民" |
| **Router** | 概率加权 ≈ 分摊错误，Gate 反证法 + 互斥规则 + 优先级更稳 |
| **树模型策略** | 黑盒不可归因，但树模型作为"假设发现器"极有价值 |
| **强化学习** | 与"限制自由度换生存"的 v1 哲学正面冲突，v4 才允许 |
| **Mamba/Transformer** | 黑盒 + IC 不稳定 + 维护成本高 + 数据量不够，当前是"技术优雅，系统不划算" |

### 当前架构的五大优势

1. **可归因**：Sharpe 下降时，能精确定位到哪条规则/哪个语义失效
2. **可回滚**：任何规则可一键关闭/调阈值/降级
3. **可迁移**：新标的只需逐条检查规则适用性，不需重新训练
4. **可审计**：每笔交易的决策路径完全透明（哪条 Gate 通过/否决、Evidence 多少分、哪个 Tier）
5. **反脆弱**：极端行情时 hard veto 自动收缩，不存在 OOD → 异常输出的问题

### 一句话哲学

> **当一个系统已经能用「结构 + 订单流 + regime + execution 宪法」把
> WHAT / IF / WHO / HOW CAREFULLY 全部覆盖时，
> NN/Router/RL 不再是能力增量，而是复杂度噪声。**
