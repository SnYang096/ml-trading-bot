# 为什么需要研究框架而非纯规则策略直接交易

> **核心论点：纯规则策略的问题不是"规则不好"，而是"规则不知道自己什么时候会死"。**
> 研究框架的价值不在于生成更多规则，而在于**证明假设、监控假设、在假设失效时安全退出**。

---

## 一、本项目中的纯规则策略实例

### 1.1 `rule_based_strategies/` 目录现状

项目中存在三套纯规则策略，它们代表了不同阶段的"纯规则"尝试：

| 策略 | 文件 | 核心逻辑 | 特征数 | 参数数 |
|------|------|---------|--------|--------|
| **SR Fake Breakout** | `bt_sr_fake_breakout/` | Backtrader + POC + 订单流确认 + 结构止损 | ~10 | 15+ |
| **Intraday Sniper** | `intraday_sniper/` | Nautilus + 多维压缩 + CVD + 突破评分 + Optuna | ~20 | 30+ |
| **SR Reversal Rule** | `sr_reversal_rule_strategy.py` | 纯特征管线 + SR 信号 + RR 标签 | ~50 | 10+ |

### 1.2 纯规则策略的典型代码结构

以 `intraday_sniper` 为例，一个纯规则策略包含：

```
strategy.py      - 1229 行主策略（if/else 决策树）
config.yaml      - 83 行参数（阈值、权重、窗口）
optimize.py      - 501 行 Optuna 参数搜索
indicators/      - 压缩、CVD、评分器
filters/         - 时间过滤、事件过滤
risk/            - 仓位管理、断路器
```

**关键问题**：`optimize.py` 用 Optuna 搜索 **18 个参数**：

```python
params = {
    "risk_per_trade": trial.suggest_float(..., 0.01, 0.1),
    "target_r_ratio": trial.suggest_float(..., 1.0, 5.0),
    "min_breakout_score": trial.suggest_float(..., 0.01, 1.0),
    "atr_period": trial.suggest_int(..., 10, 20),
    "trailing_stop_atr_mult": trial.suggest_float(..., 1.0, 4.0),
    "compression_window": trial.suggest_int(..., 15, 50),
    "compression_confidence_threshold": trial.suggest_float(..., 0.001, 0.3),
    # ... 共 18 个参数
}
```

当参数超过 8-10 个时，Optuna 的性能急剧下降（基本是随机搜索）。

### 1.3 `bt_sr_fake_breakout` 的订单流确认

```python
class BaseStrategy(bt.Strategy):
    params = dict(
        orderflow_weights=dict(
            vol_spike=1.0, cumdelta=1.5, taker=1.0, vwap=0.8, atr_delta=1.0
        ),
        orderflow_window=200,
        orderflow_percentile=0.65,
        orderflow_min_threshold=0.4,
    )
```

**5 个权重 × 3 个动态阈值参数 = 8 个需要调的"旋钮"**，每个都是人工拍脑袋。

## 二、纯规则策略的五大致命问题

### 2.1 不知道为什么赚钱

```python
# intraday_sniper 的突破评分
score = (0.15 * price_breakout +
         0.15 * volume_spike +
         0.10 * cvd_momentum_strong +
         0.05 * cvd_momentum_weak +
         0.075 * volume_and_cvd_aligned +
         0.05 * order_absorption_bar +
         0.10 * pre_breakout_silence +
         ...)  # 15 个权重

if score > min_breakout_score:
    enter_trade()
```

**问题**：
- 这 15 个权重是怎么来的？人工拍的
- 为什么 `volume_spike = 0.15` 而不是 `0.25`？没有理论依据
- 如果 Sharpe 从 1.5 降到 0.3，**不知道是哪个权重出了问题**
- 只能"全部重新优化"（本质上是重新拍脑袋）

### 2.2 不知道什么时候会失效

纯规则策略**没有假设的概念**。它只有：
- **输入**：特征值
- **输出**：交易信号
- **中间**：一堆 if/else

```
纯规则策略的思维模型：
  特征 → 规则 → 信号 → 赚钱/亏钱

研究框架的思维模型：
  假设 → 规则（假设的实现）→ 信号 → 验证假设是否成立 → 假设失效时退出
```

**关键区别**：纯规则策略在"亏钱"之后只能说"参数不对了"，研究框架可以说"这个假设不成立了"。

### 2.3 参数维度爆炸

| 参数数量 | Optuna 效果 | 可解释性 |
|---------|------------|---------|
| 2-6 | 很好 | 高（人可以理解） |
| 6-10 | 下降明显 | 中（还能手动验证） |
| >10 | 几乎是蒙的 | 低（参数之间互相干扰） |

`intraday_sniper` 有 **18 个参数**，已经超出了 Optuna 的有效范围。

**结果**：
- 参数空间巨大 → Optuna 找不到好解
- 参数之间强耦合 → 改一个影响所有
- 信号被互相抵消 → "不容易开仓"
- 实盘表现敏感 → 过拟合风险极高

### 2.4 规则之间互相打架

当规则超过 20 条时：

```
规则 A：CVD > 阈值 → 买入
规则 B：Volume < 阈值 → 不买
规则 C：Compression > 阈值 → 买入

场景：CVD 高、Volume 低、Compression 高
→ A 说买、B 说不买、C 说买
→ 谁赢？取决于代码顺序（if/else 的先后）
→ 这不是"智能决策"，是"先到先得"
```

**`bt_sr_fake_breakout` 的实际代码**就有这个问题：订单流确认 + 压缩检测 + 结构止损三套逻辑，优先级靠代码执行顺序决定。

### 2.5 跨标的迁移困难

```
在 BTC 上调好的参数：
  orderflow_percentile = 0.65
  orderflow_min_threshold = 0.4
  compression_threshold = 0.3

迁移到 ETH：
  - BTC 的 CVD 绝对值和 ETH 差 10 倍
  - ETH 的波动率分布完全不同
  - 阈值 0.65 在 ETH 上可能对应 0.3
  → 所有参数都要重新调
  → 本质上是重新开发一个策略
```

## 三、研究框架解决了什么

### 3.1 从"规则"到"假设"的思维升级

| 维度 | 纯规则策略 | 研究框架 |
|------|-----------|---------|
| **核心单元** | if/else 规则 | 可证伪的假设 |
| **失效诊断** | 整体 Sharpe 下降 → "参数不对" | 逐条假设检查 → "这个假设不成立了" |
| **修复方式** | 全部重新优化 | 禁用失效假设，保留有效假设 |
| **迁移方式** | 所有参数重调 | 逐条假设检查适用性 |
| **审计方式** | 无法审计 | 每笔交易可追溯到具体假设 |

### 3.2 BPC 研究框架的实际流程

```
Phase 0: 冻结假设
├── 7 条 BPC 假设（每条都有物理/行为学解释）
├── 如："突破前必须有压缩 → 否则是假突破"
└── 如："回踩时 CVD 不跟随 → 买方在底部接盘"
    ↓
Phase 1: 树模型发现规则
├── 训练 Return Tree（在全样本上学习）
├── 导出高频分裂规则
└── 人工审核：这条规则对应哪个假设？
    ↓
Phase 2: 稳定性检验
├── 时间切片：规则在不同时期都显著吗？
├── 阈值扰动：±20% 后结果变化大吗？
└── 分级：veto_hard / veto_soft / discard
    ↓
Phase 3: Plateau 验证
├── 阈值是否存在"对参数不敏感的稳定区间"？
├── KPI 不看 PnL，只看稳定性
└── 推荐值偏宽松侧（错误成本不对称）
    ↓
Phase 4: 写回配置（YAML）
├── 负面规则 → Gate deny
├── 正面规则 → Evidence 评分轴
└── 所有阈值用分位数（跨 symbol 泛化）
    ↓
Phase 5: 实盘监控
├── 每条规则的命中率、RR 分布
├── IC 滚动监控
└── 退化时自动降级
```

### 3.3 关键区别：假设可以被"证伪"

**纯规则策略**：
```python
# 这是一条规则，不是假设
if cvd_change_5 > 4805 and volume_ratio > 1.87:
    signal = True
# 失效时：不知道是 CVD 阈值错了还是 Volume 阈值错了
```

**研究框架**：
```yaml
# 这是一个可证伪的假设
hypothesis_4_control_not_transferred:
  description: "回踩期间控制权未从买方转移到卖方"
  features: [cvd_slope_pullback, vpin_shift, delta_during_pullback]
  negative_rule: "delta_during_pullback < -0.5"  # 如果卖方主导 → veto
  stability: time_stable + perturb_stable
  veto_level: hard
  
  # 验证记录
  last_validated: "2026-01-29"
  train_period_accuracy: 90%
  test_period_accuracy: 85%
  verdict: "假设成立，但 OFCI 具体算法是统计构造（Level 2-3）"
```

**当假设失效时**：
- 能精确说出"假设 4（控制权转移）不成立了"
- 可以只禁用这一条，其他假设继续工作
- 有记录可以追溯什么时候开始失效

## 四、纯规则策略 vs 研究框架的对比

### 4.1 以 `intraday_sniper` 为例

| 维度 | intraday_sniper（纯规则） | BPC 研究框架 |
|------|-------------------------|-------------|
| **参数管理** | 18 个参数 Optuna 搜索 | 3 hard deny + 9 evidence（每个单独 plateau 验证） |
| **失效诊断** | 重新跑 Optuna | 逐条规则 IC 监控 |
| **阈值来源** | Optuna 最优解 | Plateau 稳健区间（最差情况最好） |
| **优化目标** | maximize score（一个数字） | Robustness（min Sharpe across buckets） |
| **规则优先级** | 代码执行顺序 | Gate > Entry > Evidence > Execution（宪法层级） |
| **迁移方式** | 所有参数重新搜索 | 逐条假设检查 + 分位数阈值自动适应 |
| **代码量** | 1229 行策略 + 501 行优化 | 配置 YAML + 研究脚本（策略代码 < 300 行） |

### 4.2 核心差异图

```
纯规则策略的生命周期：
  开发 → 回测 → 上线 → 赚钱 → 亏钱 → 重新开发
  （循环，每次从头开始）

研究框架的生命周期：
  假设 → 验证 → 上线 → 监控 → 假设失效 → 只替换失效部分
  （渐进式，知识积累不丢失）
```

## 五、研究框架的三个核心能力

### 5.1 证明假设（Hypothesis Validation）

**纯规则策略不做这件事**。它只做"回测赚钱了 → 上线"。

研究框架要回答：
1. **这条规则背后的假设是什么？**（物理机制 / 行为偏差 / 纯统计）
2. **假设在 OOS 上是否成立？**（不是 holdout，是完全未见过的时间段）
3. **假设在不同 symbol 上是否成立？**（跨市场验证）
4. **假设对参数敏感吗？**（plateau 验证）

具体工具：
- `scripts/extract_negative_rules.py` — 从树模型导出负面规则
- `scripts/optimize_evidence_plateau.py` — Evidence 阈值 plateau 验证
- `scripts/optimize_entry_filter_plateau.py` — Entry Filter 阈值 plateau 验证
- `src/time_series_model/strategies/labels/archetype_audit_labels.py` — 假设审计

### 5.2 监控假设（Hypothesis Monitoring）

**纯规则策略的监控**：只看整体 PnL / Sharpe。

**研究框架的监控**：

```yaml
# 每条规则独立监控
gate_rules:
  - name: direction_crowding_veto
    metric: veto_rate
    baseline: 15%
    alert_threshold: ±50%  # 如果否决率突然翻倍或归零
    
evidence_features:
  - name: fbf_vp_skewness_confirms
    metric: rolling_IC_6m
    baseline: 0.03
    alert_threshold: < 0.01  # IC 降到 0.01 以下
    
entry_filters:
  - name: bb_compression
    metric: pass_then_sharpe
    baseline: 0.25
    alert_threshold: < 0  # 通过 filter 的交易 Sharpe < 0
```

**当某个规则的 IC 持续下降**：
```
IC 下降 → 触发诊断 → 
  1. 是 regime 变化？→ 等待 regime 恢复
  2. 是特征失效？→ 禁用该特征
  3. 是整体 archetype 失效？→ 暂停交易
```

### 5.3 安全退出（Graceful Degradation）

**纯规则策略的退出**：关掉整个策略。

**研究框架的退出**：

```
Level 1：关闭单条 Evidence 特征
  → Evidence 从 9 条变 8 条，系统继续运行
  
Level 2：关闭整个 Evidence 层
  → 只用 Gate + Entry Filter + Execution（固定参数）
  → Sharpe 降低，但系统仍然安全
  
Level 3：收紧 Gate 到只有 Hard Deny
  → 大幅减少交易，只做最确定的
  
Level 4：暂停 archetype
  → 不交易，只做影子模式监控
```

**这就是"可降级原则"**：任何组件都可以关闭，系统退化但不死。

纯规则策略做不到这一点，因为它的规则是**互相耦合**的 —— 关掉一条，整个评分函数就变了。

## 六、具体对比：为什么 `rule_based_strategies` 不能直接交易

### 6.1 `bt_sr_fake_breakout` 的问题

```python
# 订单流确认：5 个权重 + 动态阈值
orderflow_score = (
    w_vol * vol_score +
    w_cumdelta * cumdelta_score +
    w_taker * taker_score +
    w_vwap * vwap_score +
    w_atr * atr_score
) / total_weight

if orderflow_score > dynamic_threshold:
    confirm = True
```

**问题清单**：
- 5 个权重是拍脑袋的 → 没有假设支撑
- `dynamic_threshold` 用 percentile 计算 → 但 percentile 窗口（200 bars）也是拍脑袋
- 如果 3 个月后 Sharpe 下降：
  - 是 vol_spike 权重太高？
  - 是 percentile 窗口太长？
  - 是 cumdelta 这个特征本身失效了？
  - **无法回答，只能重新调参**

### 6.2 `intraday_sniper` 的问题

```python
# Optuna 优化目标
score = profit_score + trade_score + efficiency_score
if net_profit > 50: score += 50
elif net_profit > 20: score += 20
```

**评分函数本身就是一个需要验证的假设**：
- 为什么 `net_profit > 50` 奖励 50 分？
- 为什么 `trade_score = min(total_trades, 20) * 5`？
- 这个评分函数对 BTC 有效，对 ETH 也有效吗？

研究框架不会这样做。它会：
1. 定义明确的 KPI：snotio（Sharpe × (1 - Noise) × (1 - Timeout) × Info Ratio × (1 - Overlap)）
2. 每个 KPI 分量独立可监控
3. KPI 本身也要 plateau 验证

### 6.3 `sr_reversal_rule_strategy` 的启示

这是三个策略中**最接近研究框架**的一个：

```python
# 使用了特征管线、标签系统、RR 计算
from src.time_series_model.strategies.labels.sr_reversal_label import (
    SRSignalConfig, _generate_sr_reversal_signals,
)
from src.time_series_model.pipeline.training.label_utils import (
    compute_rr_label, compute_adaptive_rr_label_with_future_vol,
)
```

它已经：
- ✅ 使用了特征管线（而不是自己计算特征）
- ✅ 使用了标签系统（而不是直接 if/else）
- ✅ 使用了 RR 标签计算（有明确的 exit 逻辑）

但仍然缺少：
- ❌ 假设的显式定义
- ❌ 稳定性检验（时间切片 + 阈值扰动）
- ❌ Plateau 验证
- ❌ 逐条规则的 IC 监控
- ❌ 降级方案

## 七、研究框架的工程化实现

### 7.1 假设 → 配置（YAML 化）

```yaml
# gate.yaml
rules:
  - name: direction_crowding_veto
    hypothesis: "拥挤→反转风险"
    causal_level: 2  # 行为偏差型
    feature: ofci_pct
    condition: "> 0.85"
    action: deny
    
    # 验证记录
    stability:
      time_stable: true
      perturb_stable: true
      veto_level: hard
    plateau:
      width: 0.15
      recommended: 0.85
      confidence: medium
    monitoring:
      metric: veto_rate
      baseline: 12%
      alert_if: "> 25% OR < 5%"
```

### 7.2 树模型 = 假设发现器

```
树模型训练 → 导出规则 → 人工审核语义 → 写入 YAML → plateau 验证 → 实盘监控
```

树模型不直接做策略，而是：
1. 发现人类难以穷举的特征组合
2. 导出候选规则
3. 人工判断因果性 level
4. 通过 plateau 验证鲁棒性
5. 写入 Gate/Evidence 配置

### 7.3 Plateau = 假设的鲁棒性验证

```
Plateau 不问：这个阈值是最优的吗？
Plateau 问：   这个阈值的邻域是否都能 work？

纯规则策略用 Optuna 找 argmax（最优解） → 过拟合
研究框架用 Plateau 找稳健区间（最差情况最好）→ 鲁棒
```

### 7.4 监控 = 假设的持续验证

```
每日/每周自动运行：
1. 每条 Gate 规则的否决率 → 是否偏离 baseline
2. 每条 Evidence 特征的 IC → 是否衰减
3. 每个 Entry Filter 的通过率 + 通过后 Sharpe
4. 整体 Sharpe 的滚动窗口 → 是否低于 2σ

触发报警 → 人工介入 → 禁用/调整/等待
```

## 八、什么时候纯规则策略是合适的

纯规则策略不是完全没用。它适合：

| 场景 | 原因 |
|------|------|
| **快速验证想法** | 花 2 小时写一个 if/else 看看是否有潜力 |
| **Baseline 对照** | 和研究框架的结果做对比 |
| **Hard 风控闸门** | "账户亏 5% → 全部平仓"这种不需要假设 |
| **极简 fallback** | 研究框架全部失效时的兜底策略 |

但**不适合**作为主力交易系统，因为：
- 参数维护成本随规则数量指数增长
- 无法定位失效原因
- 无法安全降级
- 跨标的迁移基本等于重新开发

## 九、总结

### 一句话版本

> **纯规则策略是"我觉得这样赚钱"，研究框架是"我证明了这样赚钱，而且我知道什么时候不再赚钱"。**

### 研究框架的三条铁律

1. **每条规则必须对应一个可证伪的假设** — 否则就是拍脑袋
2. **每个假设必须有独立的监控指标** — 否则失效时无法定位
3. **每个假设失效时系统必须能安全降级** — 否则一条规则死 → 整个系统死

### 从 `rule_based_strategies` 到 BPC 研究框架的进化路径

```
Stage 0: 纯 if/else（bt_sr_fake_breakout）
  → 问题：不知道为什么赚钱

Stage 1: if/else + Optuna（intraday_sniper）
  → 问题：参数爆炸，过拟合

Stage 2: 特征管线 + 标签系统（sr_reversal_rule_strategy）
  → 问题：仍然没有假设的概念

Stage 3: 假设驱动 + 树模型发现 + Plateau 验证 + 监控（BPC 研究框架）
  → 解决：可证伪、可归因、可降级、可迁移
```

### 研究框架 vs 纯规则策略的终极对比

| 维度 | 纯规则策略 | 研究框架 |
|------|-----------|---------|
| **哲学** | "我能预测市场" | "我不能预测市场，但我能约束风险" |
| **寿命** | 一次性（参数过期 → 报废） | 渐进式（假设失效 → 替换，框架不变） |
| **失败模式** | 整体崩溃 | 局部退化 |
| **知识积累** | 无（每次从头开始） | 有（假设库持续积累） |
| **人的角色** | 调参工程师 | 假设审计员 |

> **一个成熟系统的标志不是"赚多少钱"，**
> **而是"失败时能多快定位到原因、多安全地退出"。**

## 七、实盘血泪：研究框架 + 监控系统 vs 纯规则 vs 黑盒模型

> **曾经跑了一年实盘，大赚的时候想调参，亏损的时候想调参，运维异常又想调参。**
> **研究框架会告诉你：没有最好的参数，不能被多赚钱的欲望驱使。**

实盘是检验系统设计哲学的唯一战场。以下对比三种系统在真实场景下的反应差异。

### 7.1 场景一：连续盈利，你想加仓 / 调参

| | 纯规则策略 | 黑盒 ML 模型 | 研究框架 + 监控 |
|---|---|---|---|
| **你的冲动** | "赚了这么多，参数肯定是对的，加大仓位！" | "模型在赚钱，再加几个特征让它更强！" | — |
| **系统反应** | 无任何机制阻止你调参 | 无法判断盈利来自 edge 还是噪声 | 监控系统自动问：盈利是否来自假设覆盖的路径？ |
| **后果** | 改了参数 → 破坏原有平衡 → 下轮行情反转时亏更多 | 加了特征 → 过拟合 → Sharpe 衰减 | Evidence 分层归因告诉你：盈利集中在 Tier 1（高置信）→ 不需要改；还是集中在 Tier 4（低置信）→ 是运气，不能加码 |
| **本质差异** | 无归因能力，只看总 PnL | 有归因但不可解释，只有 feature importance | 分层归因：知道是 Gate 拦住了坏交易，还是 Evidence 选对了好交易 |

### 7.2 场景二：连续亏损，你想调参 / 停机

| | 纯规则策略 | 黑盒 ML 模型 | 研究框架 + 监控 |
|---|---|---|---|
| **你的冲动** | "肯定是参数过期了，赶紧重新优化！" | "重新训练！换模型！加数据！" | — |
| **系统反应** | 无法区分"参数过期"和"市场暂时不适合" | 重训后无法确认新模型是否真更好 | 监控逐层诊断：Gate 通过率变了？Entry Filter 拒绝率变了？Evidence 分布漂移了？ |
| **诊断路径** | 只能看"赚没赚"，无法定位原因 | 只能看"预测准不准"，模型内部是黑盒 | 精确定位："亏损集中在 ETHUSDT + Tier 3/4 + Entry Filter 通过率下降 30%" |
| **行动** | 盲目调参（可能调对，更可能调错） | 盲目重训（50% 概率更差） | 明确行动：bb_compression 阈值在低波动环境失效 → 回实验目录跑 plateau 扫描 → 确认新阈值 → 只改一个参数 |
| **本质差异** | 修不如换 — 每次推倒重来 | 修不知修哪里 — 黑盒无法局部修复 | 精准手术 — 哪个组件出问题修哪个，其余不动 |

### 7.3 场景三：运维异常（数据中断 / API 超时 / 特征缺失）

| | 纯规则策略 | 黑盒 ML 模型 | 研究框架 + 监控 |
|---|---|---|---|
| **异常表现** | 特征出 NaN → if/else 走错分支 → 开了不该开的仓 | 模型收到脏数据 → 输出垃圾预测 → 可能反向开仓 | — |
| **系统防御** | 无（除非每个 if 都手动加 NaN 检查） | 无（模型对异常输入没有免疫力） | 可降级设计：特征缺失 → 该层自动关闭 → 系统退化到更低层级但不死 |
| **恢复能力** | 人工发现 → 人工停机 → 人工检查 → 人工重启 | 同上，且更难确认模型状态是否干净 | 自动降级 → 数据恢复后监控确认特征分布正常 → 自动恢复全功能 |
| **本质差异** | 脆弱 — 任何异常都可能致命 | 更脆弱 — 黑盒对脏数据没有概念 | 反脆弱 — 异常是系统设计的一部分 |

### 7.4 场景四：市场 Regime 切换（趋势 → 震荡 / 低波 → 高波）

| | 纯规则策略 | 黑盒 ML 模型 | 研究框架 + 监控 |
|---|---|---|---|
| **感知能力** | 无法感知（规则写死的） | 可能感知（如果训练数据包含了），但你不知道它感知到了什么 | 显式感知：波动率 regime、流动性状态、特征分布漂移全部有监控 |
| **适应方式** | 等亏够了，人工判断"市场变了"，手动换策略 | 等预测准确率下降，触发重训 | Gate 的 vol_compression 自动拒绝低波动假突破；Evidence 分布漂移告警提前预警 |
| **响应速度** | 滞后（人工判断至少 1-2 周） | 滞后（重训至少几天） | 近实时：每根 bar 都在重新评估 |
| **本质差异** | 被动挨打 | 延迟适应 | 主动防御 |

### 7.5 场景五：你不在（出差 / 休假 / 睡觉）

| | 纯规则策略 | 黑盒 ML 模型 | 研究框架 + 监控 |
|---|---|---|---|
| **无人值守** | 按写死的规则继续跑 → 市场变了没人改参数 → 持续亏损 | 模型继续预测 → 分布漂移了没人重训 → 渐进恶化 | 宪法级硬约束自动生效：daily -4% / weekly -8% / monthly -12% → 自动停机保命 |
| **你回来后** | 看到一堆亏损，不知从哪查起 | 看到模型变傻了，不知是数据问题还是模型问题 | 看到分层监控报告：Gate 通过率、Evidence tier 分布、Entry Filter 命中率、per-symbol 归因 → 5 分钟定位问题 |
| **本质差异** | 需要你永远在场 | 需要你定期重训 | 系统能独立生存，你只需周度审计 |

### 7.6 总结：实盘中三种系统的本质差距

```
纯规则策略的实盘困境：
  赚了 → 不知道为什么赚 → 想加码 → 加错了
  亏了 → 不知道为什么亏 → 想调参 → 调错了
  异常 → 系统崩溃 → 人工救火
  不在 → 没人管 → 持续恶化

黑盒 ML 的实盘困境：
  赚了 → 不知道赚的可持续吗 → 不敢加
  亏了 → 不知道哪里坏了 → 只能整体重训
  异常 → 模型吃脏数据 → 输出垃圾
  不在 → 没人重训 → 模型腐烂

研究框架 + 监控的实盘姿态：
  赚了 → 分层归因确认来源 → 知道能不能持续
  亏了 → 逐层诊断定位组件 → 精准修复
  异常 → 自动降级保命 → 数据恢复后自动恢复
  不在 → 宪法硬约束自动生效 → 回来后 5 分钟定位
```

> **实盘的终极问题不是"你的策略赚不赚钱"，**
> **而是"当它不赚钱的时候，你能多快知道、多准确定位、多安全退出"。**
>
> 纯规则和黑盒都回答不了这个问题。
> 只有研究框架 + 监控系统，才能让你**睡着觉跑实盘**。
