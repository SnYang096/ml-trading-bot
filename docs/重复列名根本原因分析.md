# 重复列名根本原因分析

## 问题链条

### 1. 特征设计层面：多个特征输出相同的列名

**根本原因**：在 `config/feature_dependencies.yaml` 中，多个特征都输出了相同的列名：

```yaml
sqs_hal_high:
  output_columns: ["sqs_hal_high", "hal_high", "poc"]  # 输出 poc 列

sqs_hal_low:
  output_columns: ["sqs_hal_low", "hal_low", "poc"]    # 也输出 poc 列
```

**为什么会有这个问题？**
- `sqs_hal_high` 和 `sqs_hal_low` 都调用了 `add_poc_hal_dimensionless_features()` 函数
- 这个函数会创建 `poc` 列作为副产品
- 两个特征都声明 `poc` 为输出列，导致重复

### 2. 特征合并逻辑：没有正确处理已存在的列

**问题代码位置**：`src/features/loader/parallel_computer.py` 第 1034-1038 行

```python
new_cols = [
    c for c in combined_result.columns if c not in result_df.columns
]
if new_cols:
    result_df = pd.concat([result_df, combined_result[new_cols]], axis=1)
```

**问题分析**：
- 这个逻辑只添加**新列**（不在 `result_df` 中的列）
- 但如果 `combined_result` 中有重复列名（比如多个 `poc`），`combined_result.columns` 本身就有重复
- 当使用 `combined_result[new_cols]` 时，如果 `new_cols` 中有重复，或者 `combined_result` 的列有重复，就会产生问题

**更严重的问题**：
- 如果 `sqs_hal_high` 先被计算，`result_df` 中已经有了 `poc` 列
- 当 `sqs_hal_low` 被计算时，它的结果中也包含 `poc` 列
- 由于 `poc` 已经在 `result_df` 中，`new_cols` 会过滤掉它
- **但是**，如果 `sqs_hal_low` 的结果中有**多个** `poc` 列（因为内部计算产生了重复），或者合并逻辑有问题，就会导致 `combined_result` 本身有重复列名

### 3. 数据传递：重复列名传播到下游

**问题链条**：
1. `sqs_hal_high` 计算 → 输出包含 `poc` 列
2. `sqs_hal_low` 计算 → 输出也包含 `poc` 列
3. 合并时，如果处理不当，`result_df` 中可能有重复的 `poc` 列
4. 当 `result_df` 作为输入传给 `extract_wpt_features` 时，`df.columns` 中就有重复列名
5. `extract_wpt_features` 中使用 `df[wpt_cols]` 时，如果 `wpt_cols` 中有重复，或者 `df.columns` 有重复，就会报错

### 4. 按月计算时的额外问题

**问题**：在按月计算时，每个月份的数据可能都包含了这两个特征的结果：
- 月份 A：`sqs_hal_high` → `poc` 列
- 月份 A：`sqs_hal_low` → `poc` 列（重复！）
- 月份 B：`sqs_hal_high` → `poc` 列
- 月份 B：`sqs_hal_low` → `poc` 列（重复！）

当合并月份数据时，如果某些月份有重复的 `poc` 列，`reindex` 操作会失败。

## 根本原因总结

### 直接原因
1. **多个特征输出相同的列名**（`poc`）
2. **合并逻辑没有正确处理重复列名**

### 深层原因
1. **特征设计不合理**：
   - `poc` 应该是一个独立的基础特征，而不是多个特征的副产品
   - 或者，应该使用不同的列名（如 `poc_hal_high`, `poc_hal_low`）

2. **合并逻辑不完善**：
   - 应该检查并处理重复列名
   - 应该使用 `update` 或 `combine_first` 而不是简单的 `concat`
   - 或者在合并前确保没有重复列名

3. **数据验证不足**：
   - 在函数入口处没有检查输入 DataFrame 是否有重复列名
   - 在合并后没有验证结果是否有重复列名

## 解决方案

### 短期方案（已实施）
1. ✅ 在数据加载和计算后立即处理重复列名
2. ✅ 在合并前检查并处理重复列名
3. ✅ 在函数入口处处理重复列名（如 `extract_wpt_features`）

### 长期方案（建议）
1. **重构特征设计**：
   - 将 `poc` 作为独立的基础特征
   - 或者使用不同的列名区分不同来源的 `poc`

2. **改进合并逻辑**：
   - 使用 `update` 或 `combine_first` 处理已存在的列
   - 或者在合并前统一去重

3. **添加数据验证**：
   - 在函数入口处检查重复列名
   - 在合并后验证结果
