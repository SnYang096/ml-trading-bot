# 稳定性验证（Stability Validation）评估

## 什么是稳定性验证？

稳定性验证是一种**因子时间稳定性测试**方法，用于验证在近期数据上选择的因子是否在更长历史数据上仍然有效。

### 实现原理

在 `ts-dim-compare` 中的稳定性验证流程：

1. **因子选择期**（近期数据）
   - 时间段：`train_start` → `train_end`（例如：2024-01-01 → 2024-12-31）
   - 在此时间段上计算因子的 IC（Information Coefficient）
   - 选择 IC 排名靠前的因子

2. **稳定性验证期**（历史数据）
   - 时间段：`validation_start` → `train_start`（例如：2021-01-01 → 2024-01-01）
   - 在验证期数据上重新计算这些因子的 IC
   - 比较两个时期的 IC 差异

3. **稳定性判断标准**
   ```python
   # 因子被认为是稳定的，如果：
   - IC 符号一致（ic_selection * ic_validation > 0）
   - IC 幅度相似（0.5 < stability_ratio < 2.0）
   # 其中 stability_ratio = |IC_validation| / |IC_selection|
   ```

4. **输出结果**
   - 稳定因子列表（IC 在两个时期保持一致）
   - 不稳定因子列表（IC 变化显著）
   - 稳定性比率（稳定因子占比）

## 评估：在哪里做更合理？

### 选项 1: 在 `ts-dim-compare` 中做（当前实现）✅ **推荐**

#### 优势
1. **因子选择阶段验证**
   - 在特征选择阶段就验证因子稳定性
   - 可以过滤掉不稳定的因子，只保留稳定的因子进入后续阶段
   - 避免用不稳定的因子训练模型

2. **早期发现问题**
   - 在模型训练前就发现因子稳定性问题
   - 节省计算资源（不需要用不稳定因子训练模型）
   - 提高模型质量（只用稳定因子）

3. **符合研究流程**
   - 你的流程：因子测试 → 时间周期优化 → **降维测试** → 滚动训练
   - 在降维测试阶段验证稳定性，正好符合流程逻辑

4. **一次性验证**
   - 因子选择是一次性的（基于近期数据）
   - 稳定性验证也是一次性的（验证历史稳定性）
   - 不需要在每次滚动训练时重复验证

#### 劣势
1. **需要额外数据加载**
   - 需要加载历史验证数据
   - 增加计算时间（但通常可接受）

2. **验证期可能不够长**
   - 如果只验证 3 年，可能不够全面
   - 但可以通过 `VALIDATION_YEARS` 参数调整

---

### 选项 2: 在 `rolling` 中做 ❌ **不推荐**

#### 为什么不适合？

1. **滚动训练的本质**
   - `rolling` 是**时间序列交叉验证**（Walk-Forward Analysis）
   - 每个月用历史数据训练，在下一个时间段测试
   - 本身就是一个**动态适应**的过程

2. **验证目标不同**
   - 稳定性验证：验证**因子选择**是否稳定
   - 滚动训练：验证**模型性能**是否稳定
   - 两者关注点不同

3. **重复验证浪费资源**
   - 如果因子已经通过稳定性验证，不需要在每次滚动训练时重复验证
   - 滚动训练应该专注于模型训练和性能评估

4. **时机不对**
   - 因子选择应该在滚动训练**之前**完成
   - 稳定性验证应该在因子选择**之后**立即进行
   - 在滚动训练时验证因子稳定性，时机已经太晚了

#### 如果要在 rolling 中做，应该验证什么？

在 `rolling` 中应该验证的是：
- ✅ **模型性能稳定性**：每个月的模型性能是否稳定
- ✅ **特征重要性稳定性**：特征重要性是否随时间变化
- ✅ **预测准确率稳定性**：预测准确率是否下降
- ❌ **不是因子 IC 稳定性**（这应该在 ts-dim-compare 中做）

## 最佳实践建议

### ✅ 推荐方案：在 `ts-dim-compare` 中做稳定性验证

**理由：**
1. **符合研究流程**：因子选择 → 稳定性验证 → 降维 → 训练
2. **早期发现问题**：在模型训练前就过滤不稳定因子
3. **节省资源**：避免用不稳定因子训练模型
4. **一次性验证**：因子选择是一次性的，稳定性验证也应该是一次性的

### 📋 使用建议

```bash
# 在降维测试阶段启用稳定性验证
make ts-dim-compare \
    SYMBOLS=BTCUSDT \
    START_DATE=2024-01-01 \
    END_DATE=2024-12-31 \
    DIM_COMPARE_FEATURE_TYPE=comprehensive \
    DIM_COMPARE_TIMEFRAME=5T \
    # 启用稳定性验证
    ENABLE_STABILITY_VALIDATION=true \
    # 自动计算：从 train_start 往前推 3 年
    VALIDATION_YEARS=3 \
    # 或者手动指定验证开始时间
    # VALIDATION_START=2021-01-01
```

### 🔍 验证结果解读

稳定性验证会输出：
- **稳定因子列表**：IC 在两个时期保持一致的因子
- **不稳定因子列表**：IC 变化显著的因子
- **稳定性比率**：稳定因子占比（建议 > 60%）

**判断标准：**
- ✅ **稳定性比率 > 60%**：因子选择质量良好
- ⚠️ **稳定性比率 40-60%**：需要检查因子选择逻辑
- ❌ **稳定性比率 < 40%**：因子选择可能有问题，需要重新评估

### 🚫 不要在 `rolling` 中做因子稳定性验证

**原因：**
1. 时机不对：因子选择应该在滚动训练之前完成
2. 目标不同：滚动训练应该关注模型性能，不是因子稳定性
3. 浪费资源：重复验证已经验证过的因子

**但可以在 `rolling` 中做：**
- 模型性能监控
- 特征重要性变化监控
- 预测准确率下降检测

## 总结

| 验证类型 | 在哪里做 | 验证内容 | 时机 |
|---------|---------|---------|------|
| **因子稳定性验证** | `ts-dim-compare` ✅ | IC 在历史数据上的稳定性 | 因子选择后，模型训练前 |
| **模型性能监控** | `rolling` ✅ | 模型性能是否下降 | 滚动训练过程中 |
| **特征重要性监控** | `rolling` ✅ | 特征重要性是否变化 | 滚动训练过程中 |

### 结论

**在 `ts-dim-compare` 中做稳定性验证是合理的**，因为：
1. ✅ 符合研究流程逻辑
2. ✅ 早期发现问题，节省资源
3. ✅ 一次性验证，不需要重复
4. ✅ 验证的是因子选择，不是模型性能

**不要在 `rolling` 中做因子稳定性验证**，因为：
1. ❌ 时机不对（因子选择应该在训练前完成）
2. ❌ 目标不同（滚动训练关注模型性能）
3. ❌ 浪费资源（重复验证）
