# 07 与 Nautilus Trader 对齐清单（回测=实盘）

本清单用于把“框架能力”转化为“结果一致性”：确保事件回测与实盘在关键点对齐。

## 一、事件与时间

- **事件类型对齐**：回测与实盘是否都使用相同的 tick/bar 类型（TradeTick/Bar 等）？
- **时间戳语义对齐**：计算与决策使用 event time，而不是接收时间。
- **bar 对齐对齐**：bar 边界、时区、聚合方式一致（EXTERNAL vs INTERNAL）。

## 二、数据连续性与补全

- **缺口检测**：是否在进入信号生成前保证 bar 连续？
- **缺口补全**：缺口数据来自同一权威源（REST/历史下载），并写入同一存储（QuestDB）？
- **重放恢复**：补全后是否会重放增量状态直到追平？

参考：
- `reference/实时流数据补全机制设计.md`
- `reference/实时流数据补全使用指南.md`

## 三、特征计算一致性

- **增量状态一致**：回测与实盘是否复用同一增量特征计算器/状态机？
- **warmup 一致**：窗口大小、warmup 判定与 NaN 策略一致。
- **参数一致**：VPIN bucket 体积、窗口长度、多时间框架配置一致。
- **fit 禁止**：实盘是否强制不 fit（仅加载训练参数）？

## 四、交易状态对账

- **对账开启**：执行引擎是否启用对账（reconciliation）？
- **对账失败处理**：失败是否进入 DEGRADED/HALT，停止开新仓？
- **重启恢复**：重启后是否刷新订单/持仓/余额，并与策略内部状态对齐？

## 五、异常发现与稳定性

- **Inf 监控**：源数据与计算后 inf 都有监控与明确处理策略（见 `docs/features/DATA_MONITORING.md`）。
- **延迟监控**：event→ingest 延迟突变是否告警/降级？
- **熔断策略**：质量闸门失败时是否停止交易并输出诊断？

## 六、最小验收（上线前必须过）

- 同一段事件流：回测回放与“实盘记录回放”产出的特征快照一致（允许浮点误差）。
- 人为制造断连：能自动补全并恢复到 NORMAL。
- 人为注入 inf：能触发质量闸门并熔断，且不会继续开新仓。
