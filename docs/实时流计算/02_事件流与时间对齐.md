# 02 事件流与时间对齐

本章定义“回测/实盘一致”的第一原则：**事件顺序与时间对齐一致**。

## 一、统一事件流（Canonical Event Stream）

建议把市场数据按以下层次理解（从底到顶）：

- **Tick 层（订单流/成交流）**：用于 VPIN、CVD、订单流聚合等。
- **Bar 层（K 线）**：用于价格/波动率/形态等时间框架特征。
- **策略时钟（signal clock）**：决定“何时产生信号/下单”，通常与 bar 的收盘边界对齐（例如每 15 分钟检查一次）。

在实盘里，tick 与 bar 可能同时到达；在回测里，tick 与 bar 需要按时间戳严格重放。两者要一致，必须明确：

- **哪个事件驱动特征更新**（tick 更新订单流，bar 更新 OHLCV 相关特征）。
- **哪个时钟驱动决策**（例如每 15 分钟检查一次信号）。

> 本项目的推荐实现：tick 更新增量状态，bar 更新多时间框架状态，定时器/边界触发进行信号融合与交易（参考 `docs/事件驱动架构说明.md`）。

## 二、时间戳语义（Event Time vs Ingest Time）

必须在全链路统一使用**事件时间**作为计算基准：

- WebSocket 接收时间只能用于监控延迟与健康状况，不应驱动特征/信号（否则回放无法复现）。
- 对外存储（QuestDB）应将 timestamp 字段作为事件时间写入，避免“写入时间”污染。

## 三、Bar 对齐规则（Bar Alignment）

确保回测与实盘的 bar 对齐一致：

- **bar 边界**：例如 15T bar 必须严格对齐到交易所/UTC 边界（需要在全系统里固定）。
- **bar 完整性**：若 bar 因缺口被补全，应当标记该 bar 是否来自补全（用于监控与回放一致性）。
- **重复/乱序**：同一 `(symbol, timeframe, timestamp)` 出现多次时，必须有确定性处理（保留第一条/最后一条/按交易所序号）。

## 四、缺口与乱序（Gap / Out-of-order）

一致性要求：

- **缺口处理必须“先补全，再计算/交易”**（至少对进入决策的时间框架）。
- **乱序处理必须“先排序，再更新状态”**，或在状态更新中具备乱序容忍（成本更高）。

缺口补全的具体机制见：

- `reference/实时流数据补全机制设计.md`
- `reference/实时流数据补全使用指南.md`

## 五、实操建议（最低实现成本的稳定策略）

- **实时**：检测到 gap → 暂停交易（或只做持仓管理）→ 补全 → 重放缺口段 → 恢复交易。
- **回测**：以持久化事件源（Catalog/QuestDB 导出）严格按 event time 重放，保证与实盘一致。
