# 06 实盘稳定性运行手册

本章是“上线可运维”的最低要求清单：让系统在断网、延迟、缺口、异常值、交易所问题下依然可控。

## 一、启动流程（必须可重复）

推荐启动顺序：

1. **加载模型与配置**（版本绑定：模型/特征/归一化参数）
2. **warmup**：
   - 从 QuestDB 加载最近窗口（优先）
   - 不足时从 Parquet 补齐
3. **质量闸门**：
   - 源数据 inf 检查（失败则拒绝启动或进入 HALT）
   - warmup 是否达到最小窗口（不足则只允许进入 DEGRADED）
4. **订阅实时数据流**（tick + bar）
5. **进入 NORMAL**（满足连续性与对账条件后）

## 二、在线健康指标（Observability）

建议至少记录与告警：

- **数据连续性**
  - bar 缺口次数、缺口长度、补全耗时
  - tick 流断连次数、断连时长
- **延迟**
  - event_ts → ingest_ts 延迟分布（p50/p95/p99）
- **特征质量**
  - inf 新增次数
  - 关键特征 NaN 比例（warmup 之外的 NaN 应视为异常）
- **交易一致性**
  - 对账失败次数
  - 订单拒绝/超时/部分成交比例

## 三、降级与熔断（Stability First）

推荐“先稳后赚”的策略：

- **DEGRADED 模式**（轻度异常）：
  - 不开新仓
  - 只做止损止盈/撤单/风控
  - 继续补全/重放直到恢复 NORMAL
- **HALT 模式**（严重异常）：
  - 停止所有主动交易动作（除紧急风控）
  - 输出告警与诊断信息
  - 等待人工介入/自动恢复条件满足后再继续

## 四、异常场景演练（建议定期跑）

- WebSocket 断连（自动重连 + 补全）
- 系统重启（warmup + 启动时补全 + 恢复一致）
- QuestDB 暂不可用（降级：仅内存缓存 + 本地落盘）
- 特征出现 inf（触发质量闸门与熔断）
- 交易所 API 超限（补全/对账退避重试）

## 五、事故排查最短路径（Playbook）

1. 先看：是否发生缺口/补全？是否处于 DEGRADED/RECOVERING/HALT？
2. 再看：inf/NaN 监控输出（源数据 vs 计算产生）
3. 再看：对账是否失败（订单/持仓不一致）
4. 必要时：导出同一时间段事件流回放复现
