# 事件驱动架构说明

## 概述

本项目已实现基于事件驱动的实盘策略和回测系统，使用 Nautilus Trader 作为底层框架。该架构实现了训练和实盘的统一，大幅减少内存使用，并支持实时特征计算。

## 架构设计

### 核心组件

```
┌─────────────────────────────────────────┐
│  训练阶段（向量化批处理）                │
│  - 快速特征计算                          │
│  - 模型训练                              │
│  - 参数优化                              │
└─────────────────────────────────────────┘
           ↓ 导出模型 + 特征配置
┌─────────────────────────────────────────┐
│  实盘/回测阶段（事件驱动）               │
│  - 使用训练好的模型                      │
│  - 事件驱动特征计算                      │
│  - 实时决策                              │
└─────────────────────────────────────────┘
```

## 核心特性

### 1. 增量特征计算器 (`IncrementalFeatureComputer`)

**位置**: `src/time_series_model/live/incremental_feature_computer.py`

**功能**:
- 流式处理 tick 和 bar 数据
- 维护状态（VPIN buckets、滚动窗口等）
- 支持跨月连续性
- 内存占用低（只保留滑动窗口）

**主要方法**:
- `on_tick(tick)`: 处理 tick 数据，更新订单流特征
- `on_bar(bar, timeframe)`: 处理 bar 数据，更新时间框架特征
- `get_features()`: 获取当前所有特征
- `get_orderflow_features(window_minutes)`: 获取订单流特征

**示例**:
```python
from src.time_series_model.live.incremental_feature_computer import IncrementalFeatureComputer

computer = IncrementalFeatureComputer(
    tick_window_minutes=30,
    bar_window_size=1000,
    vpin_bucket_volume_usd=100000.0,  # $100k per bucket
    vpin_n_buckets=50,
)

# 在事件回调中使用
computer.on_tick(trade_tick)
computer.on_bar(bar, timeframe="1H")
features = computer.get_features()
```

### 2. 事件驱动策略 (`EventDrivenStrategy`)

**位置**: `src/time_series_model/live/event_driven_strategy.py`

**功能**:
- 基于 Nautilus Trader 的事件驱动架构
- 在 `on_tick` 中计算 tick 相关特征
- 在 `on_bar` 中计算时间框架特征
- 在定时器中融合信号并执行交易
- 支持多时间框架特征融合

**主要方法**:
- `on_trade_tick(tick)`: 处理交易 tick
- `on_bar(bar)`: 处理 bar 数据
- `_on_signal_check()`: 定时器回调，执行信号融合
- `_evaluate_entry_signal()`: 评估入场信号

**配置**:
```python
strategy = EventDrivenStrategy(
    strategy_name="sr_reversal",
    instrument_id=instrument_id,
    bar_types={"15T": bar_type_15m, "1H": bar_type_1h},
    trade_size=0.001,
    check_interval_minutes=15,  # 每 15 分钟检查一次信号
    min_order_interval_minutes=15,  # 最小开仓间隔
    vpin_bucket_volume_usd=100000.0,
)
```

### 3. 事件驱动回测 (`EventDrivenBacktest`)

**位置**: `src/time_series_model/backtesting/event_driven_backtest.py`

**功能**:
- 使用 Nautilus Trader 的回测引擎
- 与实盘策略使用相同的代码
- 支持历史数据回测
- 详细的交易记录和性能分析

**使用方法**:
```python
from src.time_series_model.backtesting.event_driven_backtest import run_event_driven_backtest

result = run_event_driven_backtest(
    strategy_name="sr_reversal",
    symbol="BTCUSDT-PERP",
    data_path="data/catalog",
    start_date="2024-01-01",
    end_date="2024-12-31",
    timeframes=["15T", "1H"],
    trade_size=0.001,
    model_path="models/sr_reversal/model.pkl",
)
```

### 4. WebSocket 客户端 (`BinanceWebSocketClient`)

**位置**: `src/live_data_stream/websocket_client.py`

**功能**:
- Binance 实时数据流
- 自动重连
- 多币种订阅
- 数据回调接口

**使用方法**:
```python
from src.live_data_stream.websocket_client import BinanceWebSocketClient

def on_tick(tick):
    print(f"Tick: {tick.symbol} @ {tick.price}")

client = BinanceWebSocketClient(
    symbols=["BTCUSDT", "ETHUSDT"],
    use_futures=True,
)
client.add_callback(on_tick)

stop_event = asyncio.Event()
await client.run(stop_event)
```

## 使用流程

### 实盘交易

1. **准备模型和配置**:
   ```bash
   # 训练模型（使用向量化批处理）
   make rolling ROLLING_CONFIG=config/strategies/sr_reversal SYMBOL=BTCUSDT
   ```

2. **运行实盘策略**:
   ```bash
   python -m src.time_series_model.live.run_nautilus_strategy \
       --strategy sr_reversal \
       --symbol BTCUSDT-PERP \
       --timeframe 15T \
       --testnet
   ```

   ```bash

   export BINANCE_API_SECRET=your_secret
   export BINANCE_FUTURES_TESTNET_API_KEY=testnet_key  # 测试网

   ```

## 优势

### 1. 内存使用大幅减少
- **批处理**: 需要加载整段历史数据（GB 级）
- **事件驱动**: 只维护滑动窗口（MB 级）
- **节省**: 从 GB 级降到 MB 级

### 2. 训练和实盘统一
- **批处理**: 训练用批处理，实盘用事件驱动，逻辑不一致
- **事件驱动**: 同一套代码，只需切换数据源
- **优势**: 减少代码重复，提高可维护性

### 3. 实时性更好
- **批处理**: 需要等待数据积累，延迟高
- **事件驱动**: tick 到达即处理，延迟低
- **优势**: 更适合高频交易场景

### 4. 代码结构更清晰
- **按事件类型组织**: `on_tick` → tick 特征，`on_bar` → 时间框架特征
- **职责分离**: 特征计算、信号生成、交易执行分离
- **易于维护**: 模块化设计，易于扩展

## 注意事项

### 1. 向量化优势丢失
- **训练阶段**: 仍使用向量化批处理（快速）
- **实盘/回测**: 使用事件驱动（实时、统一）
- **建议**: 采用混合方案，训练用批处理，实盘用事件驱动

### 2. 回测速度可能变慢
- **批处理**: 批量处理，充分利用 CPU
- **事件驱动**: 逐条处理，CPU 利用率低
- **影响**: 回测时间从分钟级可能变成小时级

### 3. 特征计算复杂度增加
- **需要维护状态**: VPIN buckets、滚动窗口等
- **跨月连续性**: 需要处理跨月 bucket 状态传递
- **建议**: 使用 `IncrementalFeatureComputer` 统一管理状态
