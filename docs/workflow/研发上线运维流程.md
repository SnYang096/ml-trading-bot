# 对于B系统的策略，我们花了最多的时间，也启发了我们找到A和C，但是B系统目前确实太慢了，需要调整优化

1. 《research_roll.features_on》经常去想找更加有效的每层特征，这等于是用树模型去找特征拟合行情，这是随着数据变动会有不断地变化，而且这个过程非常慢，我们应该在策略失效后再做这个，重新看看策略能否工作，而不是经常做这个；‘而且这样用管线做也很慢，应该一步一步分析每层在新的数据下有什么变化，能不能达到训练目标，这个在策略制定的时候是有用的。 

2. 《calibrate_roll.default》调整阈值我们也经常做，这个流程同样也比较慢，根据最近数据调整阈值滚动向前，这个流程应该取代《research_roll.features_on》管线作为对未来的模拟，看看策略在长时间是否能盈利，且无大的回撤

3. 《validate_static.full_study.yaml》我们原来定位的是上线前的管线，但这个管线仍然会找特征，调阈值很慢，不符合我们Prefilter+entry是策略形态，长期不会变的假设，我们就是要验证这个假设是否还成立，成立就应该微调阈值上线。

4. 《validate_static.constrained.yaml》对应的是快速上线前的回测，这里面微调阈值即可，但是我们的管线可能会调整的比较大，导致阈值漂移

5. 根据最新的研究，我们应该有一个策略运行regime的调整工具，主要用来调ema1200-2h是否划分数据比较合理，能区分多空，而我们现在没有这个最重要的工具，相反这个是我定义的

6. 现在回测event backtest和pcm非常慢，不知道是否因为是1mins执行粒度导致的，我感觉b系统并不需要这么细的粒度，前面用这么细是因为会网格找到极小的止损，因为没有2hbar内的执行粒度，所以无法正确止损出来（这也说明了trend-scalping存在的合理性，因为只要没止损有止盈短期大概率是赢得，配合regime和entry），所以才用的细粒度，但我们现在做swing，强制sl很粗的范围，应该问题不大了，用哪个2h执行即可

7. 现在gate仍然会用订单流的指标，而gate应该是过滤数据的大尺度，但他不应该是定义数据范围（eam1200这种）和定义策略形态结构（Prefilter这种），他就应该是极端异常情况的去除，他应该工作在Prefilter以后得数据中去剔除容易产生1R的数据，2R的数据，3R的数据，然后看看哪些特征和阈值能满足这个需求
