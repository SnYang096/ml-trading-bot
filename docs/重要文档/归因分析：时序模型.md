# 归因日志的数据库 schema 设计

以下是为量化交易系统设计的 归因日志数据库 Schema，专用于记录模型开仓/平仓信号的 SHAP 特征归因，兼顾查询效率、存储成本与分析灵活性。

🗃️ 核心设计原则

1. 每条记录 = 一个 symbol 在一个时间点的一次预测 + 归因
2. 分离主表与特征明细表：避免宽表膨胀（400+ 特征）
3. 支持快速查询：
“BTC 昨天为什么开仓？”
“过去一周哪些特征最常驱动买入？”
“某特征异常贡献是否导致亏损？”
4. 兼容多模型版本（便于 A/B 测试）

📐 推荐 Schema（PostgreSQL / MySQL 兼容）
表 1：signal_attribution（主表）
存储每次预测的核心元数据和聚合指标

sql
CREATE TABLE signal_attribution (
id BIGSERIAL PRIMARY KEY,

-- 时间与资产
timestamp TIMESTAMPTZ NOT NULL, -- 预测所用K线结束时间（UTC）
instrument VARCHAR(20) NOT NULL, -- 如 'BTCUSDT'
timeframe VARCHAR(10) NOT NULL, -- 如 '240T'

-- 模型信息
model_name VARCHAR(50) NOT NULL, -- 如 'lgb_v3'
model_version VARCHAR(20) NOT NULL, -- 如 '20241110'

-- 预测结果
predicted_return DOUBLE PRECISION NOT NULL,
predicted_direction SMALLINT NOT NULL CHECK (predicted_direction IN (-1, 0, 1)), -- -1:空, 0:平, 1:多

-- 关键归因摘要（加速查询）
top_positive_feature VARCHAR(100), -- 贡献最大的正向特征名
top_positive_value DOUBLE PRECISION, -- 其 SHAP 值
top_negative_feature VARCHAR(100), -- 贡献最大的负向特征名
top_negative_value DOUBLE PRECISION, -- 其 SHAP 值
base_value DOUBLE PRECISION NOT NULL, -- SHAP 基准值（模型全局均值）

-- 实盘关联（可选）
order_id UUID, -- 若已下单，关联订单ID
position_size DOUBLE PRECISION, -- 开仓仓位大小

-- 索引
INDEX idx_instrument_time (instrument, timestamp DESC),
INDEX idx_model_time (model_name, model_version, timestamp DESC),
INDEX idx_direction_time (predicted_direction, timestamp DESC)
);

表 2：attribution_detail（明细表）
存储每个特征的具体 SHAP 贡献（按需写入，非全量）

sql
CREATE TABLE attribution_detail (
id BIGSERIAL PRIMARY KEY,
signal_id BIGINT NOT NULL REFERENCES signal_attribution(id) ON DELETE CASCADE,

feature_name VARCHAR(100) NOT NULL, -- 如 'return_vol_12'
shap_value DOUBLE PRECISION NOT NULL, -- 该特征的 SHAP 贡献
raw_feature_value DOUBLE PRECISION, -- 原始特征值（用于上下文解读）

-- 索引
INDEX idx_signal_id (signal_id),
INDEX idx_feature_name (feature_name)
);
💡 存储优化建议：
只存 Top-K 特征（如 K=20），而非全部 400 个
→ 99% 的决策由 Top-20 特征驱动，节省 95% 存储
或：只存 SHAP > 阈值 的特征（如 > 0.001）

🔍 典型查询场景
1. 查看 BTC 最近一次开多归因
sql
SELECT FROM signal_attribution
WHERE instrument = 'BTCUSDT' AND predicted_direction = 1
ORDER BY timestamp DESC LIMIT 1;
-- 再 JOIN attribution_detail 获取明细
2. 统计过去7天最活跃的驱动特征
sql
SELECT
feature_name,
AVG(ABS(shap_value)) as avg_impact,
COUNT() as trigger_count
FROM attribution_detail ad
JOIN signal_attribution sa ON ad.signal_id = sa.id
WHERE sa.timestamp > NOW() - INTERVAL '7 days'
GROUP BY feature_name
ORDER BY avg_impact DESC
LIMIT 10;
3. 检查某次亏损是否由异常特征导致
sql
-- 假设已知 order_id = 'abc123'
SELECT ad.feature_name, ad.shap_value, ad.raw_feature_value
FROM attribution_detail ad
JOIN signal_attribution sa ON ad.signal_id = sa.id
WHERE sa.order_id = 'abc123'
ORDER BY ABS(ad.shap_value) DESC;

💾 预估存储量（参考）
每天 100 个币 × 24 小时（1h 频率） = 2400 条主记录
每条主记录 + 20 条明细 = 2400 × 21 ≈ 5 万行/天
每行 ~100 bytes → 5 MB/天，完全可接受
