# EVT特征NaN处理策略分析

## 问题背景

`evt_features` 有 66.9% 的 NaN 值，需要决定是保持 NaN 还是使用很小的默认值。

## 两种方案对比

### 方案1：保持 NaN

**优点**：
- ✅ **真实反映数据不足**：NaN 明确表示"数据不足以计算特征"
- ✅ **模型可以学习**：LightGBM 会将 NaN 视为缺失值，用中位数填充（基于训练集）
- ✅ **不会引入虚假信号**：不会因为默认值而产生误导性的特征值
- ✅ **更符合统计学原理**：缺失就是缺失，不应该用任意值填充

**缺点**：
- ❌ **NaN 太多（66.9%）**：可能导致特征信息量不足
- ❌ **中位数填充可能不合理**：LightGBM 会用训练集的中位数填充，但 EVT 特征的中位数可能没有实际意义
- ❌ **警告信息**：会产生大量 NaN 警告

### 方案2：使用很小的默认值（当前实现）

**优点**：
- ✅ **减少 NaN**：所有位置都有值，不会产生 NaN 警告
- ✅ **模型更容易处理**：不需要处理缺失值
- ✅ **基于实际数据计算**：当数据不足时，使用经验分位数等基于实际数据的保守估计

**缺点**：
- ❌ **可能引入偏差**：如果默认值设置不当，可能误导模型
- ❌ **掩盖数据不足**：模型无法区分"真实计算值"和"默认值"
- ❌ **可能产生虚假信号**：默认值可能在某些情况下产生误导

### 方案3：基于实际数据的保守估计（推荐）⭐

**优点**：
- ✅ **基于实际数据**：使用经验分位数、均值等基于实际数据的统计量
- ✅ **保守估计**：当数据不足时，使用保守的估计值（如较小的风险值）
- ✅ **减少 NaN**：大部分情况下都能计算出合理的值
- ✅ **保持真实性**：值是基于实际数据计算的，不是任意填充

**缺点**：
- ❌ **计算复杂度稍高**：需要额外的计算逻辑
- ❌ **可能仍有少量 NaN**：极端情况下（如完全没有 excesses）可能仍有 NaN

## 当前实现分析

当前代码已经实现了**方案3**（基于实际数据的保守估计）：

1. **初始化时使用很小的默认值**：
   ```python
   xi_left = np.full(n, 0.01)      # 很小的默认值
   var_99_left = np.full(n, -0.01)  # 很小的默认值
   ```

2. **当 excesses 不足时，基于实际数据计算**：
   ```python
   if len(excesses_left) > 0:
       empirical_var = u_left - np.quantile(excesses_left, 0.99)
       var_99_left[df_idx] = min(empirical_var, -0.001)
   ```

3. **当拟合失败时，基于实际数据计算**：
   ```python
   except Exception:
       if len(excesses_left) > 0:
           empirical_var = u_left - np.quantile(excesses_left, 0.99)
           var_99_left[df_idx] = min(empirical_var, -0.001)
   ```

## 建议

### 推荐：保持当前实现（方案3）

**理由**：
1. ✅ **基于实际数据**：值是基于实际数据计算的，不是任意填充
2. ✅ **保守估计**：当数据不足时，使用保守的估计值，不会高估风险
3. ✅ **减少 NaN**：大部分情况下都能计算出合理的值
4. ✅ **符合业务逻辑**：EVT 特征用于风险管理，保守估计比缺失值更安全

### 可选改进：添加配置选项

如果需要更灵活的控制，可以添加一个参数：

```python
def extract_evt_features(
    df: pd.DataFrame,
    ...
    fill_na_with_default: bool = True,  # 是否用默认值填充 NaN
) -> pd.DataFrame:
```

这样用户可以根据需要选择：
- `fill_na_with_default=True`：使用基于实际数据的保守估计（当前实现）
- `fill_na_with_default=False`：保持 NaN，让模型处理

## 实际效果

### 当前实现的效果

- **66.9% NaN → 0% NaN**：所有位置都有值
- **基于实际数据**：值是基于实际数据计算的，不是任意填充
- **保守估计**：当数据不足时，使用保守的估计值（较小的风险值）

### 如果保持 NaN 的效果

- **66.9% NaN**：大部分位置是 NaN
- **LightGBM 处理**：会用训练集的中位数填充
- **可能的问题**：EVT 特征的中位数可能没有实际意义

## 结论

**推荐保持当前实现（方案3）**：
- ✅ 基于实际数据计算，不是任意填充
- ✅ 减少 NaN，提高特征可用性
- ✅ 保守估计，符合风险管理需求
- ✅ 模型更容易处理，不需要处理大量 NaN

如果确实需要保持 NaN，可以考虑：
1. 添加配置选项，让用户选择
2. 或者只在极端情况下（完全没有数据）才使用 NaN
