# 四种策略的特征需求分析

## 当前特征使用方式

### 1. SR Reversal（SR 反转）
- **基础特征**：使用 `ComprehensiveFeatureEngineer` 生成
- **策略专属特征**：`build_sr_reversal_features()` 添加
- **特征选择**：`select_sr_reversal_features()` 基于关键词过滤
- **特征类型**：`baseline,default` 或 `comprehensive`

### 2. SR Breakout（SR 突破）
- **基础特征**：使用 `ComprehensiveFeatureEngineer` 生成
- **策略专属特征**：`build_sr_breakout_features()` 添加
- **特征选择**：`select_sr_breakout_features()` 基于关键词过滤
- **特征类型**：`comprehensive`

### 3. Compression Breakout（压缩突破）
- **基础特征**：使用 `ComprehensiveFeatureEngineer` 生成
- **策略专属特征**：`build_compression_breakout_features()` 添加
- **特征选择**：`select_compression_breakout_features()` 基于关键词过滤
- **特征类型**：`comprehensive`

### 4. Trend Following（趋势跟踪）
- **基础特征**：使用 `ComprehensiveFeatureEngineer` 生成
- **策略专属特征**：`build_trend_following_features()` 添加
- **特征选择**：`select_trend_following_features()` 基于关键词过滤
- **特征类型**：`comprehensive`

## 当前问题

1. **生成所有特征再选择**：效率低，浪费计算资源
2. **特征依赖不明确**：不知道哪些特征依赖哪些特征
3. **研究实盘不一致**：研究阶段生成所有特征，实盘阶段可能只生成部分
4. **特征选择基于关键词**：可能遗漏依赖特征

## 方案适配分析

### 方案 1：特征依赖图 + 拓扑排序

**适配度：⭐⭐⭐⭐**

**优点：**
- ✅ 自动处理依赖关系
- ✅ 按需计算，只计算需要的特征
- ✅ 研究实盘统一

**缺点：**
- ⚠️ 需要重构现有特征代码（添加装饰器）
- ⚠️ 需要分析所有特征的依赖关系

**适用性：**
- 适合长期项目
- 需要完整的依赖管理
- 特征数量多，依赖关系复杂

---

### 方案 2：特征配置文件 + 依赖声明

**适配度：⭐⭐⭐⭐⭐（推荐）**

**优点：**
- ✅ **配置与代码分离**：每个策略可以有独立的特征配置
- ✅ **易于修改**：修改 YAML 即可调整特征组合
- ✅ **支持多策略配置**：可以为每个策略定义不同的特征集
- ✅ **不需要修改现有代码**：只需添加配置文件

**适用性：**
- ✅ **非常适合四种策略**：每个策略可以有独立的特征配置
- ✅ 支持策略级别的特征管理
- ✅ 易于维护和扩展

**示例配置：**
```yaml
# feature_config.yaml
strategies:
  sr_reversal:
    base_features: ["baseline", "default"]
    custom_features:
      - sr_strength_max
      - sr_confluence
      - sqs_hal_high
      - sqs_hal_low
      - volume_ratio
      - compression_score
  
  sr_breakout:
    base_features: ["comprehensive"]
    custom_features:
      - sr_strength_max
      - breakout_speed
      - order_flow_delta
      - follow_through_1
      - follow_through_2
      - momentum_decay
  
  compression_breakout:
    base_features: ["comprehensive"]
    custom_features:
      - compression_score
      - compression_confidence
      - breakout_speed
      - volume_ratio
  
  trend_following:
    base_features: ["comprehensive"]
    custom_features:
      - trend_strength
      - momentum
      - adx
      - ma_slope
```

### 方案 3：特征计算管道 + 依赖注入

**适配度：⭐⭐⭐**

**优点：**
- ✅ 管道化设计，清晰
- ✅ 支持缓存，提高效率

**缺点：**
- ⚠️ 需要重构现有代码
- ⚠️ 管道管理可能复杂

**适用性：**
- 适合需要高性能计算的场景
- 不适合快速实现需求

---

### 方案 4：混合方案

**适配度：⭐⭐⭐⭐⭐（推荐）**

**优点：**
- ✅ **最小化代码改动**：只需添加特征注册表
- ✅ **快速实现**：4-5天完成
- ✅ **向后兼容**：现有代码仍可用
- ✅ **研究实盘统一**：使用相同的计算逻辑
- ✅ **支持策略级别配置**：可以为每个策略定义特征集

**适用性：**
- ✅ **非常适合四种策略**：可以快速实现
- ✅ 支持策略级别的特征管理
- ✅ 易于维护和扩展

## 推荐方案

### 推荐：方案 2（配置文件）或 方案 4（混合方案）

#### 为什么推荐这两个方案？

1. **支持策略级别的特征管理**
   - 每个策略可以有独立的特征配置
   - 易于为不同策略定制特征集

2. **研究实盘统一**
   - 使用相同的特征配置
   - 确保一致性

3. **易于维护**
   - 配置集中管理
   - 修改特征组合只需修改配置

#### 方案 2 vs 方案 4

| 对比项 | 方案 2（配置文件） | 方案 4（混合方案） |
|--------|------------------|------------------|
| **实现速度** | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **代码改动** | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **配置灵活性** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **维护成本** | ⭐⭐⭐ | ⭐⭐⭐⭐ |

**建议：**
- **如果追求快速实现**：选择 **方案 4（混合方案）**
- **如果追求配置灵活性**：选择 **方案 2（配置文件）**

## 方案 2 + 方案 4 结合（最佳方案）

### 核心思想
1. **特征注册表**（方案 4）：定义特征及其依赖关系
2. **策略配置文件**（方案 2）：为每个策略定义特征集
3. **统一特征加载器**：根据配置按需加载特征

### 实现架构

```python
# 1. 特征注册表（方案 4）
FEATURE_REGISTRY = {
    "atr": {
        "dependencies": [],
        "module": "baseline",
        "compute_func": BaselineFeatureEngineer._compute_atr,
        "category": "technical_indicator"
    },
    "sr_strength_max": {
        "dependencies": ["atr", "sqs_hal_high", "sqs_hal_low"],
        "module": "baseline",
        "compute_func": BaselineFeatureEngineer._compute_boundary_strengths,
        "category": "sr_structure"
    }
}

# 2. 策略配置文件（方案 2）
# feature_config.yaml
strategies:
  sr_reversal:
    base_feature_types: ["baseline", "default"]
    requested_features:
      - sr_strength_max
      - sr_confluence
      - sqs_hal_high
      - sqs_hal_low
      - volume_ratio
      - compression_score
  
  sr_breakout:
    base_feature_types: ["comprehensive"]
    requested_features:
      - sr_strength_max
      - breakout_speed
      - order_flow_delta
      - follow_through_1
      - follow_through_2
      - momentum_decay

# 3. 统一特征加载器
class StrategyFeatureLoader:
    def __init__(self, registry, config_path):
        self.registry = registry
        self.config = self.load_config(config_path)
        self.loader = FeatureLoader(registry)
    
    def load_strategy_features(self, df, strategy_name, fit=True):
        """为指定策略加载特征"""
        strategy_config = self.config["strategies"][strategy_name]
        
        # 1. 生成基础特征（如果需要）
        if strategy_config.get("base_feature_types"):
            engineer = ComprehensiveFeatureEngineer(
                feature_types=",".join(strategy_config["base_feature_types"])
            )
            df = engineer.engineer_all_features(df, fit=fit)
        
        # 2. 按需加载策略专属特征
        requested_features = strategy_config["requested_features"]
        df = self.loader.load_features(df, requested_features, fit=fit)
        
        return df
```

### 使用示例

```python
# 研究阶段
loader = StrategyFeatureLoader(FEATURE_REGISTRY, "feature_config.yaml")
df_train = loader.load_strategy_features(df_raw, "sr_reversal", fit=True)

# 实盘阶段
df_live = loader.load_strategy_features(df_live_raw, "sr_reversal", fit=False)
```

### 优势

1. **策略级别管理**：每个策略有独立的特征配置
2. **依赖自动处理**：自动解析和计算依赖特征
3. **按需计算**：只计算需要的特征
4. **研究实盘统一**：使用相同的配置和计算逻辑
5. **易于扩展**：新增策略只需添加配置
