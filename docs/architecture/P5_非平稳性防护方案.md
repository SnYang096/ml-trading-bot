# P5 非平稳性防护方案 (Regime / OOD / Alpha Decay)

> 核心原则：**不新增层**，通过特征注入 + 监控增强扩展现有分层架构

## 1. 问题定义

金融市场非平稳性导致：
- **Regime 突变**：波动率/杠杆环境剧变，训练期学到的规则失效
- **OOD 漂移**：实盘特征分布偏离训练分布，模型输出不可信
- **Alpha 衰减**：Gate 规则拦截率、Evidence 特征 IC 随时间衰减，但现有"连亏 8 次"触发太滞后

## 2. 设计选择：为什么不新增层

| 方案               | 优点                           | 缺点                                |
| ------------------ | ------------------------------ | ----------------------------------- |
| 新增 Regime 层     | 语义清晰                       | 系统复杂度↑，需要独立 decide() 逻辑 |
| **特征注入现有层** | 零架构变更，树模型自动学习交互 | 需要额外监控                        |

选择**特征注入**：将 regime_state 和 ood_score 作为特征注入 Gate/Evidence 树模型，让模型自行学习"在 HIGH_VOL regime 下应如何调整决策"。

## 3. 三个组件

### 3.1 Regime State 特征注入

**原理**：复制 `RegimeDetector` (live_pcm.py) 的检测逻辑为可训练特征。

**检测规则** (优先级：HIGH_LEVERAGE > HIGH_VOL > NORMAL)：

| Regime        | 值  | 条件                                                 |
| ------------- | --- | ---------------------------------------------------- |
| NORMAL        | 0   | 默认                                                 |
| HIGH_VOL      | 1   | atr_percentile > 0.7                                 |
| HIGH_LEVERAGE | 2   | oi_zscore > 1.5 AND funding_rate_abs_zscore_50 > 2.0 |

**实现**：
- 函数：`compute_regime_state_from_df()` (`src/features/time_series/baseline_features.py`)
- 特征节点：`regime_state_f` (`config/feature_dependencies.yaml`)
  - `pass_full_df: true`
  - dependencies: `atr_percentile_f`, `oi_features_f`, `funding_rate_features_f`
  - output: `regime_state` (categorical_int: 0/1/2)
- 注入位置：所有策略 (bpc/me/fer) 的 `features_gate.yaml` + `features_evidence.yaml`

**效果**：树模型在训练时学到"HIGH_VOL 下 path_efficiency 门槛应更严格"等交互规则。

### 3.2 OOD Score → Evidence

**原理**：计算当前特征向量偏离训练分布 [q05, q95] 的比例。

**算法**：
```
ood_score = count(feature_i < q05_i OR feature_i > q95_i) / total_features
```
- NaN 值不计为 OOD
- 范围：[0.0, 1.0]，越高越偏离训练分布

**实现**：
- 函数：`compute_ood_score_from_df()` (`src/features/time_series/baseline_features.py`)
  - baseline 来源：`training_baseline.json` 中的 `feature_distributions` 字段
  - 支持 `q05/q95` 和 `p5/p95` 两种 key 格式
- 特征节点：`ood_score_f` (`config/feature_dependencies.yaml`)
  - `pass_full_df: true`
  - dependencies: [] (无硬依赖，读取所有已计算列)
  - output: `ood_score` (bounded_0_1)
- 注入位置：所有策略的 `features_evidence.yaml` (不进 Gate，Gate 关注规则逻辑)
- 部署同步：`deploy_config_to_live.py` 的 `TOP_LEVEL_CONFIGS` 包含 `training_baseline.json`

**效果**：Evidence 树模型学到"ood_score > 0.3 时应降低置信度"。

### 3.3 Alpha Decay 先行指标

**原理**：替代滞后的"连亏 N 次"触发，通过监控 Gate 规则拦截率和 Evidence 特征 IC 的衰减，**提前**检测 alpha 消失。

#### 3.3.1 Baseline 导出

`scripts/export_training_baseline.py` 在训练结束后导出：

| 字段                    | 内容                                     | 来源                |
| ----------------------- | ---------------------------------------- | ------------------- |
| `layer_kpis`            | 各层 KPI 快照                            | 训练/验证结果       |
| `feature_distributions` | 每个特征的 q05/q95/mean/std              | 训练集统计          |
| `gate_rule_hit_rates`   | 每条 gate 规则的 deny_rate + precision   | logs_gated.parquet  |
| `evidence_feature_ics`  | 每个数值特征的 Spearman IC vs forward_rr | predictions.parquet |

导出时机：`auto_research_pipeline.py` Step 10 自动调用。

#### 3.3.2 周度监控

`scripts/local_monitor_weekly.py` 新增两个检查：

**check_l4_gate_rule_decay()**：
```
对每条 gate 规则:
  current_deny_rate = 最近 N 笔的 deny 率
  baseline_deny_rate = training_baseline.gate_rule_hit_rates[rule].deny_rate
  decay = |current - baseline| / max(baseline, 0.01)
  if decay > 0.50 → ALERT
  if decay > 0.30 → WARN
```

**check_l5_evidence_ic_decay()**：
```
对每个 evidence 特征:
  current_ic = Spearman(feature, forward_rr) over recent window
  baseline_ic = training_baseline.evidence_feature_ics[feature].ic
  decay = (|baseline_ic| - |current_ic|) / max(|baseline_ic|, 0.01)
  if decay > 0.50 → ALERT
  if decay > 0.30 → WARN
```

#### 3.3.3 自动重训触发

`scripts/monitor_retrain.py` 新增第 5 个触发条件：

**check_leading_indicators()**：
- 读取最新 weekly health report
- 如果 gate_rule_decay ALERT 数量 或 evidence_ic_decay ALERT 数量 超过阈值
- 触发 `leading_indicator_decay` 信号

配置 (`config/research_pipeline.yaml`)：
```yaml
retrain_triggers:
  leading_indicator_decay: 0.5    # 50% 的规则/特征衰减 → 触发重训
```

## 4. 假设验证框架

| 层           | 假设                  | 验证方式                       | 失效时动作                   |
| ------------ | --------------------- | ------------------------------ | ---------------------------- |
| Gate         | deny 规则能拦截坏交易 | Gate hit_rate 滚动 vs baseline | hit_rate 衰减 >50% → retrain |
| Evidence     | 特征→收益映射稳定     | Spearman IC 滚动 vs baseline   | IC 衰减 >50% → retrain       |
| Entry Filter | 入场时机特征有预测力  | EF lift 滚动 vs baseline       | lift < 1.0 → 禁用 EF         |
| 全局         | 市场在训练分布内      | ood_score 实时计算             | OOD 高 → Evidence 自动降分   |
| 全局         | Regime 未剧烈变迁     | RegimeDetector 实时            | Regime 变化 → 树模型自动适配 |

## 6. Bug 修复记录

### FER CVD Activity Mask NaN → 0.0

**问题**：`fer_features.py` 的 CVD 活跃度掩码将不活跃时段的 10 个 CVD 依赖特征设为 NaN。实盘 `compute_features_batch()` 中 `pd.isna(v)` 过滤掉 NaN → 模型收不到 FER 特征 → 报 `live_feature_missing (10)`。

**根因链**：
1. `fer_features.py` L436: `.where(cvd_active, np.nan)` — CVD 不活跃时 10 列 → NaN
2. `incremental_feature_computer.py` L1482: `not pd.isna(v)` — NaN 被丢弃不进入 features dict
3. 模型收不到 10 个 FER 特征

**修复**：将 CVD 不活跃时的填充值从 `np.nan` 改为 `0.0` (语义中性值)。
- `fer_signed_efficiency`: 0.0 = 无推进效率
- `fer_efficiency_flip`: 0.0 = 无翻转
- `fer_aggressor_absorption`: 0.0 = 无吸收
- `fer_trapped_*`: 0.0 = 无 trapped 信号
- `fer_impulse_failure_*`: 0.0 = 无失败信号

**测试**：`tests/features/test_fer_features_v2.py` — `TestFERCVDNeutralValueLiveSafety` (5 个测试) 验证实盘特征提取不丢失 FER 特征。
