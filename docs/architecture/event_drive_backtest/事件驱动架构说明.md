# 事件驱动架构说明

## 文档口径（请先读）

本目录历史上混入了「基于 Nautilus Trader 的通用事件回测 / `EventDrivenStrategy`」类描述。**当前仓库里与「实盘对齐的事件回测」直接对应的主实现是** `scripts/event_backtest.py`（`EventBacktester` + `LivePCM` + `PositionSimulator`），而不是已不存在的 `src/time_series_model/backtesting/event_driven_backtest.py` 等路径。

下面按**真实代码布局**说明两套相关能力：

1. **事件回测（研究 / 管线）**：在历史 1m bar + tick 上批量算特征，再按信号时钟走 `LivePCM` / `GenericLiveStrategy`，持仓更新走与实盘共享的 `position_logic`。
2. **实盘与流式数据**：`order_flow_listener`、WebSocket、`IncrementalFeatureComputer` 等负责在线路径。

更细的管线与数据流见同目录下的《事件驱动回测设计方案.md》。

---

## 一、目录与入口（与代码一致）

```
scripts/
  event_backtest.py          # 事件回测 CLI + EventBacktester + PositionSimulator

src/time_series_model/live/
  generic_live_strategy.py   # 单策略 decide 管线（与实盘一致）
  incremental_feature_computer.py  # 增量 / 批量 IFC（回测里用 compute_features_dataframe）
  position_logic.py          # 持仓检查（enforce_position / build_position_dict），实盘与回测共用
  live_feature_plan.py       # 特征计划（与 run_live 对齐）

src/time_series_model/portfolio/
  live_pcm.py                # 多策略 PCM 仲裁（事件回测内直接实例化）

src/live_data_stream/
  websocket_client.py        # Binance WebSocket（期货/现货 URL、回调）
  order_flow_listener.py     # 实盘监听与下单路径（事件回测不直接 import，但持仓逻辑与其共享 position_logic）
  feature_storage.py         # 实盘侧 bar/tick 存储（--data-path none 时回测可读 live 数据）

src/time_series_model/backtesting/
  __init__.py                # 占位；策略专用 Nautilus 回测脚本已不在此目录
```

**说明**：若在其他文档里仍看到 `event_driven_strategy.py`、`run_nautilus_strategy.py`、`sr_*_backtest.py` 等路径，应视为过期；以本文件与 `scripts/event_backtest.py` 为准。

---

## 二、事件回测主链路（`scripts/event_backtest.py`）

- **数据**：默认 `--data-path data/parquet_data`（研究 Parquet）；`--data-path none` 时用 `--live-root` 下的实盘存储。
- **特征**：按策略 `meta.yaml` 的 timeframe 分组，为每组 timeframe 建 `IncrementalFeatureComputer`，批量 `compute_features_dataframe(bars_1min, ticks_1min)`。
- **信号**：多策略时由 `LivePCM.decide(...)` 汇总；单策略时等价于注册一个策略的 PCM。各腿仍用 `GenericLiveStrategy` 的配置与管线。
- **持仓**：`PositionSimulator` 在每根 1m bar 上调用 **`position_logic` 与实盘相同的** `build_position_dict` / `enforce_position`（文档《事件驱动回测设计方案》中的「七步」与实现细节以代码为准）。
- **产物**：控制台报告、`--export` CSV、`--output` JSON、`--trading-map` HTML（Bokeh，可选依赖）等。

CLI 全量参数以 `python scripts/event_backtest.py --help` 为准；设计文档第六节为摘要。

---

## 三、`IncrementalFeatureComputer`（摘要）

**文件**：`src/time_series_model/live/incremental_feature_computer.py`

**作用**：在实盘流式场景下维护 VPIN、订单流等状态；在事件回测里主要使用其 **批量** 路径 `compute_features_dataframe`，与「逐 tick 跑 Nautilus 回测引擎」不是同一条链路。

**构造参数（与旧文档不同）**：除 `tick_window_minutes` / `bar_window_size` / VPIN 相关参数外，事件回测常用 **`primary_timeframe`** + **`archetypes_dir`**（与 `run_live` 一致），见 `EventBacktester.__init__` 内创建 FC 的代码。

**主要方法**（节选）：

- `compute_features_dataframe(bars_1min, ticks_1min, primary_timeframe=...)`：回测主路径。
- `on_tick` / `on_bar`：流式场景；Nautilus 的 `TradeTick` / `Bar` 在已安装 `nautilus_trader` 时可用，否则可用 dict。

---

## 四、WebSocket 客户端（`websocket_client.py`）

**文件**：`src/live_data_stream/websocket_client.py`

用于订阅 Binance 成交推送、解析为内部 tick 结构并回调。与 **事件回测脚本**无硬依赖；回测默认不经过 WebSocket。

---

## 五、与向量回测的关系

- **向量回测**：`backtest_execution_layer` 等，追求速度，路径与实盘有意的简化/对齐点不同。
- **事件回测**：追求与 `GenericLiveStrategy` + `LivePCM` + `position_logic` 一致，速度较慢。

若要对比向量与事件两条链路，仓库另有 `scripts/compare_vector_event_consistency.py` 等工具；`BacktestResult` **没有** `compare_with_vectorized(predictions.parquet)` 这类内置 API（以代码为准）。

---

## 六、注意事项

1. **依赖**：事件回测核心路径依赖 `pandas` / `numpy`；交易地图依赖 **Bokeh**（未安装时跳过 HTML 或报错取决于调用路径）。
2. **Nautilus**：增量特征模块可选导入 Nautilus 类型；**不要求**安装 Nautilus 才能跑 `scripts/event_backtest.py`。
3. **性能**：瓶颈通常在 IFC 批量特征计算；1m 持仓循环仅在「有持仓」的区间展开。
