# 事件驱动回测设计方案

> 目标：在本地用历史数据运行和实盘**完全相同**的逻辑，提前发现问题。

## 一、为什么需要事件驱动回测

### 现有系统对比

|              | 研究回测 (backtest_execution_layer) | 实盘 (order_flow_listener)                  |
| ------------ | ----------------------------------- | ------------------------------------------- |
| **特征来源** | predictions.parquet (ML模型预测)    | IFC 增量/批量计算                           |
| **信号判定** | entry_direction 列 + entry_filter   | GenericLiveStrategy.decide() 五步管线       |
| **出场粒度** | 4h bar 级别 + 1mins                 | 15min 特征周期 + 1mins + 实时价格检查       |
| **持仓管理** | 向量化 trailing 模拟                | 7步逐仓检查 (time/breakeven/trailing/SL/TP) |
| **仓位计算** | 固定 risk%                          | risk_per_slot × equity 动态反算             |

**核心问题**：向量回测和实盘用的是两套不同的代码路径，无法保证一致性。

### 事件驱动回测的定位

```
研究回测 ──── 快速迭代（向量化，秒级）── 策略开发/特征筛选
事件回测 ──── 实盘验证（逐bar，分钟级）── 上线前验证/问题诊断
```

## 三、性能设计（最高优先级）

### 3.1 为什么不能用原始 ticks

| 方案                     | 数据量 (6 symbol × 3 年) | 预估耗时     |
| ------------------------ | ------------------------ | ------------ |
| 原始 ticks (~1亿条)      | ~50 GB                   | 2-4 小时     |
| 1min bars (~9.5M条)      | ~1 GB                    | 5-15 分钟    |
| **信号时钟 + 1min bars** | ~2 GB                    | **2-5 分钟** |

### 3.2 两阶段分离策略

**关键洞察**：信号生成和持仓管理可以在不同的时间粒度上运行。

```
信号生成 ← 按信号时钟 (15min/4h) 遍历 features_df    → O(N_signals)
持仓管理 ← 按 1min bars 遍历（仅在有开仓时）          → O(N_1min × N_positions)
```

**性能优化要点：**

1. **特征预计算**：`IFC.compute_features_dataframe()` 一次性批量计算，不逐 tick 流式处理
2. **1min bars 只在有持仓时遍历**：没有开仓 → 直接跳到下一个信号时钟
3. **1min bars 内存映射**：按 symbol 分文件加载，不全部驻留内存
4. **避免 DataFrame 逐行迭代**：持仓管理部分用 numpy 数组 + 预编译的价格序列

### 3.3 预期性能

```
Step 1: 特征预计算  ~6s × 6 symbols = ~36s (可并行 → ~12s)
Step 2: 信号生成    ~5ms/决策 × ~3000 bar = ~15s (全部 symbols)
Step 3: 持仓管理    关键路径 — 见下方优化
Step 4: 报告生成    <1s

总计: 单策略 ~2-3 分钟, 三策略 PCM ~5-8 分钟
```

### 3.4 持仓管理：逐 1min bar 事件触发

**核心原则：事件回测 = 实盘逻辑的精确复现，不做向量化简化。**

如果用 numpy 向量化找 SL/TP 触发点，那就和研究回测没有区别了，
违背了事件回测的设计初衷。

**唯一方案：逐 1min bar 精确模拟**

```python
# 主循环：在信号时钟（如4h）上遍历 features
for signal_ts in signal_timestamps:
    # 1. decide() 产生信号
    intents = strategy.decide(features=features_at[signal_ts], symbol=sym)
    for intent in intents:
        simulator.open_position(intent, bar_at[signal_ts], features_at[signal_ts])
    
    # 2. 遍历到下一个信号时钟之间的所有 1min bars
    for bar_1min in bars_between(signal_ts, next_signal_ts):
        closed = simulator.update(bar_1min)  # 7步持仓管理
        for trade in closed:
            result.record(trade)
```

每根 1min bar 上执行的操作 = `_enforce_open_positions()` 的 7 步：
  ① Time stop (holding_expired)
  ② Breakeven lock (profit_r >= trigger_r → SL=entry)
  ③ High/Low water mark 更新
  ④ Activation trailing (profit_r >= activation_r)
  ⑤ SL hit (low <= sl 或 high >= sl)
  ⑥ TP hit (high >= tp 或 low <= tp)
  ⑦ Execute close

**性能仍然可控**：
- 只在有持仓时遍历 1min bars，无持仓直接跳到下一信号
- 典型场景：44笔BPC交易 × 平均持仓3000根1min bar = ~13万次 update 调用
- Python dict 操作 ~1μs/次，总计 ~0.13秒
- 真正的瓶颈是 Step 1 特征计算（~36s），持仓管理可忽略不计

## 四、核心类设计

### 4.1 PositionSimulator

```python
class PositionSimulator:
    """
    持仓模拟器 — 调用 position_logic（与实盘共享 enforce/build，而非复制 listener 整段）
    
    职责:
      1. 接收 TradeIntent → 开仓（build_position_dict）
      2. 在每根 1min bar 上检查持仓状态（enforce_position）
      3. 记录每笔交易的完整生命周期
    """
    
    def open_position(self, intent: TradeIntent, bar: dict, features: dict):
        """从 TradeIntent 创建虚拟持仓"""
    
    def update(self, bar_1min: dict) -> List[ClosedTrade]:
        """用 1min bar 更新所有持仓状态"""
    
    def force_close_all(self, price: float, reason: str) -> List[ClosedTrade]:
        """回测结束时关闭所有持仓"""
```

### 4.2 EventBacktester

```python
class EventBacktester:
    """
    事件驱动回测主类
    
    用法:
      bt = EventBacktester(strategies=["bpc", "fer", "me-long"], live_root="live/highcap")
      result = bt.run(symbols=["BTCUSDT", ...], days=90)
      result.print_report()
    """
    
    def __init__(
        self,
        strategies: List[str],
        live_root: str = "live/highcap",
        strategies_root: Optional[str] = None,
        db_path: Optional[str] = None,
        data_path: Optional[str] = None,
        fee_rate: float = 0.0,
    ):
        # 每策略 GenericLiveStrategy + LivePCM.register；按 meta.yaml 分 timeframe 建 IFC
    
    def run(self, symbols: List[str], days: int = 90, ...) -> BacktestResult:
        """
        核心循环（概念）:
          1. 加载 1min bars + ticks（研究目录或 live StorageManager）
          2. IFC 批量计算特征（多 timeframe 对齐到统一时间线）
          3. 按合并后的时钟推进：LivePCM.decide(...) → intents
          4. 开仓 / 在信号间隔内对 1m bar 调用 PositionSimulator.update
          5. 汇总 BacktestResult
        """
```

### 4.3 BacktestResult

```python
@dataclass
class BacktestResult:
    trades: List[ClosedTrade]       # 每笔交易详情
    funnel: Dict[str, int]          # 信号漏斗统计

    # 另有 kill_switch_stats、open_positions_end（跨月续跑）、funnel_per_bar 等 — 以代码为准
    
    def print_report(self):
        """控制台汇总"""
    
    def export_trades_csv(self, path: str):
        """导出逐笔交易明细（CLI --export）"""
```

## 五、与实盘代码的复用关系

| 实盘代码                                           | 事件回测复用方式                                   |
| -------------------------------------------------- | -------------------------------------------------- |
| `LivePCM.decide(...)`（多策略）                     | **直接调用** — 与实盘 PCM 仲裁一致                 |
| `GenericLiveStrategy.decide()`（单策略 PCM 内）   | **直接调用** — 与实盘单策略管线一致               |
| `execution_profile_apply.compute_rr_prices()`      | **直接调用** — SL/TP 计算一致                      |
| `execution_profile_apply.compute_trailing_stop()`  | **直接调用** — Trailing 计算一致                   |
| `execution_profile_apply.holding_expired()`        | **直接调用** — Time stop 一致                      |
| `slot_sizing.compute_slot_size_from_risk()`        | **直接调用** — 仓位计算一致                        |
| `ConstitutionExecutor.resolve_risk_for_strategy()` | **直接调用** — 每策略风险一致                      |
| `ConstitutionExecutor.validate_add_position()`     | **直接调用** — 加仓规则一致                        |
| `position_logic.enforce_position` / `build_position_dict` | **直接调用** → `PositionSimulator` 包装           |
| `IFC.compute_features_dataframe()`                 | **直接调用** — 特征计算一致                        |

**注意**：listener 层仍负责订单与 IO；**价格行为与持仓规则**在 `position_logic` 中与回测共用，避免两套漂移。

## 六、CLI 接口设计

以下为常用示例；**完整参数以** `python scripts/event_backtest.py --help` **为准**。

```bash
# 单策略
python scripts/event_backtest.py --strategy bpc --days 90

# 多策略（逗号分隔即启用 PCM 注册路径，无需单独 --pcm 开关）
python scripts/event_backtest.py --strategy bpc,fer,me-long --days 180

# 多 symbol
python scripts/event_backtest.py --strategy bpc --symbols BTCUSDT,ETHUSDT --days 180

# 研究数据目录（默认 data/parquet_data）；none = 使用 --live-root 下实盘存储
python scripts/event_backtest.py --strategy fer --data-path none --live-root live/highcap --days 30

# 导出 JSON / CSV、交易地图 HTML
python scripts/event_backtest.py --strategy bpc --days 90 --output out.json --export trades.csv \
  --trading-map map.html

# 地图上与向量 trades 叠加（蓝圆）— 指向向量回测导出的 trades CSV，不是 predictions.parquet
python scripts/event_backtest.py --strategy bpc --trading-map map.html \
  --compare-trades path/to/vector_trades.csv

# 跨月续跑 / 诊断
python scripts/event_backtest.py ... --resume-state prev.json --dump-end-state end.json
python scripts/event_backtest.py ... --no-kill-switch   # 仅诊断用
```

与向量回测的系统性对比可使用 `scripts/compare_vector_event_consistency.py`；`event_backtest.py` 不提供 `--compare predictions.parquet`。

核心能力（`PositionSimulator`、`EventBacktester`、`LivePCM`、多 timeframe IFC、JSON/CSV/地图、跨月 state、`--no-kill-switch` 等）已在 `scripts/event_backtest.py` 落地；后续以该文件与单测为准增量演进。向量化「快路径」若与 1m 精确更新冲突，以 **1m 精确** 为默认（参见脚本内 `--fast` 说明）。

## 八、验证标准

事件回测正确性的验证方法：

1. **信号对齐**：事件回测的 decide() 输出 vs `simulate_bpc_e2e.py --export-signals` 输出应完全一致
2. **SL/TP 精度**：用少量已知交易手动验证 SL/TP 触发价格和时间
3. **Trailing 行为**：对比向量回测的 TrailSL 出场笔数是否接近
4. **整体指标**：Sharpe / Win% / Mean R 与向量回测在合理偏差范围内（< 10%）
5. **极端情况**：
   - 连续开仓（同 symbol 多笔）
   - 跨日持仓
   - 无 ATR 时的 fallback
   - 数据缺失（1min bar 有 gap）
