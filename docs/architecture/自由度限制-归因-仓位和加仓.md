## 二、问题 1：要不要用 NN 多头模型同时交易多个 token？（像 CS）

### 结论先给：

> **现在阶段：不需要，也不应该**

### 为什么？

你要区分两种“多 token”：

#### ❌ CS（横截面）范式

* 同一时刻
* 所有 token 都有 score
* 统一排序 → 统一分仓

**前提条件是：**

* 同步信号
* 可比的 score
* 单一时间截面

你现在的现实是：

> ETH、BTC、SOL 信号 **时间不一致**

所以你现在做 **CS 是伪 CS**，自由度会爆炸。

---

### ✅ 你现在更适合的是：

> **多 token 的“异步时序策略池”**

即：

* 每个 token 是 **独立触发**
* 但 **共享一个风险预算池**
* 不做“全市场排序”

👉 所以答案是：

> **不用 NN 多头去直接交易多个 token（CS）**
> 而是用 NN 做 **路径/质量判断**，由规则层决定“要不要占坑”。

## 三、问题 2：如果交易多个 token，要不要排序？

### 关键结论：

> **异步信号 ≠ 可以排序**

你现在排序会犯一个致命错误：

> 用「不同时间、不同市场状态」下的分数做比较

### 举例：

* ETH 的 score 是在 **低波动、横盘 regime**
* BTC 的 score 是在 **突破初期、高波动 regime**

你拿这两个 score 排序，本质是：

> 🍎 和 🍊 比甜度

**这是无意义的自由度。**

---

### 正确做法不是“排序”，而是 **门槛制 + 占坑制**

你要从：

> ❌「谁更强」

转为：

> ✅「值不值得占一个坑位」

#### Step 3：对比机制（极其关键）

✅ 示例（任选其一）：

* 当前仓位是否 **跌破结构假设（path failure）**

**规则示例：**

> 若新信号的期望 R
> 明显 > 当前最弱仓位的剩余 R
> 且旧仓位已结构失败
> → 才允许替换

👉 这是**规则层决策**，不是 NN 决策。

# 一、最小可实盘系统图

##「2 坑位 · 可替换」Path–Router–Execution 架构

> 目标：
> **在多 token、异步信号的现实世界中，保证：**
>
> * 不抢跑
> * 不排序
> * 不全仓赌第一个信号
> * 所有决策都可解释、可回放

---

## 1️⃣ 系统角色划分（只保留必要模块）

```
Market Data
    ↓
Feature Builder
    ↓
Path Model（NN，多头）
    ├─ dir_prob
    ├─ E[R] (ex-ante)
    ├─ mfe_pred
    ├─ mae_pred
    └─ expected_T
    ↓
Router（规则）
    ├─ 是否满足入场门槛
    ├─ 是否值得“占坑”
    ↓
Position Manager（核心）
    ├─ Slot #1（固定风险）
    ├─ Slot #2（固定风险）
    └─ Replacement Judge
    ↓
Execution（严格执行，不决策）
```

---

## 2️⃣ Slot（坑位）是**系统的核心抽象**

### Slot 定义（非常关键）

* 系统 **最多 2 个并行仓位**
* 每个 Slot：

  * 固定 **风险预算**（不是资金）
  * 绑定一个 token
  * 有清晰生命周期

```
Slot {
  token
  entry_time
  entry_price
  path_snapshot   # 开仓时 NN 输出
  remaining_R     # 动态更新
  state           # Active / Weak / Failed
}
```

⚠️ **模型永远不知道 Slot 的存在**
Slot 是**宪法级别的约束**

---

## 3️⃣ Router 的唯一职责（被极度阉割）

Router **只回答一个问题**：

> 👉「这个信号**值不值得占一个坑位**？」

**Router 绝不：**

* 比较 token 强弱
* 动态分配仓位
* 做全局最优

---

## 4️⃣ 新信号到来时的完整流程

### 情况 A：有空 Slot

```
新信号 → 通过入场门槛 → 直接占 Slot
```

### 情况 B：没有空 Slot

```
新信号
   ↓
Replacement Judge
   ↓
【允许替换】 or 【拒绝】
```

⚠️ **这是系统中唯一允许“推翻过去”的地方**

# 二、仓位替换判定表（风控白皮书核心）

> 原则：
> **宁可错过，也不轻易反悔**

---

## 0️⃣ 前置原则（铁律）

* ❌ **新信号 ≠ 自动更优**
* ❌ **不因“未来可能更好”而砍仓**
* ✅ 只因 **旧仓位已经“逻辑破产”**

---

## 1️⃣ 仓位替换的三段式判定

### 第一关：新信号是否“够格参与比较”

| 条件                   | 说明     |
| -------------------- | ------ |
| `E[R]_new > R_min`   | 绝对质量门槛 |
| `mae_pred < MAE_cap` | 风险不过大  |
| `expected_T < T_max` | 不允许拖仓  |

👉 不过关 → **直接拒绝**，不进入替换流程

---

### 第二关：旧仓位是否“已经站不住脚”

只允许从以下 **可解释失败** 中选：

| 失败类型         | 判定                  |
| ------------ | ------------------- |
| Path Failure | 价格行为与路径假设冲突         |
| MAE Breach   | 实际回撤 > 预测 MAE       |
| Time Decay   | 已超过 expected_T 但无进展 |
| R Collapse   | 剩余 R < 初始 R × α     |

> ⚠️ 至少 **1 条成立**，否则**禁止替换**

---

### 第三关：保守优势比较（单维）

> 不允许多维评分，不允许加权

**唯一允许的形式：**

```
E[R]_new  >  Remaining_R_old × β
```

* β ≥ 1.2（强保守）
* Remaining_R_old 是“还能赚多少”，不是浮盈

---

## 2️⃣ 替换结果

| 结果   | 行为             |
| ---- | -------------- |
| 通过三关 | 平旧仓 → 立刻占 Slot |
| 任一失败 | 忽略新信号          |

👉 **系统宁可“慢”，也不“反复横跳”**

# 三、仓位宪法（这是最重要的部分）

这是你系统的 **不可修改条款**，
你以后所有策略、模型、token 都必须遵守。

---

## 📜 仓位宪法（Position Constitution v1.0）

### 第一条：坑位不可突破

> 系统最大并行仓位数 = **2**
> 不因行情、不因模型、不因自信而改变

---

### 第二条：模型无权分仓

> NN 不得：
>
> * 决定仓位大小
> * 决定是否替换
> * 决定并行数量

模型 **只提供证据，不做裁决**

---

### 第三条：替换必须“有罪”

> 任何仓位被平掉，
> **必须能指出一个明确的失败原因**

否则视为系统错误。

---

### 第四条：不排序，不预测未来信号

> 系统不做：
>
> * token 横截面排序
> * “未来可能更强”的假设
> * 全市场最优分配

---

### 第五条：宁可空仓，也不反悔

> 若无 Slot 且不满足替换条件：
>
> * 接受错过
> * 接受遗憾
> * 不修改规则

---

### 第六条：所有自由度必须可关闭

> 每一个新增机制：
>
> * 必须能单独开/关
> * 必须能回测其“有/无”的差异

# 一、先说结论为什么不是 10%

你现在的系统特征是：

* NN（非确定性）

---

## 1️⃣ 10% 风险在数学上已经不可控

假设（这是非常保守的）：

* 单笔最大失败 = -1R
* 连续失败概率 = 3 笔（在币圈是常态）
* 同时两个 Slot

### 回撤计算：

* 每 Slot 10%
* 两个 Slot 同时失败 → **-20%**
* 再来一轮替换失败 → **-30% ～ -40%**

你**心理、系统、执行**都会开始变形。

---

## 2️⃣ 替换机制 + 高风险 = 放大回撤

你设计的系统允许：

* 平掉旧仓 → 换新仓
* 失败信号被“确认后”才换

这在**逻辑上是对的**，
但在 **10% 风险下是灾难**：

> 一次错误替换 = 两次交易损失

---

## 3️⃣ NN 的尾部风险你无法准确建模

即使你现在 Sharpe > 0：

* NN 在 regime shift 时
* 会 **连续给出“看起来都合理”的错信号**

10% 风险会让一次结构性错误直接让你停机。

# 二、为什么 1%～2% 是“安全上限”

| 字段        | 含义    |
| --------- | ----- |
| Entry_R   | 初始风险  |
| Add_Count | 加仓次数  |
| Add_Risk  | 加仓总风险 |
| Max_R     | 最大浮盈  |
| Final_R   | 最终 R  |
| Drawdown  | 最大回撤  |
| Hold_Time | 持仓时间  |

### 核心对比指标（只看 3 个）

1️⃣ **尾部贡献率**

```
Tail_Contribution = Final_R - Initial_R
```

2️⃣ **风险效率**

```
Efficiency = Final_R / Max_Drawdown
```

3️⃣ **失败惩罚**

```
Loss_Multiple = 平均亏损 / 初始 R
```

👉 如果 **C > B > A** 在 1️⃣ 和 2️⃣
👉 且 C 的 3️⃣ 没爆
👉 人工加仓 = 真优势
