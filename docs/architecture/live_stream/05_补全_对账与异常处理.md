# 05 补全、对账与异常处理

主线目标：**先保证 1m 数据连续与系统模式安全，再做特征与交易**。

## 一、缺口补全（Gap Filling）

典型缺口来源：WebSocket 断连、进程重启、交易所延迟/乱序。

与当前实现一致的说明见：

- [`数据补全架构.md`](./数据补全架构.md)（Vision / Klines、`run_live.py` 后台重试、`GapFiller`）
- 代码：`src/live_data_stream/gap_filler.py`、`src/live_data_stream/order_flow_listener.py`

推荐流程（概念上与实现对齐）：

1. **检测**：按事件时间检查 1m 连续性；
2. **降级**：见 `SystemMode`（`NORMAL` / `DEGRADED` / `OFFLINE` / `ABNORMAL`）——异常下可能**禁止开新仓**；
3. **补全**：REST / Vision 拉缺口并写入 **Parquet**（非 QuestDB 默认路径）；
4. **重放/追赶**：用补全后的序列刷新 `IncrementalFeatureComputer` 所需窗口；
5. **恢复**：模式回到 `NORMAL` 后再恢复常规交易。

## 二、对账（Reconciliation）

两层含义：

- **市场数据对账**：本地 1m 序列是否连续、是否与磁盘一致；
- **交易状态对账**（启用 `OrderManager` 时）：订单/成交/持仓是否与交易所一致。

实现上依赖 `order_management` 与监听线程的配合；策略层仍建议保留**关键不变量断言**（例如持仓与信号方向一致性）。

## 三、异常发现（Detection）

建议最少覆盖：

- **Inf/NaN**：源数据 vs 计算产生（见 [`docs/features/DATA_MONITORING.md`](../../features/DATA_MONITORING.md)）；
- **乱序/重复**：同一 `(symbol, ts)` 重复 bar；
- **延迟抖动**：监控接收延迟分布；
- **模式异常**：`ABNORMAL` 等需人工复位路径（见 `system_mode.py`）。

## 四、异常处理与自愈（Handling）

与 `SystemMode` / `OrderFlowListener` 中的策略一致：**先稳后赚**——异常时限制开新仓、保留风控与补全；严重错误进入需人工介入的状态（见代码与 [`06_实盘稳定性运行手册.md`](./06_实盘稳定性运行手册.md)）。
