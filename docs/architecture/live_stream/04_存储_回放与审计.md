# 04 存储：回放与审计

要让「事件回测 ≈ 实盘」可验证、可追溯，需要把**市场输入**与关键**中间态**落到可重放的介质上。

## 一、当前主线的存储形态（Parquet）

本仓库实盘默认 **不以 QuestDB 为权威主存**；权威落盘由 **`StorageManager`（`src/live_data_stream/feature_storage.py`）** 完成，根路径由 **`run_live.py` 的 `--live-root`**（如 `live/highcap`）决定。

目录与用途的详细说明见：

- [`src/live_data_stream/README_STORAGE.md`](../../../src/live_data_stream/README_STORAGE.md)

典型内容包括：**1m ticks/bars**、按策略需要落盘的特征片段等（以代码与 README_STORAGE 为准）。

## 二、为什么「可回放」优先

为定位线上问题（特征异常、规则漂移、PCM 拒单等），需要：

- **复现**：用同一段落盘数据重放，对比当时特征与决策；
- **对账**：核对「看到的市场」与订单/持仓状态（若启用 `OrderManager`）；
- **审计**：回答「当时为何允许/拒绝交易」（特征快照 + `gate_ok` + funnel 等）。

## 三、写入一致性（At-least-once）

现实链路多为 **at-least-once**（可能重复推送）。要求是：

- **存储层可幂等**：对 bar 可用 `(symbol, timeframe, ts)` 去重；tick 需有稳定键或确定性「后写覆盖」策略；
- **计算层可去重**：重复事件下增量状态更新须确定（跳过/覆盖）。

## 四、回放最小闭环（工程上）

1. 从 `live_root` 读出 1m bar / tick（按事件时间排序）；
2. 用与实盘相同的 **`IncrementalFeatureComputer`** 更新状态；
3. 在信号时钟点输出特征快照与 `GenericLiveStrategy` / `LivePCM` 决策；
4. 与 `scripts/event_backtest.py` 或当时落盘的诊断结果做 diff（可选）。

## 五、建议记录的审计字段（低成本）

- **事件延迟**：接收时间 − 事件时间（仅监控，不驱动特征）；
- **补全标记**：是否经 `GapFiller` / Vision 补洞；
- **版本信息**：策略目录 Git SHA、模型/配置版本号（若有）。
