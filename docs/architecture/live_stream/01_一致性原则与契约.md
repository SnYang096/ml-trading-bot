# 01 一致性原则与契约

本项目同时存在两类计算形态：

- **训练/研究（向量化批处理）**：一次性加载较长区间数据，计算全量特征，训练模型。
- **事件回测/实盘（事件驱动增量）**：tick/bar 事件到达即处理，维护窗口与状态，输出“当前时点”的特征与信号。

要保证“回测 = 实盘”，关键不是复用同一个 DataFrame pipeline，而是让两条路径遵守同一份**特征契约与数据契约**。

## 一、数据契约（Data Contract）

- **统一事件语义**：回测与实盘必须使用同一种事件类型集合（`TradeTick`/`QuoteTick`/`Bar` 等）与同一含义的字段。
- **统一时间语义**：
  - 明确使用 **事件时间（event time）** 还是 **接收时间（ingest time）** 做计算驱动；默认应以事件时间为准。
  - 明确 bar 的时间戳含义（bar 开始/结束、是否对齐到交易所边界）。
- **统一清洗规则**：重复、乱序、缺失、异常值（inf/NaN/负量）处理策略必须一致。
- **统一 warmup**：任何需要历史窗口的特征，都必须定义 warmup 的最小窗口与“warmup 期间输出是否允许 NaN”。

## 二、特征契约（Feature Contract）

每个特征必须被定义为：

- **输入**：所需的事件流（tick/bar）、所需的历史窗口（长度、时间跨度）、所需的外部参数（如 VPIN bucket 体积）。
- **输出**：在“当前事件时间点”产出的一个或多个标量/向量（最终用于模型与交易）。
- **确定性**：在相同输入事件序列下，事件回测与实盘必须输出一致结果（允许仅由浮点误差造成的微小差异）。

建议对每个特征明确两种实现（可共存）：

- **Vectorized 实现（训练/研究）**：在 DataFrame 上批量计算（快、便于训练）。
- **Incremental 实现（事件回测/实盘）**：在事件流上增量更新（稳、低延迟）。

如果某个特征短期只有 vectorized 版本，则必须明确：

- 实盘/事件回测是否允许通过“滑动窗口 DataFrame”调用 vectorized 版本（通常会更慢、且更容易引入对齐差异）。
- 该特征是否会被纳入生产交易（生产交易优先使用 incremental 特征）。

## 三、一致性验证（必须有）

至少要具备以下三类验证：

- **Golden replay**：从同一份持久化事件（**本仓库主线为 `live_root` 下 Parquet**，由 `StorageManager` 等写入）回放，比较事件回测与实盘「同一时刻」的特征快照；可选再接外部时序库，但不是默认前提。
- **Index/time alignment check**：对齐检查（bar 边界、时区、缺口补全后重算一致）。
- **Data quality check**：在关键节点检测 inf/NaN/异常分布变化（参见 [`docs/features/DATA_MONITORING.md`](../../features/DATA_MONITORING.md)）。

## 四、工程落点（本项目现状）

- **实盘主线**：`scripts/run_live.py` → `OrderFlowListener` + **`IncrementalFeatureComputer`** + **`GenericLiveStrategy`**（配置驱动，多策略时 **`LivePCM`** 仲裁）；详见本目录 README 与 [`实盘特征计算机制.md`](./实盘特征计算机制.md)。
- **事件回测对齐**：`scripts/event_backtest.py` 与上述共享 **`position_logic` / `GenericLiveStrategy` 管线**（说明见 [`docs/architecture/event_drive_backtest/`](../event_drive_backtest/)）；**不是**以 Nautilus 回测引擎为默认主路径。
- **批处理特征主线**：训练侧 YAML + `FeatureComputer` / FeatureStore（与实盘增量路径需遵守同一特征契约）。
- **必须避免的误区**：把训练 DataFrame 的一次性计算方式直接当成 WebSocket 实时流的计算方式（延迟、状态、缺口处理会失真）。
