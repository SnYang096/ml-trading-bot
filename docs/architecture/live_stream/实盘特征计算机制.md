# 实盘特征计算机制

## 一、总览：两个关键数据结构

实盘特征系统围绕两个核心变量运转：

| 变量 | 类型 | 含义 |
|---|---|---|
| `live_feature_set` | `Set[str]` | 需要的**特征列名**集合（如 `atr`, `roc_20`） |
| `live_feature_nodes` | `List[str]` | 需要计算的**DAG 节点**列表（如 `atr_f`, `roc_f`） |

**关键约束**：`live_feature_set` 中每一列，必须有对应的节点在 `live_feature_nodes` 中（或由 OHLCV 原始列 / step 2 order flow 提供）。否则该列会在最终输出中缺失。

## 二、特征提取：从 YAML 到节点列表

入口函数：`extract_features_from_archetypes(archetypes_dir)`

位置：`src/time_series_model/live/live_feature_plan.py`

### 2.1 四层 YAML 扫描

```
archetypes/
├── gate.yaml          → _extract_features_from_gate()
├── evidence.yaml      → _extract_features_from_evidence()
├── entry_filters.yaml → _extract_features_from_entry_filters()
└── direction.yaml     → _extract_features_from_direction()
```

每层各自提取引用的特征列名（如 `roc_20`, `fer_trapped_longs_score`），合并为 `feature_columns` 集合。

### 2.2 Gate when 子句解析

gate.yaml 的规则结构支持嵌套逻辑：

```yaml
# 简单形式
when:
  roc_20:
    value_gt: 1.2654

# 复合形式 (FER prefilter)
when:
  all_of:
    - fer_trapped_longs_score:
        value_lt: 3.4965
    - fer_trapped_shorts_score:
        value_lt: 3.7941
```

解析函数 `_extract_features_from_when(when)` 递归遍历：
- 遇到**保留关键字** (`and`, `or`, `not`, `all_of`, `any_of`, `min_matches`, `min_matches_any`) → 递归进入子节点
- 遇到**其他键** → 视为特征列名

保留关键字定义：
```python
_WHEN_RESERVED = frozenset(
    {"and", "or", "not", "all_of", "any_of", "min_matches", "min_matches_any"}
)
```

> **曾经的 Bug**：旧代码只处理 `and`/`or`/`not`，遗漏 `all_of`/`any_of`，导致 FER gate 的 `all_of` 被当成特征名加入 `live_feature_set`，触发 `live_feature_missing: all_of` 警告。

### 2.3 列名 → DAG 节点映射

收集完所有特征列后：

1. 从 `feature_dependencies.yaml` 构建 **列名 → 节点** 反向索引
2. 每个列选择**输出最少的节点**（最精确匹配）
3. 汇总得到 `selected_nodes`

```
特征列          →  候选节点                    →  选中
atr             →  atr_f (输出1列)              →  atr_f ✓
roc_20          →  roc_f (输出2列)              →  roc_f ✓
soft_phase_*    →  bpc_soft_phase_f (输出26列)  →  bpc_soft_phase_f ✓
```

### 2.4 补充逻辑

在返回前，还有几个关键补充：

**a) ef_* 派生特征展开**

`_DERIVED_FEATURE_SOURCE_DEPS` 表记录 entry filter 动态特征的底层依赖：
```python
{
    "ef_liquidity_silence": ["vol_percentile_approx"],
    "ef_vol_regime_shift": ["bb_width_normalized_pct"],
}
```
如果 archetype YAML 引用了 `ef_liquidity_silence`，会自动把 `vol_percentile_approx` 和对应节点加入。

**b) OHLCV + atr 强制添加**

```python
live_feature_set |= {"open", "high", "low", "close", "volume", "atr"}
```

**c) atr_f 节点注入**

`atr` 列被加入 `live_feature_set`，但如果没有 archetype 直接引用 `atr` 列（比如 ME 策略的 gate/evidence 不引用 `atr`），`atr_f` 不会出现在 `selected_nodes` 中。

问题：`load_features_from_requested` **只返回被请求节点的输出列**，依赖节点的输出被静默丢弃。如果 `atr_f` 只是 `me_soft_phase_f` 的依赖而不在请求列表中，`atr` 列会被计算但不会出现在返回结果中。

解决：显式检查并注入：
```python
atr_candidates = col_to_nodes.get("atr")
if atr_candidates and not any(n for n, _ in atr_candidates if n in selected_nodes):
    best = min(atr_candidates, key=lambda x: x[1])
    selected_nodes.add(best[0])
```

## 三、特征计算：IFC 的 4 步流程

入口函数：`IncrementalFeatureComputer.compute_features_batch()`（下文简称 IFC）

位置：`src/time_series_model/live/incremental_feature_computer.py`

### 3.0 输入

- `bars_tf`：重采样后的大时间框架 bars（如 240T / 60T），包含 OHLCV 列
- `ticks_1min`：最近的 1min ticks 数据（来自磁盘）
- `live_feature_nodes`：需要计算的 DAG 节点列表
- `live_feature_set`：最终需要保留的特征列集合

### 3.1 Step 1: 重采样

将 1min bars 按 `primary_timeframe` 重采样为大 bars（如 240T, 60T）。

### 3.2 Step 2: Order Flow 特征

如果有 tick 数据，计算：
- VPIN / Trade Clustering（`extract_order_flow_features`）
- VPIN 派生特征
- Trade Cluster 派生特征
- Footprint 特征

这些特征通过 `join` 合并到 `bars_tf`。

### 3.3 Step 3: DAG 特征（核心复杂逻辑）

#### 3a. 分离 filtered / skipped

```
live_feature_nodes
    │
    ├─ tick 依赖？── 是 ─→ skipped（已在 Step 2 计算或不可用）
    │
    └─ 否 ─→ filtered（进入 first pass）
```

**tick 依赖检测**：`_detect_tick_dependent_nodes(feats_cfg)` 通过 `inspect.signature` 检查 compute_func 是否接受 `ticks` 或 `ticks_loader_json` 参数。递归沿依赖链传播。

#### 3a-fix: skipped 节点的非 tick 依赖加入 filtered

```
bpc_soft_phase_f (skipped — 依赖 vpin_features_f 是 tick 节点)
    ├── atr_f          (非 tick) → 加入 filtered ✓
    ├── bb_width_f     (非 tick) → 加入 filtered ✓
    └── vpin_features_f (tick)   → 不加入 ✗
```

目的：确保 second pass 时 `bpc_soft_phase_f` 的非 tick 输入已被 first pass 计算。

#### First Pass: load_features_from_requested

```python
bars_tf = self._feature_loader.load_features_from_requested(
    bars_tf, requested_features=filtered, fit=False
)
```

StrategyFeatureLoader 按 DAG 顺序计算 `filtered` 中的所有节点。

**关键行为（陷阱所在）**：

```python
# strategy_feature_loader.py line 742-743
all_cols = list(dict.fromkeys(list(df.columns) + output_cols))
all_cols = [c for c in all_cols if c in result_df.columns]
return result_df.loc[:, all_cols]
```

`output_cols` 只包含 **requested_features** 的输出列。依赖节点（如 `atr_f` 作为 `me_soft_phase_f` 的依赖被计算）的输出列**不在** `output_cols` 中 → 被静默丢弃。

这就是为什么 `atr_f` 必须显式出现在 `live_feature_nodes` 中。

#### 3b. Second Pass: 计算 skipped 中可运行的节点

```
skipped 节点
    │
    ├─ 是叶 tick 节点？── 是 ─→ 跳过（truly skip）
    │
    └─ 否（如 bpc_soft_phase_f）─→ 检查 required_columns
         │
         ├─ required_columns ⊆ 当前列 ─→ 进入 second pass ✓
         │
         └─ required_columns 不满足 ─→ 跳过
```

**准入检查只看 required_columns**，不合并 column_mappings。原因：

- `required_columns` = 必须有的列（如 close, high, low, atr, volume）
- `column_mappings` = 可选参数映射（如 `ofci_pct → ofci_pct`），compute_func 中这些参数有默认值 `None`

> **曾经的 Bug**：旧代码合并 `required_columns + column_mappings` 的所有列做检查。ofci_pct 是 tick 依赖产出，first pass 后不存在 → 检查失败 → bpc_soft_phase_f 无法进入 second pass → 26 个 BPC 特征全部缺失。

second pass 执行时，**column_mappings 过滤**：
```python
avail_mappings = {}
for param, src in raw_mappings.items():
    if isinstance(src, str) and src in bar_cols_updated:
        avail_mappings[param] = src
```
只保留 `bars_tf` 中实际存在的映射，缺失的跳过（函数用默认值 None）。

### 3.4 Step 4: 输出过滤

```python
if self.live_feature_set:
    keep = [c for c in bars_tf.columns if self._want(str(c))]
    bars_tf = bars_tf[keep]
```

只保留 `live_feature_set` 中的列。如果某列在 `live_feature_set` 但没被任何步骤计算出来 → `live_feature_missing` 警告。

## 六、归一化算法与 warmup 校验

### 6.1 `bounded_0_1` 是值域契约，不是算法

`feature_dependencies.yaml` 中的 `output_normalization_map` 标注的是**输出值域契约**（告诉下游这个列值在 [0,1]），
不指定具体算法。不同特征用不同算法达到 [0,1]：

| 算法类型 | 实现方式 | 代表特征 | 窗口 | warmup 风险 |
|---------|---------|---------|------|------------|
| 小窗口 rolling rank | `.rolling(40).rank(pct=True)` | `fer_signed_efficiency_pct` | 40 (min_periods=20) | **无** |
| 大窗口 rolling percentile | `.rolling(540).apply(percentile)` | `atr_percentile` | 540 (min_periods=540) | **高** |
| 大窗口 percentile_window | `percentile_window: 540` (yaml 参数) | `jump_risk_pct` | 540 | **高** |
| z-score → 分段线性映射 | `np.where(z < -2, ...)` | `vol_percentile_approx` | 20 | **无** |
| 显式 clip 归一化 | `np.clip(x, 0, 1)` | `wick_compression_score` | N/A | **无** |
| 固有有界 | R² 天然在 [0,1] | `trend_r2_20` | N/A | **无** |

### 6.2 `.rank(pct=True)` vs `.apply(percentile)` 区别

两者都计算滚动百分位排名，但不完全等价：

```python
# 方式 A: pandas 内置 (C 优化，快 ~50x)
ser.rolling(window, min_periods).rank(pct=True)
# ties 处理: 相同值取平均排名 → rank / count

# 方式 B: 自定义 apply (Python 回调，慢)
ser.rolling(window, min_periods=window).apply(
    lambda arr: float(np.mean(arr <= arr[-1]))
)
# ties 处理: 经验 CDF → P(X <= x)
```

| 维度 | `.rank(pct=True)` | `.apply(percentile)` |
|------|-------------------|----------------------|
| 性能 | C 优化，快 ~50x | Python 回调，慢 |
| ties | 平均排名 / count | CDF: count(<=x) / total |
| 数值 | 无 tie 时 ≈ 等价 | 有 tie 时略有差异 |
| NaN 产出 | min_periods 不满 → NaN | min_periods 不满 → NaN |

**决定 warmup 风险的是窗口大小，不是用 rank 还是 apply。**

### 6.3 元数据驱动的 warmup 校验

warmup 校验不能靠后缀猜测（`_pct` 曾导致 `fer_signed_efficiency_pct` 误报）。

当前方案从 `feature_dependencies.yaml` 元数据识别需要保护的特征：

```
_get_warmup_check_features() 判定规则:
  1. compute_params 包含 percentile_window >= 100
  2. 或 compute_func 名含 "percentile"
```

逐类型验证：

| 类型 | 代表 | yaml 特征 | 命中? | 需 warmup? | 判定 |
|------|------|----------|-------|-----------|------|
| 小窗口 rank | `fer_signed_efficiency_pct` | func 不含 percentile, 无 pct_window | 不命中 | 不需要 | 正确 |
| 大窗口 rank | `atr_percentile` | func=`compute_atr_percentile_from_series` | 命中 | 需要 | 正确 |
| 大窗口 param | `jump_risk_pct` | `percentile_window: 540` | 命中 | 需要 | 正确 |
| z-score 映射 | `vol_percentile_approx` | func 不含 percentile, 无 pct_window | 不命中 | 不需要 | 正确 |
| 显式 clip | `wick_compression_score` | func 不含 percentile, 无 pct_window | 不命中 | 不需要 | 正确 |

`_validate_warmup()` 在 `compute_features_batch`（实盘）和 `compute_features_dataframe`（事件回测）
中共享调用，确保两条路径的校验逻辑一致。

## 七、代码路径统一架构

重构后，特征计算只有一个核心实现：

```
_compute_features_core()          ← 共享核心 (steps 1-3)
    ├── Step 1: 重采样 1min → 大 bars
    ├── Step 2: Order Flow (tick 数据)
    └── Step 3: StrategyFeatureLoader + second pass
         │
         ├── compute_features_batch()     ← 实盘入口
         │     ├── Step 4: 提取最后一行 → Dict[str, float]
         │     └── Step 5: _validate_warmup()
         │
         └── compute_features_dataframe() ← 事件回测入口
               ├── Step 4: 过滤列 → DataFrame
               └── Step 5: _validate_warmup()
```

- **之前**: batch 和 dataframe 各 ~350 行重复代码，dataframe 无 warmup 校验
- **之后**: 共享 `_compute_features_core()`，净减 ~200 行，两路径校验一致

## 九、关键陷阱速查

1. **load_features_from_requested 只返回 requested 的输出列**
   - 依赖节点的输出被静默丢弃
   - 任何需要出现在最终结果中的列，对应节点必须在 `live_feature_nodes` 中

2. **gate when 子句的逻辑运算符必须在 _WHEN_RESERVED 中**
   - 新增逻辑运算符关键字时必须同步更新
   - 否则会被误识别为特征列名

3. **column_mappings 是可选的，required_columns 是必须的**
   - compute_func 中 column_mappings 对应的参数应有默认值 None
   - 准入检查不应因可选参数缺失而阻止节点

4. **tick 依赖是传递的**
   - 节点 A 依赖节点 B，B 是 tick 依赖 → A 也是 tick 依赖
   - 但 A 的其他非 tick 依赖（如 atr_f）需要 3a-fix 加入 first pass

5. **ef_* 派生特征需要手动注册底层依赖**
   - `_DERIVED_FEATURE_SOURCE_DEPS` 字典维护映射
   - archetype YAML 只引用 ef_* 名称，自动检测无法发现底层 DAG 依赖

6. **warmup 校验靠元数据不靠后缀**
   - `_pct` 后缀可能是小窗口 rank（不需要 warmup）也可能是大窗口 percentile_window（需要）
   - 判定依据：`percentile_window >= 100` 或 `compute_func` 含 "percentile"
   - 新增归一化类型时，确保大窗口特征的 yaml 配置包含上述标识

7. **`bounded_0_1` 是值域契约，不是算法**
   - 不同特征用不同算法达到 [0,1]（rank / CDF / clip / z-score 映射 / 天然有界）
   - 不可根据 normalization_map 推断 warmup 需求
