# 06 实盘稳定性运行手册

上线运维的最低清单：在断网、延迟、缺口、异常值、交易所异常下系统仍可控。

## 一、启动流程（须可重复）

1. **加载策略与配置**（`config/strategies/...`、`config/constitution/...` 等），确认 **`--live-root`** 与 symbols。
2. **Warmup**：从 **`StorageManager` Parquet** 加载历史 1m ticks/bars（不足时由 `GapFiller` / API / Vision 补洞，见 [`数据补全架构.md`](./数据补全架构.md)）。
3. **质量闸门**：关键特征缺失率、inf 检查；`SystemModeManager` 根据 **1m bar 条数** 判定 `OFFLINE` / `DEGRADED` / `NORMAL`（常量见 `src/live_data_stream/system_mode.py`）。
4. **订阅 WebSocket**（`BinanceWebSocketClient`）并进入主循环（`run_live.py`）。
5. **NORMAL**：满足连续性与模式条件后，由 `OrderFlowListener` 放开常规交易路径（具体以代码为准）。

## 二、在线健康指标（Observability）

建议记录/告警：

- **数据连续性**：缺口次数、补全耗时、Vision 后台重试结果；
- **延迟**：事件时间 vs 接收时间（仅监控）；
- **特征质量**：`live_feature_missing`、关键列 NaN 比例；
- **交易一致性**（若启用 OM）：拒单、超时、对账失败次数。

可选：`StatsCollector` / metrics 导出（见 `run_live.py` 与 `metrics_exporter`）。

## 三、降级与熔断

与 `SystemMode` 对齐的实用原则：

- **DEGRADED**：观察为主、限制开新仓；继续补全与追赶；
- **ABNORMAL**：停止新单、需人工复位（见 `system_mode.py` 与 listener 内触发条件）；
- **HALT**（业务语义）：等同「主动停机/停策略进程」，由运维执行。

## 四、演练建议

- WebSocket 断连与自动重连；
- 进程重启 + warmup + 尾部 gap；
- 关键特征列缺失或 inf 风暴；
- 交易所 API 限速（补全退避）。

## 五、事故排查最短路径

1. 当前 **`SystemMode`**？是否正在补洞 / 降级？
2. **`live_root` Parquet** 是否缺段、尾部是否未跟上？
3. **特征**：缺失列、NaN、与 `实盘特征计算机制.md` 中 DAG 是否一致？
4. **PCM / 宪法**：拒单原因、funnel 字段（若有落盘）；
5. 必要时用 **`scripts/event_backtest.py`** 同区间复现。
