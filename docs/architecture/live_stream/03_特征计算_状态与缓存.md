# 03 特征计算：状态与缓存

本章总结“实盘/事件回测”的特征计算机制，目标是：**低延迟、可恢复、与回放一致**。

## 一、两条计算路径与定位

- **训练（向量化批处理）**：`StrategyFeatureLoader` / `FeatureComputer`（YAML 依赖 + 并行 + 缓存）。
  - 优点：快、适合整段数据。
  - 局限：天然无状态、实时流并行/缓存粒度不合适。
- **事件回测/实盘（事件驱动增量）**：以“事件 → 状态更新 → 当前特征快照”为主线。
  - 推荐围绕增量状态对象实现（例如 **`IncrementalFeatureComputer`** + 实盘侧 **`GenericLiveStrategy`** / `OrderFlowListener` 驱动）。

关键原则：**生产交易（实盘/事件回测）优先走增量路径**；训练可以走向量化路径，但必须对齐特征契约（见 `01_一致性原则与契约.md`）。

## 二、状态管理（State Management）

实时流/事件回测的特征计算必须维护状态，典型包括：

- **滑动窗口**：最近 N 条 bar、最近 X 分钟 tick。
- **聚合桶**：VPIN buckets、订单流统计桶。
- **跨时间框架缓存**：15T、1H 等多时间框架的“各自窗口与最后更新时间”。

状态管理的工程要求：

- **可序列化**：用于进程重启恢复（至少可从存储重放恢复到一致状态）。
- **可重放**：给定同一事件序列，状态演化确定。
- **有 warmup 定义**：输出 NaN 是否允许、何时开始产出有效特征。

## 三、缓存策略（Cache）

向量化训练路径可以使用多层缓存（内存/磁盘/按月缓存），但实盘不应依赖“整段 DataFrame 哈希缓存”，原因：

- 单条/小批数据缓存命中率低；
- hash 粒度与“增量更新”的复用方式不匹配；
- 缓存一致性难以保证（尤其在补全/重放后）。

实盘更合适的缓存是：

- **状态即缓存**：窗口、累积统计、桶状态就是“可复用计算结果”。
- **小粒度 memoization**：对昂贵但可增量更新的子计算做局部缓存。

## 四、fit/normalization 的一致性

训练阶段可能存在 `fit=True`（拟合归一化/分位数等参数），实盘必须确保：

- **实盘永远不“再 fit”**：只加载训练导出的参数/模型。
- **参数版本绑定**：模型版本、特征版本、归一化参数版本必须绑定并可追溯。

这类规则是“回测/实盘一致”最容易被破坏的点之一。

## 五、质量监控（Inf/NaN/异常值）

生产交易必须具备“数据质量闸门”：

- **源数据质量监控**：加载/重采样后检查 inf（源数据问题）。
- **计算质量监控**：特征计算后检查新增 inf（计算问题）。

参考：[`docs/features/DATA_MONITORING.md`](../../features/DATA_MONITORING.md)。

建议把监控结果与运行策略绑定：

- 轻微异常：记录 + 降级（跳过交易/跳过部分特征）。
- 严重异常：熔断（停止开新仓，等待恢复/人工介入）。
