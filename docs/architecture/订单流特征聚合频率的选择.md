# 订单流特征聚合频率的选择：1秒 vs 1分钟

> **文档创建时间**: 2026-01-11  
> **核心问题**: 订单流特征是用1秒聚合好，还是1分钟聚合？如果用1分钟，会不会导致VPIN、trade_cluster、footprint失效？

---

## 📋 总纲

### 核心答案

> **推荐：1分钟聚合（对于你的"4H分析 + 15分钟执行"系统）  
> VPIN和trade_cluster可以用1分钟聚合，footprint本来就适合1分钟聚合。  
> 1秒/100ms聚合在几年回测中很慢，而且对于4H级别的策略来说，增益有限。**

---

## ⚠️ 不同特征对聚合频率的敏感性

### 特征1：Footprint（足迹）

#### Footprint的定义

```yaml
Footprint的核心:
  - 测量"某个价格水平的累积成交"
  - 时间粒度: 1分钟或5分钟的bar（标准做法）
  - 观察对象: 某个价格水平的"累积成交"
  
典型用法:
  - "大块吸收"（Large absorption）
  - "价格在某个位置被大量订单吸收"
```

#### Footprint的聚合频率要求

```yaml
Footprint的聚合频率:
  
  标准做法:
    - ✅ 1分钟聚合（推荐）
    - ✅ 5分钟聚合（也可以）
    - ⚠️ 1秒聚合（可能太细，不必要）
    - ❌ 100ms聚合（过度，没必要）
  
  原因:
    - Footprint看的是"累积成交"，不需要秒级精度
    - 1分钟聚合已经足够捕捉"大块吸收"
    - 更细粒度（1秒/100ms）反而增加噪声
  
  结论:
    - ✅ 1分钟聚合完全足够
    - ✅ 不会因为用1分钟聚合而失效
```

**结论**：Footprint用1分钟聚合是标准做法，不会失效。

### 特征2：VPIN（Volume-Synchronized Probability of Informed Trading）

#### VPIN的定义

```yaml
VPIN的核心:
  - 测量"知情交易者"的占比
  - 计算方式: 按volume bucket计算imbalance
  - 典型bucket大小: 每50 BTC一个bucket（可调）
  
计算方法:
  1. 把volume切成固定的buckets（比如每50 BTC一个）
  2. 每个bucket里，计算buy/sell的不平衡
  3. 不平衡越大 → 越可能有"知情者"在单向交易
  
例子:
  Bucket 1: buy=40, sell=10 → imbalance=0.6
  Bucket 2: buy=38, sell=12 → imbalance=0.52
  Bucket 3: buy=45, sell=5  → imbalance=0.8
  
  VPIN = avg(0.6, 0.52, 0.8) = 0.64
```

#### VPIN的聚合频率要求

```yaml
VPIN的聚合频率:
  
  关键点:
    - VPIN按volume bucket计算，不是按时间bucket
    - 时间粒度只是"如何切tick数据"，不影响bucket定义
  
  可以用1分钟聚合吗？
    - ✅ 可以（推荐）
    - 原因：1分钟bar内的tick数据，可以按volume bucket聚合
    - 但需要保存tick的buy/sell标识（不能只存总量）
  
  1秒聚合 vs 1分钟聚合:
    - 1秒聚合: 每个1秒bar内的tick → 按volume bucket聚合
    - 1分钟聚合: 每个1分钟bar内的tick → 按volume bucket聚合
    - 只要bucket定义相同（如每50 BTC），结果应该相似
  
  结论:
    - ✅ 1分钟聚合可以用
    - ✅ 只要在1分钟bar内按volume bucket计算即可
    - ⚠️ 但如果1分钟bar内的volume < 50 BTC，可能需要调整bucket大小
```

#### VPIN用1分钟聚合的实现

```python
# VPIN用1分钟聚合的实现
def compute_vpin_1min_bars(bars_1min, bucket_size=50):
    """
    参数:
      bars_1min: 1分钟bar数据，每个bar包含：
        - timestamp
        - volume (total)
        - buy_volume (cumulative)
        - sell_volume (cumulative)
      bucket_size: volume bucket大小（如50 BTC）
    
    实现:
      1. 将1分钟bar内的tick按volume bucket聚合
      2. 计算每个bucket的imbalance
      3. 取平均得到VPIN
    """
    buckets = []
    current_bucket = {'buy': 0, 'sell': 0, 'volume': 0}
    
    for bar in bars_1min:
        # 假设bar内tick已聚合为buy_volume和sell_volume
        bar_buy = bar['buy_volume']
        bar_sell = bar['sell_volume']
        bar_total = bar_buy + bar_sell
        
        # 按volume bucket切分
        while bar_total > 0:
            remaining = bucket_size - current_bucket['volume']
            take_buy = min(remaining, bar_buy)
            take_sell = min(remaining - take_buy, bar_sell)
            
            current_bucket['buy'] += take_buy
            current_bucket['sell'] += take_sell
            current_bucket['volume'] += take_buy + take_sell
            
            bar_buy -= take_buy
            bar_sell -= take_sell
            bar_total -= (take_buy + take_sell)
            
            if current_bucket['volume'] >= bucket_size:
                # 计算bucket imbalance
                imbalance = abs(current_bucket['buy'] - current_bucket['sell']) / bucket_size
                buckets.append(imbalance)
                current_bucket = {'buy': 0, 'sell': 0, 'volume': 0}
    
    # VPIN = 平均imbalance
    vpin = mean(buckets) if buckets else 0.0
    return vpin
```

**关键点**：
- VPIN按volume bucket计算，不是按时间bucket
- 只要在1分钟bar内保存buy_volume和sell_volume，就可以按volume bucket聚合
- 1分钟聚合不会导致VPIN失效

### 特征3：Trade Clustering（交易聚类）

#### Trade Clustering的定义

```yaml
Trade Clustering的核心:
  - 测量"成交的时间分布模式"
  - 时间粒度: tick级别（每笔成交）
  - 观察对象: 成交的"时间分布模式"
  
典型用法:
  - "这些订单是连续到达的，还是分散的？"
  - "是一笔大单，还是多笔小单？"
  - "时间间隔是否有规律？"
```

#### Trade Clustering的聚合频率要求

```yaml
Trade Clustering的聚合频率:
  
  关键点:
    - Trade Clustering需要tick的时间戳（精确到毫秒/秒）
    - 需要看"时间分布模式"（不是累积量）
  
  可以用1分钟聚合吗？
    - ⚠️ 可能丢失部分信息（时间分布模式）
    - 但可以用"1分钟bar内的统计量"代替
    - 例如：bar内的tick数量、平均时间间隔、最大时间间隔
  
  1秒聚合 vs 1分钟聚合:
    - 1秒聚合: 可以看1秒内的tick分布模式
    - 1分钟聚合: 只能看1分钟内的统计量（tick数量、平均间隔）
    - 可能丢失"秒级的时间分布模式"
  
  结论:
    - ⚠️ 1分钟聚合可能丢失部分时间分布信息
    - ✅ 但可以用"1分钟bar内的统计量"代替（tick数量、平均间隔）
    - ✅ 对于4H级别的策略，可能足够
```

#### Trade Clustering用1分钟聚合的实现

```python
# Trade Clustering用1分钟聚合的实现
def compute_trade_cluster_1min_bars(bars_1min):
    """
    参数:
      bars_1min: 1分钟bar数据，每个bar包含：
        - timestamp
        - tick_count (bar内的tick数量)
        - volume (total)
        - avg_tick_interval (平均tick间隔，可选)
    
    实现:
      1. 用bar内的tick_count作为"clustering"指标
      2. 或用avg_tick_interval作为"时间分布"指标
      3. 或用volume/tick_count作为"单笔大小"指标
    """
    cluster_features = []
    
    for bar in bars_1min:
        tick_count = bar['tick_count']
        volume = bar['volume']
        
        # 特征1: tick密度（高密度 = 强clustering）
        tick_density = tick_count / 60.0  # 每秒钟的tick数
        
        # 特征2: 平均单笔大小（小单多 = 强clustering）
        avg_tick_size = volume / tick_count if tick_count > 0 else 0
        
        # 特征3: 时间分布（如果有avg_tick_interval）
        # 小间隔 = 强clustering
        clustering_score = compute_clustering_score(
            tick_density,
            avg_tick_size,
            bar.get('avg_tick_interval', None)
        )
        
        cluster_features.append(clustering_score)
    
    return cluster_features
```

**关键点**：
- Trade Clustering需要时间分布模式，1分钟聚合可能丢失部分信息
- 但可以用"1分钟bar内的统计量"代替（tick数量、平均间隔）
- 对于4H级别的策略，可能足够

## ✅ 推荐方案：1分钟聚合

### 为什么推荐1分钟聚合？

#### 1. 回测性能

```yaml
回测性能对比（几年数据）:
  
  100ms聚合:
    - 数据量: ~3.15亿条/年
    - 回测速度: 极慢（可能需要几天）
    - 问题: 数据量过大，计算开销高
  
  1秒聚合:
    - 数据量: ~3150万条/年
    - 回测速度: 很慢（可能需要几小时）
    - 问题: 数据量仍然很大，计算开销高
  
  1分钟聚合:
    - 数据量: ~52.5万条/年
    - 回测速度: 中等（可能需要几十分钟）
    - 优势: 数据量合理，计算开销可接受
```

#### 2. 特征质量（对于4H级别策略）

```yaml
特征质量对比（对于4H策略）:
  
  Footprint:
    - 100ms/1秒: 过度精确，不必要
    - 1分钟: ✅ 足够（标准做法）
    - 5分钟: ✅ 也可以
  
  VPIN:
    - 100ms/1秒: 过度精确，不必要
    - 1分钟: ✅ 可用（按volume bucket聚合）
    - 5分钟: ⚠️ 可能丢失信息（如果5分钟volume < bucket_size）
  
  Trade Clustering:
    - 100ms/1秒: 精确，但4H策略不需要
    - 1分钟: ⚠️ 可能丢失部分时间分布，但可用统计量代替
    - 5分钟: ❌ 丢失较多
```

#### 3. 你的系统特点（4H分析 + 15分钟执行）

```yaml
你的系统特点:
  - 4H分析（Archetype识别）
  - 15分钟执行
  - Mean策略依赖"吸收语义"
  
  对聚合频率的要求:
    - ✅ 不需要秒级精度（4H策略）
    - ✅ 需要捕捉"吸收语义"（1分钟足够）
    - ✅ 回测性能重要（几年数据）
  
  结论:
    - ✅ 1分钟聚合最适合
```

数学上它就是：

[
FP(p) = \sum_{t \in \Delta T} V_t(p)
]

注意这个求和：

* 没有顺序
* 没有时间权重
* 只有“在这个区间内”

---

### 为什么 1min 不会损失信息？

因为你真正关心的是：

* 某个 price level 是否被大量成交“填满”
* 是否出现 **absorption**

而 absorption 的定义是：

> **大量 volume 没有推动价格穿越**

这在数学上是：

```text
volume large
Δprice small
```

👉 **这是一个区间性质，不是瞬时性质**

所以：

* 1 秒内发生 200 笔
* 1 分钟内发生 200 笔

**对 absorption 的判断是等价的**

---

## 3️⃣ CVD / VWAP：它们是严格的积分量

这两个最简单：

### CVD

[
CVD(t) = \sum_{i=1}^{N(t)} sign_i \cdot volume_i
]

这是一个**纯积分**

积分有一个非常重要的性质：

> **你怎么分段，结果都一样**

tick → 1min 只是改变积分的“分段方式”，
不改变结果。

---

### VWAP

[
VWAP = \frac{\sum p_i v_i}{\sum v_i}
]

你在 tick 上算
你在 1min bar 上算

**完全等价**

换句话说：

# 三、为什么“60 倍压缩 ≠ 60 倍信息损失”

这里是整个问题的核心数学原因。

---

## 信息是否丢失，取决于：

> **你的特征是否依赖于“微观顺序”**

### 两类特征的本质差异

| 特征类型                         | 是否怕聚合 | 原因            |
| -------------------------------- | ---------- | --------------- |
| 积分型（sum, imbalance）         | ❌ 不怕     | 顺序无关        |
| 密度型（count / volume density） | ❌ 不怕     | 只看总量        |
| volume-bucket 型                 | ❌ 不怕     | bucket 不按时间 |
| 顺序型（LSTM / Hawkes）          | ✅ 怕       | 依赖 tick 顺序  |

👉 你现在**没有**在用最后一类。

---

## 用信息论语言说一句狠话

你现在做的是：

> **先人为定义了 sufficient statistics**
> 然后再问“是否丢失信息”

答案是：

> **你丢弃的是你本来就不打算用的信息**
