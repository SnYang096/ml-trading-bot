# 如何把树模型导出的 if/else 规则，对应到我目前的四种 execution（TC / TE / FR / ET）？

你这段整理**方向是对的，而且已经到“可以工程化”的阶段**。
我直接按你关心的核心问题来回答：

> **我如何把树模型导出的 if/else 规则，对应到我目前的四种 execution（TC / TE / FR / ET）？**

关键结论先给出一句话版：

> **树模型规则 ≠ Execution 触发器
> 树模型规则 =「语义证据 → Execution 的 gate / bias / risk modifier」**

下面我给你一套**可直接落地的映射方法**，不是泛泛而谈。

---

## 一、先统一语言：4 种 Execution 的“语义前提”

先把 TC / TE / FR / ET 的**“成立条件”说清楚**，这样规则才有地方放。

| Execution                 | 核心语义         | 本质问题                           |
| ------------------------- | ---------------- | ---------------------------------- |
| **TC** Trend Continuation | 趋势已成立，继续 | 趋势是否仍然**健康**               |
| **TE** Trend Expansion    | 趋势刚放大/加速  | 是否存在**点火/放量/结构突破**     |
| **FR** Failure Reversion  | 突破失败后的回归 | 是否存在**假突破 / 吸收 / 背离**   |
| **ET** Exhaustion Turn    | 趋势末端反转     | 是否存在**衰竭 / 拥挤 / 动能耗尽** |

👉 树模型导出的规则，**几乎全部在回答这四个“是否”问题之一**。

---

## 二、把树规则“翻译”为 4 类 Execution 的 Gate / Bias

下面是你给的规则中，**真正有价值、且能稳定泛化的映射**。

---

## 1️⃣ Trend Continuation（TC）

### TC 的“允许条件”

趋势要能继续，至少要满足：

* 订单流 **没有明显背离**
* 波动率 **不是衰竭态**
* 资金/情绪 **不是 exhaustion**

### 来自树模型的 TC 否决证据（Gate）

来自 `trend_following` 的强负规则，本质都是：

> **“趋势已经不健康了”**

#### 典型规则 → 抽象

```text
cvd_long <= very_negative
AND
bb_width_normalized low
```

抽象为：

```python
def gate_tc(features):
    if (
        orderflow_long_is_weak(features) and
        volatility_is_compressing(features)
    ):
        return VETO_TC
```

#### 你给的真实例子

```text
bb_width_normalized <= 4.86 
AND cvd_long <= -34895
```

**语义解读**：

> “波动不放大 + 长期订单流持续负 → 趋势已经死了”

✅ **TC Gate（否决）**
❌ 不要当作 TC 的 entry 条件

---

## 2️⃣ Trend Expansion（TE）

TE 是 **最容易被“伪信号”污染** 的 execution，所以树规则在这里价值最大。

### TE 的正向证据（Bias）

来自 **compression_breakout / sr_breakout** 的正规则：

#### 典型规则族

```text
cvd_change_5 > X 
AND volume_ratio > Y
```

你给的 top rule：

```text
cvd_change_5 > 4805 
AND volume_ratio > 1.87
```

### 抽象成 TE 的 “Expansion Score”

```python
te_score = f(
    normalized_cvd_change,
    volume_surge,
    structure_break
)
```

用于：

* 提高 TE execution 权重
* 允许更激进的 size / pyramid

❗**注意**：
这不是“只要满足就开 TE”，而是：

> **只有在 Router 选 TREND 时，TE 才可能被允许**

---

## 3️⃣ Failure Reversion（FR）

你这里的树规则 **含金量非常高**，而且你已经抓到重点了。

### FR 的核心语义

> **“看起来像突破，但底层订单流/结构不支持”**

### 来自树模型的 FR 证据

主要集中在：

* `sr_reversal_rr_reg_long`
* `sr_breakout` 的负规则

#### 典型规则

```text
cvd_long very negative
AND price not collapsing
```

或

```text
liquidity_void_retracement small
AND cvd_change negative
```

### 抽象为 FR Gate

```python
def gate_fr(features):
    if (
        breakout_attempt_detected(features) and
        orderflow_divergence(features)
    ):
        return ALLOW_FR
```

📌 你提到的这一条非常典型：

```text
liquidity_void_retracement <= 0.037 
AND cvd_change_5 <= -848
```

**语义**：

> “流动性真空拉了一下，但没有真实买盘 → 假突破概率高”

✅ FR **允许信号**
❌ 不要给 TC / TE

---

## 4️⃣ Exhaustion Turn（ET）

ET 是最少、但**最危险**的 execution。

### ET 的唯一可靠来源：**语义特征**

你自己也发现了：

> 只有 trend_following 里出现了 funding_*_scene_score

这是非常重要的信号。

### ET 的核心触发语义

来自你列的规则：

```text
funding_exhaustion_scene_score > threshold
AND macd_signal weak
```

### 抽象为 ET 的 “Exhaustion Probability”

```python
p_exhaustion = g(
    funding_exhaustion_score,
    momentum_decay,
    volatility_peak
)
```

用于：

* 允许 ET execution
* 或对 TC / TE 做强 veto

⚠️ **原则**：

> 没有“语义 exhaustion”，不要做 ET
> 树模型中只有 funding_* 是合格的 exhaustion 语义

---

## 三、你问得非常对的一点：这些规则放哪一层？

你自己的总结已经接近满分了，我只帮你**定死边界**：

### ❌ 不应该做的

* ❌ 把树规则直接当 execution entry
* ❌ 照抄阈值上线
* ❌ 用树规则替代 Router / NN

### ✅ 正确用法（强烈推荐）

| 层级               | 树规则角色                   |
| ------------------ | ---------------------------- |
| **Gate 层**        | 否决危险状态（最重要）       |
| **Router 边界**    | 调整 TREND / MEAN 权重       |
| **Execution Bias** | 提高/降低某 archetype 的概率 |
| **Risk Modulator** | 调 size / SL / pyramid       |

# 二、把 funding / cvd / liquidity 统一成 **Regime Embedding**

你现在的问题是：

> 每个 execution 都在看不同“语义碎片”，怎么统一？

答案是：
**做一个低维语义 embedding，而不是堆规则。**

---

## 1️⃣ Regime Embedding 定义（8~12 维）

```text
R = [
  flow_strength        # cvd, delta, aggressive volume
  flow_directionality  # imbalance / consistency
  liquidity_state      # void / absorption
  volatility_state     # compression / expansion
  participation        # volume regime
  crowding             # funding, OI skew
  exhaustion           # funding_scene, momentum decay
  structural_validity  # breakout vs fake
]
```

每一维 ∈ [-1, +1] 或 [0,1]

---

## 2️⃣ 这些维度怎么来？

| 维度                | 来源                         |
| ------------------- | ---------------------------- |
| flow_strength       | cvd_long, cvd_change         |
| flow_directionality | cvd slope, imbalance         |
| liquidity_state     | void_retracement, absorption |
| volatility_state    | bb_width, atr_z              |
| participation       | volume_ratio                 |
| crowding            | funding_rate_percentile      |
| exhaustion          | funding_scene_score          |
| structural_validity | breakout + orderflow         |

---

## 3️⃣ Execution = Regime 的不同“投影”

```python
tc_score = dot(R, W_tc)
te_score = dot(R, W_te)
fr_score = dot(R, W_fr)
et_score = dot(R, W_et)
```

Router / Execution **不再 directly 用原始特征**。

---

## 4️⃣ 你会发现一个重要现象

> **树模型规则 ≈ Regime Embedding 的 axis-aligned slice**

这就是为什么树能“解释”，但 NN 更适合长期。

# 三、你问得最尖锐的问题：

## 为什么树模型里一个数学特征都没有？WPT / Hurst / Hilbert / Spectrum 呢？

### 答案一句话：

> **不是它们“没测试”，而是它们在“真实交易语义”上几乎不成立。**

下面是专业解释。

---

## 1️⃣ 树模型最怕什么特征？

> **怕“连续变化但语义不稳定”的特征**

而你列的这些，全中：

| 特征     | 致命问题                 |
| -------- | ------------------------ |
| Hurst    | 非平稳 + 窗口依赖        |
| Hilbert  | 相位在 regime 切换时跳变 |
| Spectrum | 对采样 / 微结构极其敏感  |
| WPT      | 能量不守恒、尺度不稳     |

树模型需要的是：

> **稳定的 if → then 语义**

---

## 2️⃣ 为什么这些特征在回测可能“看起来有效”？

因为：

* 它们**捕捉的是“价格形态”**
* 不是**行为因果**

> 价格形态 ≠ 交易对手行为

---

## 3️⃣ 为什么订单流 / funding 可以？

因为它们满足三点：

1. **行为驱动（谁在干什么）**
2. **因果稳定（买卖 → 价格）**
3. **跨市场泛化**

数学特征几乎全部不满足。

---

## 4️⃣ 那这些数学特征是不是完全没用？

不是，但**位置不一样**👇

### ✅ 正确使用位置

| 位置                          | 是否适合    |
| ----------------------------- | ----------- |
| Tree rules                    | ❌           |
| Execution gate                | ❌           |
| Regime embedding              | ⚠️（可弱化） |
| NN latent features            | ✅           |
| Meta risk / regime classifier | ✅           |

👉 **它们适合做“软感知”，不适合做“硬规则”。**

## 一、Regime Embedding 的最终定义（8 维）

这 8 维不是“指标”，而是**交易因果语义轴**：

```text
R = [
  r_flow_strength        # 真钱是否在进/出
  r_flow_directionality  # 是否单边主导
  r_liquidity_state      # void / absorption
  r_volatility_state     # compression / expansion
  r_participation        # 市场是否有人
  r_crowding             # 是否拥挤（funding/OI）
  r_exhaustion           # 是否衰竭
  r_structure_validity   # 结构是否被 orderflow 证实

```

**范围建议**：`[-1, 1]`（对 Router / Execution 非常友好）

---

## 二、Head 结构（极小 MLP）

```python
class RegimeEmbeddingHead(nn.Module):
    def __init__(self, in_dim):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(in_dim, 64),
            nn.ReLU(),
            nn.Linear(64, 32),
            nn.ReLU(),
            nn.Linear(32, 8),
            nn.Tanh()
        )

    def forward(self, x):
        return self.net(x)   # shape: [B, 8]
```

* **极小**：参数 < 5k
* **稳定**：不会抢主模型容量
* **可解释**：每一维有明确语义

## 四、监督信号怎么来？（关键）

## `model.py`

```python
import torch
import torch.nn as nn
import torch.nn.functional as F


class UnifiedSafetyHead(nn.Module):
    def __init__(
        self,
        path_dim=4,          # trend / mean / chop / notrade
        regime_dim=8,
        exec_ctx_dim=5,
        hidden_dim=128
    ):
        super().__init__()

        input_dim = (
            path_dim
            + regime_dim
            + exec_ctx_dim
            + 2   # ood_score, survival_prob
        )

        self.backbone = nn.Sequential(
            nn.Linear(input_dim, hidden_dim),
            nn.LayerNorm(hidden_dim),
            nn.ReLU(),

            nn.Linear(hidden_dim, hidden_dim),
            nn.LayerNorm(hidden_dim),
            nn.ReLU(),
        )

        # extinction probability (核心)
        self.extinction_head = nn.Linear(hidden_dim, 1)

        # execution control
        self.allow_head = nn.Linear(hidden_dim, 1)
        self.size_head = nn.Linear(hidden_dim, 1)
        self.speed_head = nn.Linear(hidden_dim, 1)

    def forward(
        self,
        path_logits,
        regime_emb,
        ood_score,
        survival_prob,
        exec_ctx
    ):
        """
        All inputs are float tensors, no thresholds.
        """

        x = torch.cat([
            path_logits,
            regime_emb,
            ood_score.unsqueeze(-1),
            survival_prob.unsqueeze(-1),
            exec_ctx
        ], dim=-1)

        h = self.backbone(x)

        extinction_prob = torch.sigmoid(self.extinction_head(h))

        allow_prob = torch.sigmoid(self.allow_head(h))
        size_cap = torch.sigmoid(self.size_head(h))
        speed_cap = torch.sigmoid(self.speed_head(h))

        return {
            "extinction_prob": extinction_prob,
            "allow_prob": allow_prob,
            "size_cap": size_cap,
            "speed_cap": speed_cap
        }
```

## `loss.py`

```python
def unified_safety_loss(
    outputs,
    y_extinct,
    hazard_score=None,
    lambda_hazard=0.3,
    lambda_smooth=0.01
):
    """
    outputs: dict from model
    y_extinct: {0,1}
    hazard_score: optional ∈[0,1]
    """

    p_ext = outputs["extinction_prob"]

    # 主监督：灭绝风险
    loss_ext = F.binary_cross_entropy(p_ext, y_extinct)

    loss = loss_ext

    # 辅助：hazard 平滑排序
    if hazard_score is not None:
        loss_hazard = F.mse_loss(p_ext, hazard_score)
        loss += lambda_hazard * loss_hazard

    return loss
```

> ⚠️ 注意
>
> * **不在 loss 里 hard clamp allow / size**
> * allow / size 的约束来自 extinction 概率的学习

# 🧩 1️⃣ `revenge_scheduler.py`（完整可用代码）

> 职责：
> **只看账户 survival，不看单笔盈亏**
> 输出：`revenge_strength ρ_t ∈ [ρ_min, ρ_max]`

```python
# revenge_scheduler.py
import math
from dataclasses import dataclass


@dataclass
class SurvivalState:
    rolling_max_dd: float        # e.g. -0.18
    extinction_rate_30d: float   # e.g. 0.15
    recovery_speed: float        # normalized, higher = faster recovery
    forced_exit_freq: float      # per day


@dataclass
class RevengeConfig:
    rho_min: float = 0.1
    rho_max: float = 2.0
    gamma: float = 2.5


class RevengeScheduler:
    def __init__(self, config: RevengeConfig):
        self.cfg = config

    def compute_ass(self, s: SurvivalState) -> float:
        """
        Account Survival Score ∈ [0,1]
        """
        dd_score = max(0.0, 1.0 + s.rolling_max_dd)   # dd = -0.2 → 0.8
        ext_score = math.exp(-3.0 * s.extinction_rate_30d)
        rec_score = min(1.0, s.recovery_speed)
        forced_penalty = math.exp(-2.0 * s.forced_exit_freq)

        ass = (
            0.35 * dd_score +
            0.25 * ext_score +
            0.25 * rec_score +
            0.15 * forced_penalty
        )
        return float(max(0.0, min(1.0, ass)))

    def revenge_strength(self, s: SurvivalState) -> float:
        ass = self.compute_ass(s)
        rho = self.cfg.rho_min + \
              (self.cfg.rho_max - self.cfg.rho_min) * ((1.0 - ass) ** self.cfg.gamma)
        return float(rho)

    def near_death(self, s: SurvivalState) -> bool:
        ass = self.compute_ass(s)
        return ass < 0.25 or s.extinction_rate_30d > 0.3
```

* ❌ 不依赖任何交易规则
* ❌ 不依赖 execution 结果
* ✅ 只依赖账户生命状态
* ✅ 单调、连续、稳定


**注意**：
👉 **C 收益下降是正确结果**

# 🧠 4️⃣ “创伤记忆是否过拟合”的检验方法（极重要）

> 这是 Safety 系统最容易犯的错误

---

## 🧪 实验 1：记忆反转测试（Memory Flip Test）

### 方法

* 交换两个 archetype 的仇恨记忆
* 其余保持不变

### 判定

```text
若表现变化 ≈ 0 → 记忆无效
若系统崩溃 → 记忆过拟合
合理情况 → 中等劣化
```

---

## 🧪 实验 2：时间洗牌测试（Temporal Shuffle）

* 打乱 extinction 发生时间
* 保留次数

👉 若行为大变
= **模型记住了“时间”，不是风险**

---

## 🧪 实验 3：虚假创伤注入

* 注入从未真实发生的 extinction
* 只改 memory，不改 execution

👉 若 Safety 强烈反应
= **PTSD 过拟合**

---

## 🧠 合格 Safety 的标准（非常重要）

> **记仇，但不迷信过去**

数学化表达：

```math
∂Penalty / ∂Memory  <  ∂Penalty / ∂CurrentRegime
```
