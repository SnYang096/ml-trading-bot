# Entry Archetype × 正交层 × 订单流角色：系统定法文档

> **Entry Archetype 决定"打哪种仗"**
> **HTF / Liquidity / Vol 决定"敢不敢 & 下多少"**
> **Order Flow 决定"这枪是不是在骗我"**

---

# 一、总纲

* **趋势场景（ME / TC）**：
  订单流 **确认"钱是否真的在顺着结构继续进来"**
  核心是：*持续性 / 单边性 / 不对称扩张*

* **反转场景（FER）**：
  订单流 **识别"谁被困住、谁在被清算"**
  核心是：*失败 / 反转瞬间 / 流动性错配*

系统级铁律：

> **趋势 用订单流做「确认」（Confirm）**
> **反转 用订单流做「定罪」（Signal / Gate）**

---

# 二、核心架构：双层 Archetype + 四层正交设计

## ⚠️ 修正历史

### 修正 1：HTF 不是 archetype（已完成）

旧文档把 **HTF Bias + LTF Entry** 当成 archetype 与 ME / BPC 并列。
HTF 是 **正交约束层**，不产生 entry signal。

### 修正 2：5 个语义形态 → 3 个 Router Archetype（本次）

5 个语义形态（ME / BPC / FBF / LSR / AER）在研究层面完全合理，
它们描述的是 **5 种市场失败 / 延续的语义形态**，是认知分辨率。

但如果把 5 个都当成实盘独立 archetype 会出 3 个长期问题：

1. **Router 不稳定**：样本被切太细，archetype 边界模糊（尤其 FBF / LSR / AER）
2. **Execution 心智负担过大**：不同 stop / size / timing，debug 成本爆炸
3. **过拟合"失败的细节"**：市场失败形态会演化，但"失败这个事实"不会

正确做法：**语义合并、工程收敛**。
5 个用于理解，3 个用于交易。

---

## 正确的系统架构

### 第一层：Router Archetype（3 个，互斥）

Router **只输出这 3 个之一**，同一时刻只能选一个。

| Router Archetype | 缩写  | 博弈本质          | 合并来源             |
| ---------------- | --- | ------------- | ---------------- |
| **Momentum Expansion** | ME  | 钱在**加速进来**     | ME               |
| **Trend Continuation** | TC  | 趋势中的可预测延续     | BPC              |
| **Failure Exhaustion Reversal** | FER | 市场已经失败，清算已发生  | FBF + LSR + AER  |

#### ME vs TC 的区别

| 对比  | ME       | TC（原 BPC）  |
| --- | -------- | ---------- |
| 节奏  | 加速       | 呼吸         |
| Vol | 扩张       | 压缩 → 扩张    |
| 流动性 | 高        | Silence    |
| 风险  | 追价       | 结构内        |

#### FER 合并理由

FBF / LSR / AER 在交易层面共享 4 个铁律：

1. 都是 **逆当前 impulse**
2. 都必须 **订单流定罪**
3. 都需要 **时间窗失效**
4. 都必须 **降仓**

它们的区别（失败在哪里、失败的证据是什么）**不该是 Router 级区分**，
而是 FER 内部的 sub-mode。

### 第一层补充：Semantic Sub-Type（仅用于 Execution）

在 FER 内部，sub-mode **不参与 Router 训练**，只用于：

* stop placement
* TP target
* time_window
* confidence scaling

```yaml
fer_sub_mode:
  - failed_breakout     # 叙事崩塌
  - liquidity_sweep     # 流动性陷阱
  - auction_exhaustion  # 动能衰竭
```

### 第二层：正交约束层（不互斥，同时生效）

| 正交层               | 回答的问题        | 角色                     |
| ----------------- | ------------ | ---------------------- |
| **HTF Bias**      | 你现在逆着大钱吗？    | direction / confidence / risk |
| **Liquidity Regime** | 市场吃不吃单？      | 油门能不能踩                 |
| **Volatility Regime** | 价格会不会乱甩？     | 刹车要不要提前                |

### 第三层：Feature Group × Role（per Archetype）

| Role              | 含义           |
| ----------------- | ------------ |
| 🔫 **Signal**    | 决定是否产生 entry |
| 🛡 **Gate**      | 不满足就不交易      |
| 🔍 **Confirm**   | 只做否定性检查      |
| 📏 **Sizer**     | 只影响仓位        |
| ❌ **Disable**    | 在该 archetype 下关闭 |

### 第四层：DSL 规则配置

# 三、5 个语义形态 × 订单流详解

> 5 个语义形态是**认知分辨率**，不是工程复杂度。
> 此章保留完整语义描述，供理解和 Execution sub-mode 参考。

---

## 🔴 趋势族

### ① ME：Momentum Expansion → Router: `ME`

**（动能 × 波动率告诉你钱在加速流入）**

> 赌什么：市场不是涨，而是 **加速涨**

#### 🔑 一级

* **`trade_cluster_semantic`**
  * 连续方向性 cluster
  * cluster 间距缩短（time compression）

* **`fp_scene`**
  * 单边 imbalance **连续而非脉冲**

* **`wpt_scene`**
  * 能量是否 **跨 bar 持续**

#### 🔑 二级

* **`liquidity_void_scene`**
  * 价格是否在 **low resistance zone** 快速推进

* **`vpin_scene`**（二阶修正：从 Disable 升级为 Confirm）
  * 不看绝对值，看方向是否 align
  * 条件：`vpin.scene.directional_consistency == true`
  * 角色：**燃料一致性检查器**

#### ❌ 用处不大的

* wick_scene（不是失败结构）

> 👉 *"单有没有来得越来越快"*

---

### ② BPC：Breakout → Pullback → Continuation → Router: `TC`

**（结构再次被市场证明）**

> 赌什么：Pullback 不是撤退，而是 **强手换手**

#### 🔑 一级

* **`fp_scene`（footprint imbalance scene）**
  * Pullback 期间：counter-side 但 **未形成主导 imbalance**
  * Continuation 前：主方向 imbalance **重新扩张**

* **`trade_cluster_scene`**
  * Pullback 是否发生在 **低 aggressiveness cluster**
  * Continuation 是否出现 **新的大单 cluster**

#### 🔑 二级

* **`vpin_scene / vpin_semantic`**
  * Pullback 阶段 VPIN **不上升**

* **`wick_scene`**（二阶修正：从 Disable 升级为 Confirm）
  * 角色：**"反向尝试被瞬间吸收"**
  * 条件：`wick_scene.rejection_at_support == true`

#### ❌ 用处不大的

* dtw_reversal（方向相反）

> 👉 *"反向单有没有真的掌控订单簿？"*

---

## 🔵 反转族 → 统一归入 Router: `FER`

> FER 内的 3 个子语义**共享 Gate / invalidate / sizing 铁律**，
> 区别仅在于"失败发生在哪里"和"用什么证据定罪"。

### ③ FBF：Failed Breakout Fade → FER sub-mode: `failed_breakout`

**（叙事崩塌）**

> 收割什么：最后一批"相信会突破"的人

#### 🔑 一级

* **`wick_scene_semantic_scores_f`**：长 wick + 高成交量
* **`fp_scene`**：aggressor 很多，但 **价格不走**
* **`vpin_scene`**：突破时 VPIN **快速上升**

#### 🔑 二级

* **`trade_cluster_scene`**：cluster 集中在极值区，无 follow-through

> 👉 *"很多人冲进去，但市场拒绝给他们空间"*

---

### ④ LSR：Liquidity Sweep → Rejection → FER sub-mode: `liquidity_sweep`

**（流动性陷阱）**

> 收割什么：被止损"引出来"的流动性

#### 🔑 一级

* **`liquidity_void_scene`**：sweep 后立刻回到高流动区
* **`fp_scene`**：sweep 时单边 aggressor，反转时 opposite imbalance
* **`trade_cluster_scene`**：**单笔巨大但失败的 cluster**

#### 🔑 二级

* **`wick_scene`**：典型 stop-hunt wick

> 👉 *"流动性被吃了，但方向没走成"*

---

### ⑤ AER：Auction Exhaustion Reversal → FER sub-mode: `auction_exhaustion`

**（动能衰竭）**

> 收割什么：趋势末期的"信仰充值"

#### 🔑 一级

* **`trade_cluster_semantic`**：cluster **变大但推进变小**
* **`wpt_scene`**：能量峰值后衰减
* **`fp_scene`**：imbalance 仍在，但 **delta 不再带动价格**

#### 🔑 二级

* **`dtw_scene`**：与历史"衰竭模板"相似度上升

> 👉 *"单还在，但已经推不动价格了"*

---

# 四、正交约束层详解

---

## HTF Bias（方向 / 信心 / 风险分配器）

**HTF 永远不产生 entry signal，只改"敢不敢 & 下多少"**

| 功能         | 含义               |
| ---------- | ---------------- |
| direction  | long / short 倾向  |
| confidence | 仓位放大 or 缩小       |
| risk       | 是否允许某些 archetype |

示例：
* HTF = up → ME 做多是顺势，FER 做空是逆势
* HTF = flat → ME 需要更严格，FER 反而更安全

### Order Flow 与各层的关系

| Router Archetype | Order Flow 角色                       |
| ---------------- | ------------------------------------ |
| ME               | Confirm acceleration / no absorption |
| TC               | Confirm pullback not hijacked        |
| FER              | Prove failure & trapping             |
| HTF（正交层）         | ❌ 不直接用订单流做信号                        |

---

## Liquidity Regime（市场吃不吃单）

| 状态                    | 市场特性     | 对 Entry 的影响  |
| --------------------- | -------- | ------------ |
| **High Liquidity**    | 成交多、盘口厚  | 适合 ME        |
| **Liquidity Silence** | 成交骤降     | 适合 TC        |
| **Liquidity Void**    | 阻力很小     | ME 会加速       |
| **Liquidity Trap**    | 单很多但不走   | FER          |

> 流动性 ≠ 方向，只决定：**"这一枪打出去，会不会卡壳"**

---

## Volatility Regime（价格会不会乱甩）

| 状态                   | 语义   | 影响                |
| -------------------- | ---- | ----------------- |
| **Low Vol**          | 压缩   | 适合 TC             |
| **Vol Expansion**    | 扩张   | 适合 ME             |
| **High Chaotic Vol** | 抽风   | 降仓 / 禁交易          |
| **Vol Decay**        | 动能衰减 | FER 候选            |

---

# 七、DSL 规则配置

## 统一 DSL 结构

```yaml
archetype: <name>           # Router 级：ME / TC / FER

gate:        # 必须先成立（hard filter）
  - <condition>

signal:      # 触发候选（软信号，可打分）
  - <condition>

confirm:     # 成交/订单流确认
  - <condition>

invalidate:  # 一旦触发，立即 NO_TRADE
  - <condition>

execution:
  side: long | short
  entry: market | limit
  size: <expression>
  stop: <expression>
  take_profit: <expression>
```

表达式约定：
* `feature(x)` → semantic node 输出
* `trend(tf)` → router / trend head
* 所有条件默认 **AND**，可用 `any:` / `all:`

---

## 🔴 ① ME：Momentum Expansion

```yaml
archetype: momentum_expansion

gate:
  - trend("htf") == up
  - bb_width > percentile(80)

signal:
  - wpt.energy > rolling_max(20)
  - wpt.scene.energy_continuity > 0.7

confirm:
  - feature("trade_cluster_semantic.directional") == true
  - liquidity_void.ahead == true
  - vpin.scene.directional_consistency == true

invalidate:
  - delta_price_efficiency < threshold_low
  - feature("fp_scene.absorption") == true

execution:
  side: long
  entry: market
  size: base_size * regime_confidence * 1.2
  stop: entry - 1.2 * atr
  take_profit: trailing_atr(2.0)
```

> 不是追突破，是在追 **"越来越快"**

---

## 🔴 ② TC：Trend Continuation（原 BPC）

```yaml
archetype: trend_continuation

gate:
  - trend("htf") == up
  - sr_structure.breakout_confirmed == true
  - compression.score > 0.75

signal:
  - pullback.depth_atr < 0.6
  - feature("fp_scene.counter_imbalance") < 0.3

confirm:
  - feature("trade_cluster_scene.new_trend_cluster") == true
  - feature("volume_profile_scene.acceptance") > 0.6
  - feature("wick_scene.rejection_at_support") == true

invalidate:
  - feature("fp_scene.counter_imbalance") > 0.6
  - pullback.depth_atr > 1.0

execution:
  side: long
  entry: market
  size: base_size * regime_confidence * 1.0
  stop: pullback.low - 0.5 * atr
  take_profit: entry + 2.5 * atr
```

> 回踩期间反向单没接管 + 新的钱重新进来

---

## 🔵 ③ FER：Failure Exhaustion Reversal（统一 + sub-mode 派发）

### Router 级统一 Gate / Signal / Confirm

```yaml
archetype: failure_exhaustion_reversal

gate:
  any:
    - sr_structure.false_breakout == true            # → sub: failed_breakout
    - liquidity_void_scene.sweep_detected == true    # → sub: liquidity_sweep
    - all:                                           # → sub: auction_exhaustion
      - trend("htf") == weakening
      - rsi_divergence == true

signal:
  any:
    - feature("wick_scene.long_wick") == true
    - feature("fp_scene.aggressive_but_stuck") == true
    - feature("fp_scene.stop_run") == true
    - feature("fp_scene.opposite_imbalance") > 0.6
    - feature("trade_cluster_semantic.cluster_size_up_price_down") == true
    - wpt.scene.energy_decay > 0.7

confirm:
  - feature("trade_cluster_scene.trapped_cluster") == true
  - feature("fp_scene.delta_no_followthrough") == true

invalidate:
  - time_since_signal > fer_time_window              # 铁律：时间窗失效
```

### Sub-mode Execution 派发表

| 参数             | failed_breakout | liquidity_sweep | auction_exhaustion |
| -------------- | --------------- | --------------- | ------------------ |
| entry          | market          | market          | **limit**          |
| size_mult      | 0.6             | 0.7             | **0.5**            |
| stop           | wick.high + 0.3×atr | sweep_extreme + 0.4×atr | recent_extreme + 0.5×atr |
| take_profit    | mean_value_area | value_area_mid  | vwap               |
| time_window    | 5 bars          | 5 bars          | **8 bars**         |

> Sub-mode 由 Gate 条件自动推断，**不需要 Router 额外输出**。
> 所有 sub-mode 共享：逆 impulse、订单流定罪、降仓、时间窗失效。

# 八、工程约束（系统宪法）

---

## 铁律 1：Router 只输出 3 个 Archetype

```text
Router output ∈ { ME, TC, FER, NO_TRADE }
```

* 同一时刻只能选一个
* 5 个语义形态只在 Execution 层区分
* 否则 Router 样本切太细，边界模糊

---

## 铁律 2：Confirm 永远不参与 Router 训练

> **Confirm = execution sanity check，不是 regime evidence**

如果 Router 学 Confirm：
👉 模型会为了通过 Confirm 而扭曲 Signal

工程上要**强制隔离**。

---

## 铁律 3：FER 必须有「时间窗失效」

```yaml
invalidate:
  - time_since_signal > X bars
```

否则会得到：**"对的反转，但来得太晚"**

---

## 铁律 4：FER 默认 Size ≤ 趋势的 0.5~0.7

```yaml
size: base_size * regime_confidence * archetype_risk_multiplier
```

* ME / TC：`archetype_risk_multiplier ≈ 1.0 ~ 1.2`
* FER：`≈ 0.5 ~ 0.7`（永远不许满仓）

不是因为胜率，而是因为 **失败代价不对称**。

---

## 铁律 6：什么时候可以重新拆回 4–5 个

当且仅当能证明某个 FER sub-mode：
**在相同 Regime / HTF 下，拥有稳定、显著不同的收益分布**。

例如：
* LSR 的 R:R 明显更高
* AER 的胜率低但尾部肥
* FBF 对 HTF 敏感度极高

未证明之前，保持 3 个。

# 十一、反转类型总览 & ME / BPC / FER 对照（2026-04）

本章把上文「5 个语义形态 / 3 个 Router」与**当前仓库实现**对齐，并单独拉出你关心的 **「吸收型区间反转」**（图表上双底/双顶 + 吸收），避免与 **上冲失败衰竭反转** 混为一谈。

## 11.1 是不是「反转」？——先分三类

| 类型 | Router / 策略目录 | 博弈一句话 | 是否「反转」 |
|------|-------------------|------------|--------------|
| **趋势延续** | `TC` / `config/strategies/bpc` | 突破后回踩、强手换手后再走 | 否，顺势结构 |
| **动能扩张** | `ME` / `config/strategies/me` | 波动与参与加速，钱在「越冲越猛」 | 否，顺势加速 |
| **失败衰竭反转** | `FER` / `config/strategies/fer` | 单边 impulse 推进失败 → 反向清算 | **是**，逆最近一段 impulse |
| **（文档层）FBF / LSR / AER** | 统一归入 FER 的 **sub-mode**（第二章） | 失败发生在结构边界 / 扫流动性 / 拍卖衰竭 | **是**，证据维度不同 |
| **吸收型区间反转（提议）** | **尚未**单独 Router；可与 FER 混淆 | 波动区内 **双底/双顶 + 关键位吸收** 后的反向 | **是**，但因果叙事更接近「区间换手完成」而非「单段 impulse 失败」 |

说明：

- **ME 不是 FER 的代码级「反面」**（没有成对取反函数）。对照关系是 **语义上的**：ME 吃 **延续与加速**，FER 吃 **失败与清算**；同一根 K 上两者常互斥，但实现上是两套独立规则与特征。
- **BPC** 与 FER 的互补关系见 `docs/architecture/BPC_FER_互补对冲设计.md`（其中 FER 方向规则若与现网不一致，以 `config/strategies/fer/archetypes/direction.yaml` 为准）。

## 11.2 FER 在代码里对应什么（与第二章 sub-mode 映射）

| 文档 sub-mode | 含义（第二章） | 当前实现抓手（特征 / 规则） |
|---------------|----------------|-----------------------------|
| `failed_breakout` | 叙事崩塌、假突破 | `fer_sr_failed_breakout_*`、`direction.yaml` 优先规则 `fer_sr_failed_breakout_direction_signed` |
| `liquidity_sweep` | 扫流动性后拒绝 | 订单流 + `liquidity_void_*`、SR/VP 相关 gate/prefilter（随配置变） |
| `auction_exhaustion` | 动能衰竭、仍冲但不走 | `fer_impulse_failure_score`、`fer_signed_efficiency*`、`fer_aggressor_absorption` 等（见 `fer_features.py`） |
| **吸收型区间反转** | 双底/双顶 + 吸收 | **无独立子模式**；若硬塞进同一套 `entry_filters` OR，易与「impulse failure」语义打架 → 见 §11.4 |

主特征入口：`src/features/time_series/fer_features.py`（`compute_fer_failure_signals_from_series`，含 SR 子语义与 **结构锚点** `fer_range_pos_20`、`fer_ols_pos`、`fer_ols_width_norm`）。  
依赖注册：`config/feature_dependencies.yaml` → `fer_failure_signals_f`。

## 11.3 当前决策栈（与代码一致）

所有走 `GenericLiveStrategy` 的策略（**bpc / fer / me**）在同一根 bar 上的顺序为（**与漏斗图四层顺序一致**）：

1. **Prefilter** — `archetypes/prefilter.yaml`
2. **Direction** — `archetypes/direction.yaml`
3. **Gate** — `archetypes/gate.yaml`
4. **Entry Filter** — `archetypes/entry_filters.yaml`（OR 多分支）
5. **Evidence** — `archetypes/evidence.yaml`（若有）
6. **Execution** — `archetypes/execution.yaml`

实现：`src/time_series_model/live/generic_live_strategy.py` → `decide()`。

**FER 研究侧方向（2026-04）**：仅 `fer_sr_failed_breakout_direction_signed` 与 `fer_impulse_failure_direction_signed`，**无**全样本 CVD/ROC 兜底（避免非失败上下文强行给方向）。见 `config/strategies/fer/archetypes/direction.yaml`。

## 11.4 「吸收型区间反转」要不要并进 FER？

| 方案 | 说明 |
|------|------|
| **同一 `fer` 内加 OR 分支** | 实现快；长期易 **语义混杂**（回测难以区分「失败反转」vs「区间吸收反转」）。 |
| **独立策略目录**（如 `fer_range` / `fer_absorption`） | 特征可复用 `fer_failure_signals_f`，但 **prefilter/entry/direction 单独维护**；PCM、滚动历史目录分离，**推荐**在你很在意叙事纯度时采用。 |
| **仅调阈值 / 滚动优化** | 改善通过率，**不解决**「四层在不同 bar 亮起」的时序问题；需 **跨 bar 对齐** 设计，见 `docs/architecture/跨bar记忆与分阶段决策_设计草案.md`。 |
