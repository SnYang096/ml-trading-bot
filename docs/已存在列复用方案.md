# 已存在列复用方案

## 问题

当多个特征输出相同的列名（如 `poc`）时，由于并行计算机制，不能简单地检查列是否存在就跳过计算。需要一种机制来：
1. 避免重复计算
2. 处理并行计算的竞态条件
3. 复用已存在的列值

## 解决方案

### 方案1：在合并时使用 `combine_first`（已实施）

**原理**：
- `combine_first` 会优先使用已有的值，只填充 NaN
- 这样可以避免重复计算，同时处理并行计算的情况

**实现位置**：
- `src/features/loader/parallel_computer.py` 中的所有合并位置

**代码示例**：
```python
existing_cols = [c for c in feature_df.columns if c in result_df.columns]

# 处理已存在的列：使用 combine_first 合并
if existing_cols:
    for col in existing_cols:
        if col in feature_df.columns and col in result_df.columns:
            # combine_first: 如果 result_df[col] 有值，保留；如果是 NaN，用 feature_df[col] 填充
            result_df[col] = result_df[col].combine_first(feature_df[col])
```

**优点**：
- ✅ 处理并行计算的竞态条件
- ✅ 避免重复计算（优先使用已有的值）
- ✅ 填充缺失值（如果已有列有 NaN，用新值填充）

**缺点**：
- ⚠️ 仍然会计算特征（即使列已存在），只是合并时复用已有值
- ⚠️ 如果两个特征并行计算，都会计算，但合并时只保留第一个的值

### 方案2：在特征计算前检查列是否已存在（已实施）

**原理**：
- 在特征计算函数中，检查输出列是否已存在且完整
- 如果已存在且非 NaN 值比例 > 50%，跳过计算

**实现位置**：
- `src/features/loader/feature_wrappers.py` 中的 `compute_sqs_hal_high` 和 `compute_sqs_hal_low`
- `src/features/time_series/baseline_features.py` 中的 `add_poc_hal_dimensionless_features`

**代码示例**：
```python
# 检查 poc 列是否已存在且完整（避免重复计算）
poc_exists = "poc" in result.columns
if poc_exists:
    poc_non_na_ratio = result["poc"].notna().sum() / len(result) if len(result) > 0 else 0
    if poc_non_na_ratio > 0.5:
        print(f"       ℹ️  'poc' column already exists with {poc_non_na_ratio*100:.1f}% non-NaN values, reusing it")
        # 跳过 poc 的计算
        need_poc = False
```

**优点**：
- ✅ 真正避免重复计算（如果列已存在且完整，直接跳过）
- ✅ 节省计算资源

**缺点**：
- ⚠️ 并行计算时可能有竞态条件（两个特征同时检查，都发现列不存在，都开始计算）
- ⚠️ 需要设置阈值（如 50%）来判断列是否"完整"

### 方案3：将共享列作为独立特征（建议的长期方案）

**原理**：
- 将 `poc` 作为独立的基础特征
- 其他特征（如 `sqs_hal_high`, `sqs_hal_low`）依赖 `poc` 特征
- 通过依赖关系确保 `poc` 只计算一次

**实现方式**：
```yaml
# config/feature_dependencies.yaml
poc:
  module: baseline
  compute_func: compute_poc
  dependencies: ["wpt_price_reconstructed"]
  output_columns: ["poc"]
  category: sr_structure

sqs_hal_high:
  dependencies: ["atr", "wpt_price_reconstructed", "poc"]  # 添加 poc 依赖
  output_columns: ["sqs_hal_high", "hal_high"]  # 移除 poc
  # 在计算函数中，直接使用已有的 poc 列，不重新计算

sqs_hal_low:
  dependencies: ["atr", "wpt_price_reconstructed", "poc"]  # 添加 poc 依赖
  output_columns: ["sqs_hal_low", "hal_low"]  # 移除 poc
  # 在计算函数中，直接使用已有的 poc 列，不重新计算
```

**优点**：
- ✅ 从根本上解决问题（`poc` 只计算一次）
- ✅ 通过依赖关系确保计算顺序
- ✅ 代码更清晰，职责分离

**缺点**：
- ⚠️ 需要重构特征设计
- ⚠️ 需要修改配置文件

## 当前实施方案

**组合使用方案1和方案2**：
1. **方案2**：在特征计算前检查列是否已存在，如果存在且完整，跳过计算
2. **方案1**：在合并时使用 `combine_first`，确保即使并行计算，也能正确合并

**效果**：
- 如果 `sqs_hal_high` 先计算，`poc` 列会被创建
- 当 `sqs_hal_low` 计算时，如果检测到 `poc` 列已存在且完整，会跳过 `poc` 的计算
- 即使两个特征并行计算，合并时 `combine_first` 会确保只保留一个值

## 性能影响

- **方案1**：仍然会计算特征，但合并时复用已有值（性能提升有限）
- **方案2**：真正避免重复计算（性能提升明显）
- **方案3**：从根本上避免重复计算（最佳性能）

## 建议

1. **短期**：继续使用方案1+方案2的组合
2. **长期**：考虑实施方案3，重构特征设计
