# 实时流数据补全使用指南

## 一、问题场景

### 1.1 数据缺失的原因

实时流可能因为以下原因导致数据缺失：

1. **网络中断**：WebSocket 连接断开，消息丢失
2. **系统重启**：程序崩溃或服务器重启
3. **延迟接收**：WebSocket 消息延迟或乱序
4. **交易所故障**：交易所服务器临时故障

### 1.2 数据类型的区别

- **K线数据**：可以通过实时 API 和 WebSocket 获取，也可以下载历史数据
- **Tick 数据**：
  - ✅ 实时数据：通过 WebSocket aggTrades stream 获取
  - ✅ 历史数据：只能从币安官方数据下载页面 (https://data.binance.vision/) 下载

### 1.2 数据缺失的影响

- ⚠️ **特征计算错误**：某些特征需要连续数据
- ⚠️ **模型预测偏差**：特征错误导致预测不准确
- ⚠️ **交易信号错误**：可能导致错误的交易决策

## 二、解决方案

### 2.1 自动补全机制

系统会自动检测并补全缺失数据：

1. **实时检测**：收到新数据时，检测与前一条数据的间隔
2. **自动下载**：发现缺失后，自动从交易所 API 下载
3. **自动写入**：下载的数据自动写入 QuestDB 和内存
4. **定期检查**：后台任务定期检查数据完整性

### 2.2 手动补全

系统重启后，可以手动补全停机期间的数据：

```python
# 补全指定时间范围的数据
data_manager.manual_fill_gaps(
    start_time=pd.Timestamp("2024-01-01 10:00:00"),
    end_time=pd.Timestamp("2024-01-01 12:00:00"),
)
```

### 3.3 系统启动时补全

```python
def on_start(self):
    """策略启动时初始化"""
    # 1. 加载 Warmup 数据
    warmup_df = self.data_manager.initialize()
    
    # 2. 检查并补全最近 24 小时的数据
    if len(warmup_df) > 0:
        recent_start = warmup_df["timestamp"].iloc[-1] - pd.Timedelta(hours=24)
        recent_end = warmup_df["timestamp"].iloc[-1]
        
        missing_count = self.data_manager.manual_fill_gaps(
            recent_start, recent_end
        )
        
        if missing_count > 0:
            print(f"✅ 启动时补全了 {missing_count} 条缺失数据")
            # 重新加载数据
            warmup_df = self.data_manager.initialize()
```

## 四、工作流程

### 4.1 实时数据接收流程

```
WebSocket 收到新 K线
    ↓
检测与前一条的间隔
    ↓
间隔 > 1.5 × 期望间隔？
    ↓ 是
计算缺失的时间戳
    ↓
从交易所 API 下载缺失数据
    ↓
验证数据质量
    ↓
写入 QuestDB
    ↓
插入到内存历史数据
    ↓
继续处理新数据
```

### 4.2 定期检查流程

```
后台任务（每 60 秒）
    ↓
检查最近 100 条数据
    ↓
检测缺失
    ↓
有缺失？
    ↓ 是
下载并补全
```

### 4.3 启动时补全流程

```
系统启动
    ↓
加载 Warmup 数据
    ↓
检查最近 24 小时数据完整性
    ↓
查询 QuestDB 缺失的时间戳
    ↓
从交易所下载缺失数据
    ↓
写入 QuestDB 和内存
    ↓
完成初始化
```

## 六、注意事项

### 6.1 API 限制

- ⚠️ 交易所 API 有频率限制
- ✅ 系统会自动控制下载频率
- ✅ 使用 `enableRateLimit=True` 避免超限

### 6.2 历史数据范围

- ⚠️ 某些交易所只保留最近 N 天的数据
- ⚠️ 超出范围的数据无法下载
- ✅ 建议定期备份到 Parquet

### 6.3 数据质量

- ✅ 下载的数据会自动验证
- ✅ 过滤异常数据（价格、成交量不合理）
- ✅ 检查时间戳连续性

### 6.4 性能影响

- ⚠️ 数据补全可能阻塞主流程
- ✅ 建议异步执行
- ✅ 使用后台任务定期检查

## 七、故障排查

### 7.1 常见问题

#### 问题1：无法下载缺失数据

**可能原因**：
- 超出交易所历史数据范围
- API 凭证无效
- 网络连接问题

**解决方案**：
- 检查时间范围是否在交易所保留范围内
- 验证 API 凭证
- 检查网络连接

#### 问题2：下载的数据不匹配

**可能原因**：
- 时间戳对齐问题
- 交易所时区问题

**解决方案**：
- 系统会自动处理时间戳对齐（允许 10% 误差）
- 确保使用正确的时区

#### 问题3：补全速度慢

**可能原因**：
- API 频率限制
- 网络延迟
- 缺失数据量大

**解决方案**：
- 使用 `enableRateLimit=True`
- 批量下载（系统自动处理）
- 考虑异步补全

## 九、总结

### 9.1 核心功能

✅ **自动检测**：收到新数据时自动检测缺失  
✅ **自动补全**：发现缺失后自动从交易所下载
