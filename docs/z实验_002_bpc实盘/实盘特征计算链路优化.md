# 实盘特征计算链路优化

> 日期: 2026-02-12

## 问题

实盘 `IncrementalFeatureComputer` 通过 `config/live/live_feature_plan.yaml` 加载特征节点，
该文件引用了 NN 多头模型的 `feature_plan.yaml`（tier0 + tier1），导致：

- 加载 **52+ 个 NN tier nodes**，产出 101+ 输出列
- BPC 策略实际只需要 gate / evidence / entry_filters 引用的 **22 个特征列 → 14 个 nodes**
- **86% 的计算是浪费的** NN-specific 特征

## 根因

```yaml
# config/live/live_feature_plan.yaml (旧)
base_feature_plan: "config/nnmultihead/path_primitives_4h_80h_min/feature_plan.yaml"
base_feature_plan_mode: "tiers"
```

NN 多头模型已不再使用，但 live 仍引用其 feature plan 作为特征来源。

## 修复方案：Archetypes Auto-Detect

不再依赖任何静态 feature plan YAML，改为从 archetypes 目录自动提取：

```
live/highcap/config/strategies/bpc/archetypes/
  ├── gate.yaml          → 15 个特征列
  ├── evidence.yaml      → 9 个特征列
  └── entry_filters.yaml → 4 个特征列 (仅 enabled=true)
                           ─────────────
                           22 unique → 14 feature nodes
```

### 提取逻辑

| archetype | 提取方式 |
|-----------|---------|
| gate.yaml | 扫描 `hard_gates` / `soft_filters` / `guardrails` / `system_safety` 的 `when` 字段 |
| evidence.yaml | 取每个 evidence item 的 `feature` 字段 |
| entry_filters.yaml | 取 `enabled: true` 的 filter 的 `conditions[].feature` |

### 节点去重策略

同一特征列可能被多个 node 输出（如 `spectrum_price_flatness` 有 6 个 spectrum node 都产出）。
选择 **output_columns 最少的 node**（最具体/最小计算量）。

## 改动文件

| 文件 | 改动 |
|------|------|
| `src/time_series_model/live/live_feature_plan.py` | 新增 `extract_features_from_archetypes()` |
| `src/time_series_model/live/incremental_feature_computer.py` | 新增 `archetypes_dir` 参数，优先于 `live_feature_plan_path` |
| `scripts/run_live.py` | 从 `MLBOT_STRATEGIES_ROOT` + `bpc/archetypes` 推导路径 |
| `scripts/start_live_test.sh` | 去掉 `MLBOT_BPC_FEATURE_PLAN_YAML` |

## 数据流对比

```
旧:
  live_feature_plan.yaml
    → config/nnmultihead/.../feature_plan.yaml
      → tier0 (39 nodes) + tier1 (14 nodes) + overlay
        = 52+ nodes → 101+ columns

新:
  archetypes/ (gate + evidence + entry_filters)
    → 22 unique columns
      → 14 feature nodes → FeatureComputer 自动解析依赖
```

  ├── nnmultihead/               ← NN 模型（保留，未来可用）

# 我现在的研发流程没问题吧，因为研发流程是分层输出的，所以nn模型不用不输出即可对吧
研发流程是分层独立输出的，每层只依赖上一层的产物：

Tree Model (训练) → predictions.parquet
      ↓
Gate 优化 → gate.yaml
      ↓
Evidence 优化 → evidence.yaml
      ↓
Entry Filter 优化 → entry_filters.yaml
      ↓
Execution/Holding 配置 → execution.yaml / holding.yaml

NN 多头模型（config/nnmultihead/）是一条完全独立的平行路径，和 BPC 树模型研发管线没有任何交叉依赖。不用它就不跑那条路径，其他一切不受影响。

## 2026-02-12 新架构：磁盘批量计算

### 问题

之前的流式计算架构存在以下问题：

1. **Lookback 不足**：`on_bar()` 只使用当前 bar_buffer（480 条 1min bars = 8 小时），无法满足长 lookback 需求：
   - `atr_percentile_f`: window=540 bars × 4h = **90 天**
   - `sr_strength_max_f`: window=60 bars × 4h = **10 天**
   - `sma_200_position_f`: timeperiod=200 bars × 4h = **33 天**

2. **VPIN 不一致**：
   - 研发使用自适应桶（adaptive bucket）：桶大小 = 7天滚动平均分钟成交量 × K
   - 实盘使用固定桶，与研发不一致

3. **订单流特征中间状态无法恢复**：
   - VPIN/Trade Clustering/Footprint 等特征需要批量重算，无法从中间状态恢复
   - warmup 时需要完整历史数据才能正确计算

### 新架构：磁盘 + 批量计算

> 2026-02-12：重构为磁盘批量计算架构，与研发流程完全一致。

#### 架构变更

**旧架构（流式）**：
\`
ticks/bars → on_tick/on_bar → buffer → _update_timeframe_features() → 特征
    ↑                              ↑
    实时数据                    O(n²) DataFrame重建
\`

**新架构（批量）**：
\`
disk ticks/bars (90+天) → compute_features_batch() → 重采样4h + 批量计算 → 特征
         ↑                                                   ↑
    磁盘读取，完整历史                            与研发完全一致
\`

#### 实现细节

1. **`IncrementalFeatureComputer.compute_features_batch()`**：
   - 读取 100 天 1min bars → 重采样为 4h bars
   - 读取 8 天 1min ticks → 计算 VPIN 自适应桶
   - 调用 `StrategyFeatureLoader` 批量计算 OHLCV 特征
   - 返回最后一行特征字典

2. **`OrderFlowListener._compute_and_save_15min_features()`**：
   - 从磁盘读取 100 天 bars + 8 天 ticks
   - 调用 `feature_computer.compute_features_batch()`
   - 保存 + 传给决策引擎

3. **`_restore_state()` 简化**：
   - 不再回放 bars/ticks 到 feature_computer
   - 只恢复 memory_window（BPC 决策需要）
   - 特征计算完全从磁盘批量进行

#### 特征计算流程

\`mermaid
graph TD
    A[磁盘 1min bars (100天)] --> B[重采样为 4h bars]
    C[磁盘 1min ticks (8天)] --> D[VPIN 自适应桶计算]
    B --> E[StrategyFeatureLoader 批量计算]
    D --> E
    E --> F[提取最后一行特征]
    F --> G[决策引擎]
\`

#### Lookback 保障

| 特征 | 研发计算 | 实盘计算 | 保障 |
|------|---------|---------|------|
| atr_percentile | 540 bars × 4h = 90天 | 100天数据 | ✅ 足够 |
| sr_strength_max | 60 bars × 4h = 10天 | 100天数据 | ✅ 足够 |
| sma_200 | 200 bars × 4h = 33天 | 100天数据 | ✅ 足够 |
| VPIN | 7天滚动均值 | 8天数据 | ✅ 足够 |

#### 代码变更

| 文件 | 变更 |
|------|-----|
| `incremental_feature_computer.py` | 新增 `compute_features_batch()`，简化 `on_tick/on_bar` |
| `order_flow_listener.py` | 重写 `_compute_and_save_15min_features()`，简化 `_restore_state()` |
| `run_live.py` | 恢复 `MLBOT_LIVE_WARMUP_DAYS=7` |
| `prepare_warmup_ticks.sh` | 恢复默认 6 个月数据 |

#### 优势

1. **与研发完全一致**：批量计算，相同算法，相同参数
2. **lookback 充足**：100天数据覆盖所有长窗口特征
3. **VPIN 一致**：使用自适应桶算法，与研发相同
4. **无需状态恢复**：批量计算，无需维护中间状态
5. **性能优化**：避免 O(n²) 重复计算，减少内存压力
