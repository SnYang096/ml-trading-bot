# 04 存储：回放与审计

要让“回测 = 实盘”可验证、可追溯，必须把实盘输入事件与关键中间产物持久化。

## 一、推荐分层存储

- **实盘实时写入（权威事件源）**：QuestDB（或同级别时序库）
  - tick / bar /（可选）特征快照
- **离线研究/训练与归档**：Parquet（按月/按日）

本项目的推荐存储架构详见：

- `reference/实时数据存储架构设计.md`

## 二、为什么“可回放”比“能查询”更重要

为了定位线上问题（特征异常、模型漂移、信号错判）你需要：

- **复现**：用同一份事件源重放出当时的特征与信号。
- **对账**：核对当时“看到的市场”与“下单/成交/持仓状态”。
- **审计**：回答“为什么当时下了这单？”（特征快照 + 模型输出 + 风控决策）。

因此建议至少保证以下之一：

- 能从 QuestDB 导出某段时间的事件流用于回放；
- 或在写入时同步落盘到 Parquet（用于长期留存与离线回放）。

## 三、写入一致性（Exactly-once vs At-least-once）

现实系统通常是 **at-least-once**（可能重复）。一致性要求是：

- **存储层可幂等**：以 `(symbol, timeframe, timestamp)`（bar）或 `(symbol, trade_id)`（tick）作为去重键（或近似去重策略）。
- **计算层可去重**：遇到重复事件时状态更新必须确定（跳过/覆盖）。

## 四、回放最小闭环

一次完整回放（用于一致性验证/故障定位）至少包括：

1. 从存储读出事件流（tick + bar，按 event time 排序）
2. 用与实盘相同的增量计算器更新状态
3. 在信号时钟点输出特征快照/模型输出/决策结果
4. （可选）与当时实盘记录做 diff

## 五、建议额外存储的审计字段（低成本高收益）

- **事件延迟**：ingest_ts - event_ts
- **补全标记**：是否来自 gap fill
- **质量标记**：是否触发异常值过滤/替换
- **版本信息**：模型版本、特征版本、归一化参数版本、代码版本（git sha）
