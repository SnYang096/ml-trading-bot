# 数据补全架构

> 2026-02-27 定稿。本文记录实盘系统 warmup 阶段的数据补全策略。

## 核心原则

| 数据类型 | 来源 | 说明 |
|---------|------|------|
| **Ticks（buy/sell 拆分）** | Binance Vision 下载 / 本地上传 | 有完整买卖方向信息，用于 VPIN / Trade Clustering / Footprint |
| **API 补数据（warmup gap）** | Klines（OHLCV） | `exchange.fetch_ohlcv()`，60 条/小时，秒下 |
| **aggTrades API** | **仅用于实时断线重连** | warmup 路径已移除，BTC 每小时 ~50 万笔太慢 |

## Warmup 补数据策略

```
┌─────────────────────────────────────────────────────────┐
│                   Parquet 加载 (ticks + bars)            │
│              历史数据: ~200 天 1min ticks/bars            │
└────────────────────────┬────────────────────────────────┘
                         │ 检测 gap
                         ▼
┌─────────────────────────────────────────────────────────┐
│              内部 gap (数据中间的空洞)                     │
│                                                          │
│   ≤1h  → Klines API（秒下）                              │
│   >1h  → 推后台 Vision 重试                               │
└────────────────────────┬────────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────────┐
│              末尾 gap (最后数据 → 当前时间)               │
│                                                          │
│   昨天及以前 → Binance Vision (每日 CSV 包)              │
│     ├─ 成功: 写入 parquet                                │
│     ├─ 部分成功: 已下载天保存，残余 >1h → 后台重试       │
│     └─ 404: 数据未生成 → 后台重试                        │
│                                                          │
│   今天部分 → Klines API (最多补最近 1h)                  │
└────────────────────────┬────────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────────┐
│              模式判定 (SystemModeManager)                 │
│                                                          │
│   ≥2400 bars (40h) + 充足历史数据 → 直接 NORMAL          │
│   ≥240 bars (4h) → NORMAL                                │
│   ≥120 bars (2h) → DEGRADED (观察不下单)                 │
│   <120 bars → OFFLINE (等待实时数据累积)                  │
└─────────────────────────────────────────────────────────┘
```

## 后台 Vision 重试

- **触发条件**: Vision 下载返回 404（次日 ~10:00 UTC 才生成）
- **实现**: `run_live.py` 中的 asyncio task `_background_vision_retry()`
- **频率**: 每 15 分钟重试一次
- **处理**: 下载成功 → 解析 CSV → 聚合 1min bars → 写入 parquet
- **不需要重启**: 后台补的数据保存到磁盘，用于下次启动；当前进程已经是 NORMAL

## 1 天 Ticks Gap 对特征的影响

| 特征 | 计算方式 | 1 天 gap 影响 |
|------|---------|--------------|
| **VPIN** | volume-bucketed 买卖失衡 | gap 期间无 VPIN 输出，恢复后立即准确 |
| **Trade Clustering** | 连续同向成交统计 | 窗口 100 笔 tick，新数据立即覆盖 |
| **Footprint** | 单 bar 内价格层级 delta | 只看当前 bar 内 ticks，无影响 |
| **CVD** | 累积买卖差 | 从 buy_qty/sell_qty 直接算，klines 补的 gap 用 0 |
| **order_flow_delta** | CVD 归一化 | fallback 到 0.0，不影响 Gate 判定 |
| **RSI / MACD / ATR** | EMA / 滚动窗口 | klines OHLCV 补齐，完全正常 |
| **SR strength** | 全窗口批量 | 200+ 天历史 >> 1 天 gap |

**结论**: klines 补齐的 gap 保证了 OHLCV 类特征（RSI/MACD/ATR/SR）完全正常。
订单流特征在 gap 期间输出 0/NaN（安全 fallback），恢复后立即准确。
对于 200+ 天历史数据的系统，1 天 gap 影响微乎其微。
