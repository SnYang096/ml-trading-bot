# 05 补全、对账与异常处理

本章把“稳定性机制”统一成一条主线：**先保证数据连续与状态一致，再做特征与交易**。

## 一、缺口补全（Gap Filling）

典型缺口来源：

- WebSocket 断连/丢包
- 程序重启（停机期间数据缺失）
- 交易所延迟/乱序

推荐机制（与回放一致）：

1. **检测**：按 event time 检查连续性（bar 期望间隔 / tick 期望最小粒度）
2. **暂停交易**：进入“只管理持仓，不开新仓”的降级模式（避免基于错误特征决策）
3. **补全**：从 REST API 或历史数据源下载缺口段
4. **写入存储**：补全数据也写入 QuestDB（并标记补全来源）
5. **重放**：用补全后的完整事件序列重放增量状态，直到追上最新
6. **恢复交易**：连续性恢复后再开新仓

详细设计/指南见：

- `reference/实时流数据补全机制设计.md`
- `reference/实时流数据补全使用指南.md`
- `reference/Tick数据实时流与历史数据说明.md`

## 二、对账（Reconciliation）

“对账”有两层：

- **市场数据对账**：本地事件流是否连续、是否与存储一致。
- **交易状态对账**：订单/成交/持仓/余额是否与交易所一致（尤其在断连、拒单、部分成交、重试后）。

建议将对账结果绑定到策略状态机：

- 对账失败 → 熔断（停止开仓）→ 自动恢复流程（重连/刷新状态/必要时撤单）
- 对账成功 → 正常交易

Nautilus Trader 的执行引擎支持对账能力（例如启用 `reconciliation`），但仍建议在策略层保留“关键状态一致性断言/告警”。

## 三、异常发现（Detection）

建议最少覆盖以下异常类型，并统一处理策略：

- **Inf/NaN 异常**：
  - 源数据阶段发现 inf → 判定为源数据问题（需要修复数据源/加载逻辑）
  - 计算后新增 inf → 判定为计算问题（需要保护分母/对数输入/滚动窗口）
  - 参考：`docs/features/DATA_MONITORING.md`
- **乱序/重复**：同一时间戳重复 bar / 同一 trade_id 重复 tick
- **延迟抖动**：事件延迟突然放大（可能导致决策基于“过期信息”）
- **分布漂移**：特征分布/模型输出分布发生突变（需要告警与降级）

## 四、异常处理（Handling）与自愈（Self-healing）

建议策略层实现明确的“运行模式”（状态机）：

- **NORMAL**：连续性 OK、对账 OK
- **DEGRADED**：存在小异常（延迟升高/轻微缺口），允许只做风控/不开新仓
- **RECOVERING**：执行补全与重放
- **HALT**：严重异常（质量闸门失败、对账失败），停止交易，等待人工介入

每个状态都要明确：

- 是否允许开新仓
- 是否允许加仓/减仓/止损止盈
- 是否允许更新模型输入（特征快照）
