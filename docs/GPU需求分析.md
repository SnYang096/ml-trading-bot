# GPU 需求分析

## 一、特征计算 GPU 需求分析

### 1.1 当前特征计算架构

根据代码分析，你的特征计算系统主要使用：

#### ✅ CPU 计算的特征（99%）

1. **基础技术指标**
   - TA-Lib（RSI、MACD、ATR 等）
   - NumPy/Pandas 计算
   - **完全在 CPU 上运行**

2. **统计特征**
   - WPT（小波包变换）- PyWavelets
   - Hilbert 变换 - SciPy
   - Hurst 指数 - NumPy
   - 频谱特征 - NumPy/SciPy
   - **完全在 CPU 上运行**

3. **订单流特征**
   - VPIN、Trade Clustering
   - CVD、Delta 计算
   - **完全在 CPU 上运行**

4. **波动率特征**
   - GARCH - statsmodels
   - 扩展波动率特征 - NumPy
   - **完全在 CPU 上运行**

5. **形态匹配特征**
   - DTW（动态时间规整）- NumPy
   - **完全在 CPU 上运行**

#### ⚠️ 可选 GPU 计算的特征（1%）

**深度学习序列特征**（`dl_sequence_features`）

- **状态**：可选功能，默认未启用
- **位置**：`src/features/time_series/dl_sequence_features.py`
- **依赖**：PyTorch（在 requirements.txt 中已注释）
- **使用场景**：如果需要 Transformer/Mamba 序列特征
- **GPU 需求**：如果启用，可以使用 GPU 加速

**结论**：**特征计算基本不需要 GPU**，除非你启用了深度学习序列特征。

### 1.2 特征计算性能分析

| 特征类型 | 计算方式 | CPU 耗时 | GPU 需求 |
|---------|---------|---------|---------|
| 基础指标（RSI、MACD） | TA-Lib | < 1ms | ❌ 不需要 |
| WPT 特征 | PyWavelets | 10-50ms | ❌ 不需要 |
| Hilbert 特征 | SciPy | 5-20ms | ❌ 不需要 |
| 订单流特征 | NumPy | 5-15ms | ❌ 不需要 |
| GARCH 特征 | statsmodels | 20-100ms | ❌ 不需要 |
| DTW 特征 | NumPy | 10-50ms | ❌ 不需要 |
| **DL 序列特征** | **PyTorch** | **50-200ms** | **✅ 可选** |

**实时流特征计算总耗时**：约 100-300ms（CPU），完全满足实时交易需求。

## 三、实时系统 GPU 需求总结

### 3.1 实时交易流程

```
WebSocket 数据 → 特征计算（CPU）→ 模型推理（CPU）→ 决策引擎（CPU）→ 订单执行
```

### 3.2 GPU 需求评估

| 组件 | GPU 需求 | 原因 |
|------|---------|------|
| **特征计算** | ❌ **不需要** | 99% 特征使用 NumPy/SciPy/TA-Lib，完全在 CPU 上 |
| **模型推理** | ❌ **不需要** | LightGBM 推理在 CPU 上，延迟 < 1ms |
| **决策引擎** | ❌ **不需要** | 纯 Python 逻辑，CPU 足够 |
| **订单执行** | ❌ **不需要** | API 调用，网络延迟是瓶颈 |
| **数据存储** | ❌ **不需要** | QuestDB/Parquet，CPU 足够 |

### 3.3 性能分析

**实时交易延迟分解**：

```
总延迟 ≈ 网络延迟 + 特征计算 + 模型推理 + 决策生成 + 订单提交

典型值：
- 网络延迟（WebSocket）：10-50ms
- 特征计算（CPU）：100-300ms
- 模型推理（CPU）：< 1ms
- 决策生成（CPU）：< 1ms
- 订单提交（网络）：10-50ms

总计：约 120-400ms
```

**瓶颈分析**：
- ❌ **不是 GPU**：特征计算和模型推理都在 CPU 上，性能足够
- ✅ **网络延迟**：WebSocket 和 API 调用的网络延迟是主要瓶颈
- ✅ **特征计算**：某些复杂特征（如 GARCH、DTW）可能较慢，但仍在可接受范围

## 四、GPU 使用场景（可选）

### 4.1 什么时候需要 GPU？

#### 场景1：深度学习序列特征

如果你启用了 `dl_sequence_features`（Transformer/Mamba）：

```python
# 在 features.yaml 中启用
- dl_sequence_features  # 需要 GPU 加速
```

**GPU 需求**：
- **训练时**：需要 GPU（可选，CPU 也可以但很慢）
- **推理时**：需要 GPU（可选，CPU 也可以但延迟较高）

#### 场景2：大规模并行训练

如果你需要：
- 同时训练多个模型
- 大规模超参数搜索
- 快速迭代实验

**GPU 需求**：可选，可以加速但非必需

### 4.2 如果使用 GPU

#### 训练阶段（可选）

```python
# LightGBM 启用 GPU（可选）
model = lgb.LGBMClassifier(
    device="gpu",  # 启用 GPU
    gpu_platform_id=0,
    gpu_device_id=0,
)
```

**优势**：
- 训练速度提升 2-5 倍（取决于数据量）
- 对于大规模数据集有帮助

**劣势**：
- 需要安装 CUDA 和 GPU 驱动
- 增加系统复杂度
- 对于中小规模数据集，提升不明显

#### 推理阶段（不推荐）

```python
# LightGBM 推理使用 GPU（不推荐）
# 因为推理本身已经很快（< 1ms），GPU 反而可能增加延迟
```

**结论**：**推理阶段不需要 GPU**，CPU 已经足够快。

## 五、最终建议

### 5.1 实时交易系统：**不需要 GPU**

**理由**：

1. ✅ **特征计算**：99% 使用 CPU 库（NumPy、SciPy、TA-Lib），性能足够
2. ✅ **模型推理**：LightGBM 在 CPU 上推理延迟 < 1ms，完全满足实时需求
3. ✅ **系统复杂度**：不使用 GPU 可以简化部署和维护
4. ✅ **成本**：不需要 GPU 服务器，降低运行成本

### 5.2 模型训练阶段：**GPU 可选**

**推荐**：

- **小到中等数据集**（< 100万样本）：CPU 足够，训练时间可接受
- **大规模数据集**（> 100万样本）：可以考虑 GPU 加速训练

**但注意**：
- LightGBM GPU 支持需要额外配置
- 对于大多数交易策略，数据集规模不会太大
- CPU 训练通常已经足够快（几分钟到几小时）

### 5.3 如果启用深度学习特征：**需要 GPU**

如果你启用了 `dl_sequence_features`：

- **训练时**：强烈推荐 GPU（CPU 训练会非常慢）
- **推理时**：推荐 GPU（CPU 推理延迟可能达到 50-200ms）

**但注意**：
- 深度学习特征在 requirements.txt 中已注释，说明你可能不需要
- 大多数策略使用传统特征已经足够

## 七、总结

### 7.1 核心结论

| 阶段 | GPU 需求 | 推荐 |
|------|---------|------|
| **特征计算** | ❌ 不需要 | CPU 完全足够 |
| **模型推理** | ❌ 不需要 | CPU 延迟 < 1ms |
| **模型训练** | ⚠️ 可选 | 小数据集 CPU 足够，大数据集可考虑 GPU |
| **深度学习特征** | ✅ 需要 | 如果启用，需要 GPU |

### 7.2 最终建议

**实时交易系统：完全不需要 GPU**

- 特征计算：CPU 性能足够（100-300ms）
- 模型推理：CPU 延迟极低（< 1ms）
- 系统复杂度：不使用 GPU 可以简化部署
- 成本：不需要 GPU 服务器

**模型训练：GPU 可选**

- 大多数情况下 CPU 足够
- 只有大规模数据集或深度学习特征才需要 GPU

**建议**：先使用 CPU 运行，如果性能不够再考虑 GPU。
