## Pipeline 总览

训练是**两轮迭代**流程。Prefilter/Direction 分析只需要 **features + labels**（不需要模型），
通过 `--prepare-only` 直接导出全周期 `features_labeled.parquet`，无需先跑完整训练。

```
═══ Round 0: 数据准备 (--prepare-only, 不训练模型) ═══

Step 1: Feature Store 构建
  ↓
Step 2: 导出 features_labeled.parquet (--prepare-only, 全周期)
  ↓
═══ Round 1: 校准轮 (用 features_labeled.parquet 分析，更新配置) ═══
  ↓
Step 3: Prefilter 语义分层分析 (验证 archetype 特征区分力)
Step 4: Direction 验证 (方向模块质量检查，规则未变则跳过)
  ↓ (人工审核，更新 prefilter.yaml / direction.yaml)
  ↓
═══ Round 2: 正式训练 (带 prefilter + direction) ═══
  ↓
Step 5: Gate 正式训练 + 优化
  ↓
Step 6: Entry Filter 优化
  ↓
Step 7: Execution 优化
  ↓
Step 8: 单策略回测
  ↓
Step 8.5: PCM 联合回测 (Regime-Aware v3, 需 ≥2 策略)
```

> **Evidence 已删除**（原 Step 6）：实测 Spearman r=-0.05，p=0.48，仓位缩放等于随机噪声。
> 删除后 Sharpe 不变或微升。参见 `docs/z实验_005_统一研究/架构设计_简化版.md` 第三节。

> **features_labeled.parquet 内容**: 全周期 (train+test) 的特征列 + 标签 (success_no_rr_extreme) + 元数据 (timestamp, symbol, direction 等)。
> 与 predictions.parquet 的区别: 没有模型预测值 pred，但覆盖全周期（不仅是 holdout）。

### Step 3: Prefilter 语义分层分析

> **前置依赖**: Step 2 产出的 `features_labeled.parquet`（全周期数据）

验证 ME archetype 特征的区分力，确认 pre_filter 条件是否合理。
使用 `--meta-algorithm --promote` 路径与自动管线行为一致（LightGBM → SHAP∩Gain → holdout 验证 → range 模式）。

| 方法              | 含义                             | 评价                           |
| ----------------- | -------------------------------- | ------------------------------ |
| **ks**            | KS 统计量                        | 原始方法，现在默认             |
| **lift**          | pass 区 / deny 区收益倍数        | 效果一般                       |
| **combined**      | lift + ks 加权                   | 居中                           |
| **bad_rate_lift** | deny 区 bad_rate（标签=0）集中度 | **现在管线主力**，uplift=+1.69 |
| **positive_rr**   | pass 区 rr > 0.8R 的比例         | 你说的 >0.8R 方案              |

**关于 bad_rate_lift 和 gate 的区别**：

|          | Prefilter (bad_rate_lift)                  | Gate                                 |
| -------- | ------------------------------------------ | ------------------------------------ |
| **目标** | 识别"哪些条件下 ME 不成立"（拒绝噪声数据） | 识别"哪些条件下 ME 成立但交易质量差" |
| **标签** | `success_no_rr_extreme`（0/1）二分类       | 同样的标签                           |
| **信号** | 极值区间划分（range）                      | 连续值打分                           |

**本质上很相似**，都用同一个标签。但 Prefilter 用 **阈值硬切**（range 区间），Gate 用 **连续值软判**。如果 Prefilter 筛得太狠，Gate 就没数据可学了——这就是为什么管线设置了 `min_pass_rate: 0.05`（至少保留 5% 数据）。

```bash

# top_rules（默认路径）	meta-algorithm（新路径）
# 特征发现	百分位扫描（P5/P10/P90/P95）	LightGBM + SHAP∩Gain
# 模式	单阈值（>= P95）	range 双边区间（>= P5 + <= P10）
# 验证	累积 AND + split 前样本检查	holdout + 3-fold robustness
# 当前状态	代码里有，但效果差	代码里有，现在是管线默认
# 结论：管线（auto_research_pipeline.py）现在默认走 --meta-algorithm 路径，top_rules 是旧逻辑，理论上还存在但基本不用了。你当前 prefilter.yaml 里的 3 条规则都是 meta-algorithm 生成的（KS=0.50 和 0.22），这就是为什么它们能找到 range 模式而 top_rules 找不到。
```

**Prefilter 参数速查**:

| 参数                          | 说明                                                                           |
| ----------------------------- | ------------------------------------------------------------------------------ |
| `--logs`                      | features_labeled.parquet 路径 (必须)                                           |
| `--strategy`                  | 策略名: bpc / me / fer / lv (必须)                                             |
| `--config`                    | prefilter.yaml 路径 (默认: `config/strategies/{strategy}/prefilter.yaml`)      |
| `--all-features`              | 发现模式: 跳过 prefilter.yaml, 扫描 parquet 全部数值特征 (169+)                |
| `--select-recent N`           | Mode A: 用最近 N 个月做特征选择，全周期做 temporal rolling (隐含 `--temporal`) |
| `--temporal`                  | 时间稳定性分析: rolling bad_rate 曲线 + CV + 多窗口自动选择                    |
| `--percentiles`               | 百分位列表 (默认: 5,10,20,80,90,95)                                            |
| `--start-date` / `--end-date` | 过滤数据时间范围                                                               |
| `--output`                    | 输出 JSON 报告路径 (可选)                                                      |

**两种模式的区别**:

|          | 默认 (配置模式)                                           | `--all-features`                            |
| -------- | --------------------------------------------------------- | ------------------------------------------- |
| 候选来源 | `prefilter.yaml` candidates → `feature_dependencies.yaml` | parquet 全部数值列 (169+)                   |
| 用途     | 已有候选的评估与排名                                      | 启发式想不出时，用数据发现新候选            |
| 筛选条件 | 按 config 的 _f 分组解析                                  | 自动过滤: 非数值/标签/元数据列排除          |
| 回写     | → `prefilter.yaml` last_evaluation                        | → `prefilter.yaml` last_evaluation (若存在) |

**配置文件链路** (配置模式):
```
config/strategies/me/prefilter.yaml  (候选 _f 分组名)
  → config/feature_dependencies.yaml  (output_columns 解析实际列名)
  → parquet 中匹配存在的列 → 逐列分位数分层
```

> **想扩大搜索范围？** 两种方式：
> 1. `--all-features`: 直接扫描 parquet 全部数值特征，无需修改任何配置
> 2. 在 `prefilter.yaml` 的 candidates 里加入更多 `_f` 分组名，如 `evt_features_f`、`spectrum_features_f` 等

**自动回写**: 脚本运行后自动更新 `prefilter.yaml` 的 `last_evaluation` 段，
包含正信号/反信号/低端信号 + temporal CV。人工审核后才更新 `archetypes/prefilter.yaml`。

**关注指标**：
- 正信号（high=good）: bad rate 差异 > 5% → 可做 pre_filter 正向条件
- 反信号（high=bad）: bad rate 差异 > 5% → 可做反向 hard_gate
- temporal CV < 0.3 稳定，< 0.5 一般，>= 0.5 不稳

**操作**: 审核报告后，更新 `config/strategies/me/archetypes/prefilter.yaml`。

**Direction 参数速查**:

| 参数                 | 说明                                                                                             |
| -------------------- | ------------------------------------------------------------------------------------------------ |
| `--logs`             | features_labeled.parquet 路径                                                                    |
| `--strategy`         | 策略名: bpc / me / fer / lv                                                                      |
| `--compare-features` | 对比模式: 读 `config/strategies/{strategy}/direction.yaml` 候选，按 Median Lift 排名             |
| `--all-features`     | 发现模式: 扫描 parquet 里全部数值特征 (需配合 `--compare-features`)                              |
| `--temporal`         | 时间稳定性: rolling [2m,3m,4m,6m] 窗口 Median Lift + CV + 衰减检测，稀疏特征自适应降低样本量要求 |

**两种模式的区别**:

|          | `--compare-features`                | `--compare-features --all-features`      |
| -------- | ----------------------------------- | ---------------------------------------- |
| 候选来源 | `direction.yaml` candidates (17 个) | parquet 全部数值列 (169+)                |
| 用途     | 已有候选的评估与排名                | 启发式想不出时，用数据发现新候选         |
| 筛选条件 | 跟随 config 的 transform            | 自动检测: [0,1]→center_sign，有正负→sign |
| 过滤     | 无 (config 的都分析)                | Short 占比 15%-85% 才保留                |
| 结果回写 | → `direction.yaml` last_evaluation  | → `direction.yaml` last_evaluation       |

**自动回写**: `--compare-features` 运行后自动更新 `config/strategies/me/direction.yaml` 的 `last_evaluation` 段，
包含 positive_lift + fallback + temporal_cv + recent_decay。人工审核后才更新 `archetypes/direction.yaml`。

**Phase 1 通过标准** (覆盖率)：
- 覆盖率 100% (每个 bar 都有方向)
- Short 占比 > 15%

**Phase 2 通过标准** (方向质量)：
- `median(rr_in_direction) > median(forward_rr_long)` → 方向模块优于 always-long
- `mean(rr_in_direction) > 0` → 按方向交易总体不亏
- Bad rate(按方向) < Bad rate(always-long) → 踩坑率降低
- Short 子集样本量 >= 1080 → 统计可信

**--compare-features 评估指标**:
- Median Lift = median(rr×direction) - median(rr_always_long)，正值 = 方向有正贡献
- temporal CV < 0.3 稳定，< 0.5 一般，>= 0.5 不稳
- 近 3 窗口 >= 2 个负 Lift → 信号衰减

> **ME 方向特征的特殊性**: ME 的 alpha 不在方向准确率，而在 Gate（识别大波动）+ 执行层盈亏比（小止损大止盈）。
> 方向特征只需满足 **全周期 Lift > 0**（略好于 always-long），不要求每个窗口都正。
> temporal CV 高（如 3.53）是 ME 策略结构的自然特征，不是缺陷。
> 实际 alpha 质量应看 Step 5 Gate 训练后的回测表现（盈亏比 × 胜率 的组合）。

**操作**: 如果 Phase 2 不通过，检查 `direction.yaml` 规则或暂时禁用 `direction_aware`。
