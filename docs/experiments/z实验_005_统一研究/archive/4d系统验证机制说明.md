# 4D 系统验证机制说明

## 概览

4D 系统的每个模块有不同的验证机制：

| 模块      | 优化方式   | KPI Gate 文件                           | 验证脚本                                      |
| --------- | ---------- | --------------------------------------- | --------------------------------------------- |
| Prefilter | 自动优化   | `config/kpi_gates/prefilter_layer.yaml` | `analyze_archetype_feature_stratification.py` |
| Gate      | 自动优化   | `config/kpi_gates/gate_layer.yaml`      | `optimize_gate_unified.py`                    |
| Evidence  | 自动优化   | `config/kpi_gates/evidence_layer.yaml`  | `optimize_evidence_plateau.py`                |
| Direction | **不优化** | **无**                                  | `direction_strict_validation.py`              |

---

## 1. Direction — 确定性模块，不需要优化

### 为什么没有 KPI Gate

Direction 是 **deterministic** 的——方向由因果结构决定，不让模型学：

| Archetype | 方向来自 | 因果来源 | 原则     |
| --------- | -------- | -------- | -------- |
| BPC       | 结构方向 | 突破     | 跟结构走 |
| ME        | 动能方向 | 加速     | 跟钱走   |
| FER       | 失败方向 | 反转     | 跟失败走 |

没有"阈值搜索"或"参数调优"，因此不走 kpi_gates 体系。

### 有效性如何验证

`direction_strict_validation.py` 做两阶段验证：

**Phase 1 — 覆盖率验证**：
- direction.yaml 规则能否给 100% 的样本分配方向
- 每条规则的命中数统计

**Phase 2 — 方向质量验证（核心）**：

```
核心公式: rr_in_direction = forward_rr × direction

H0: direction 与 returns 无关 (随机方向, median ≈ 0)
H1: median(rr × direction) > 0 (方向信号有信息)

通过标准:
  1. median(rr × direction) > 0
  2. 置换检验 p < 0.05 (500 次 permutation)
  3. short 子集样本 >= 1080
  4. short 占比 > 15%
```

### Pipeline 集成

在 `auto_research_pipeline.py` 中，Direction Validate 在 Gate Train **之前**执行：

```
Direction Validate → Gate Train → Gate Optimize → Evidence Train → Evidence Optimize
```

如果方向验证失败，后续步骤没有意义。

### 候选特征对比

`direction_strict_validation.py --compare-features` 会：
1. 扫描 direction.yaml 中声明的候选特征
2. 对每个候选计算 `median(rr × sign(feature))`
3. 做置换检验
4. 按 median_rr_in_dir 降序排名
5. `--promote` 自动选择最佳候选写入 direction.yaml

### direction_research.py 的作用

这是一个**对比研究脚本**（非管线组件），用于比较：
- 旧逻辑：硬编码列名检测 (`{arch}_breakout_direction` → `entry_direction` → `direction`)
- 新逻辑：direction.yaml 规则驱动

输出方向一致率和变更明细，验证从旧逻辑迁移到 direction.yaml 的正确性。

## 2. Evidence 与 Gate 的特征分离

### 问题

Gate 和 Evidence 都从同一个 LightGBM 模型的 SHAP∩Gain 发现候选特征，存在 double counting 风险。

### 答案：不需要手动裁剪 features_evidence.yaml

`features_evidence.yaml` 是 **训练特征集**，应该尽量宽泛，让 Return Tree 自由发现。特征排除发生在 **优化阶段**，而非训练阶段。

### 自动排除机制（`--gate-yaml`）

`optimize_evidence_plateau.py --gate-yaml gate.yaml` 在优化时自动执行三步排除：

```
步骤 1: _extract_gate_features(gate.yaml)
         → 解析 hard_gates 的 when 条件，提取 gate 使用的特征名
         → 例: {bpc_volume_compression_pct, bpc_bb_compression, bpc_cvd_z}

步骤 2: _find_gate_correlated(df, gate_features, candidates, threshold=0.7)
         → 对每个 evidence 候选，计算与 gate 特征的 Pearson |r|
         → |r| > 0.7 的候选被排除（防止 double counting）

步骤 3: _spearman_prescreen(df, remaining, rr_col, p_threshold=0.1)
         → 在 gate 放行子集上做 Spearman(feature, RR)
         → 只保留 r > 0 且 p < 0.1 的特征
```

### 数据流

```
features_evidence.yaml (宽泛, ~80 个特征组)
        ↓
  Return Tree 训练 → predictions.parquet
        ↓
  Gate Apply → logs_gated.parquet (gate 放行子集)
        ↓
  optimize_evidence_plateau.py --gate-yaml gate.yaml
        ↓
  [自动排除 gate 特征] → [排除高相关] → [Spearman 预筛]
        ↓
  evidence.yaml (精选 2-6 个特征 + 权重 + 方向)
```

### 各 Archetype 的 Gate-Evidence 重叠风险

| Archetype | Gate 特征                                                   | Evidence 特征                                  | 重叠风险                   |
| --------- | ----------------------------------------------------------- | ---------------------------------------------- | -------------------------- |
| BPC       | bpc_volume_compression_pct, bpc_bb_compression, bpc_cvd_z   | shd_pct, macd_signal_atr                       | 低 (不同家族)              |
| FER       | fer_trapped_longs/shorts_score, fer_volume_price_divergence | roc_5, cvd_change_5, fer_signed_efficiency 等  | **高** (4/6 同属 fer 家族) |
| ME        | atr_percentile, me_cvd_alignment, evt_var/es_99_right       | vpin_volatility_20, vpin_ma20, evt_scale_right | **中** (EVT 家族重叠)      |

`--gate-yaml` 自动处理这些重叠，无需人工维护。

## 3. KPI Gate 阈值速查

### Prefilter Layer (`prefilter_layer.yaml`)

| 指标             | 阈值         | 含义              |
| ---------------- | ------------ | ----------------- |
| min_ks_statistic | 0.05         | KS 分布分离度下限 |
| max_ks_pvalue    | 0.01         | KS 统计显著性     |
| min_robustness   | 0.3          | time-fold 一致性  |
| max_rules        | 4            | 最多 4 条规则     |
| deny_rate        | [0.03, 0.70] | 拒绝率合法范围    |

### Gate Layer (`gate_layer.yaml`)

| 指标           | 阈值         | 含义                                      |
| -------------- | ------------ | ----------------------------------------- |
| min_lift       | 1.05         | bad 浓度提升                              |
| min_effect     | 0.10         | mean_rr(allow) - mean_rr(deny)            |
| min_gate_score | 0.0          | Youden's J: tail_capture - good_deny_rate |
| min_robustness | 0.4          | time-fold 一致性                          |
| deny_rate      | [0.05, 0.70] | 拒绝率合法范围                            |

### Evidence Layer (`evidence_layer.yaml`)

| 指标                   | 阈值  | 含义                               |
| ---------------------- | ----- | ---------------------------------- |
| min_bad_suppression    | 0.05  | P(score<0.3                        | bad) - P(score<0.3  | good) |
| min_good_amplification | 0.05  | P(score>0.7                        | good) - P(score>0.7 | bad)  |
| min_sharpness          | -0.02 | mean(good_score) - mean(bad_score) |

### Direction（无 KPI Gate 文件）

| 指标                   | 阈值    | 含义             |
| ---------------------- | ------- | ---------------- |
| median(rr × direction) | > 0     | 方向信号优于随机 |
| 置换检验 p-value       | < 0.05  | 统计显著         |
| short 子集样本         | >= 1080 | 统计可信         |
| short 占比             | > 15%   | 方向平衡         |
