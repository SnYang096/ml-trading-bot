# 抓肥尾、杠杆与 Archetype 统一口径（BPC / ME / FER / LV）

## 0. 这份文档解决什么

把以下问题统一到一份口径里，避免多文档反复切换：

1. 抓肥尾是否等于“押 BTC 大牛市 + 高杠杆”？
2. 合约可多可空，如何“充分利用杠杆”，尤其是你提到的“无风险”？
3. BPC / ME / FER / LV 到底如何分工，避免语义混淆？

---

## 1) 抓肥尾，不等于高杠杆梭哈

### 结论先行

- 抓肥尾的本质是：**让少数大盈利单活下来**（holding + pyramiding + 风险控制）。
- 不是：把目标改成“几十倍收益 + 高杠杆”。
- 和“稳健年化几十个点”**不冲突**，冲突的是“高杠杆 + 高回撤 + 高频爆仓”。

### 正确理解

- 肥尾收益来自收益分布右尾，通常由少数大行情贡献大部分利润。
- 你系统里的 alpha 来源可以是：
  - BPC：结构重定价后延续；
  - ME：动量扩散加速；
  - FER：控制权翻转；
  - LV：杠杆脆弱触发的被迫移动。
- 这些 alpha 是否兑现，主要取决于**持仓管理**，而不是把杠杆开到很高。

### 与稳健收益的关系

- 稳健目标（如年化几十个点）靠的是：
  - 低相关策略组合；
  - 单笔风险上限；
  - 容忍小亏、放大利润；
  - 长期复利。
- 追求“单次几十倍”通常意味着更高路径风险，不是稳健系统的主目标。

## 2) 关于“无风险利用杠杆”：现实里几乎不存在

### 先把话说死

- **方向性交易中的无风险杠杆不存在**。
- 你能做多做空，只是双向交易能力，不代表无风险。
- 任何杠杆都会放大：
  - 价格路径风险；
  - 流动性风险；
  - 强平风险；
  - 资金费率与滑点成本。

### 真正接近“低风险”的只有相对价值/对冲结构

- 例如现货-永续基差、跨交易所价差、统计套利等。
- 但也不是无风险，仍有：
  - 基差收敛不达预期；
  - 交易对手与平台风险；
  - 流动性踩踏；
  - 执行和风控失效。

### 你可以用的“安全杠杆”框架（实用）

- 单笔固定风险：每笔只亏账户净值 `0.25%~0.75%`（而不是固定仓位）。
- 波动率缩放：波动上来自动降杠杆，波动回落再恢复。
- 分层加仓（pyramiding）只用浮盈加，不用本金硬加。
- 触发型降杠杆：连续亏损、相关性抬升、盘口流动性下降时，自动 cut leverage。
- 强制生存约束：任何时点优先“不会被抬走”，再谈收益最大化。

> 一句话：不是“无风险地把杠杆开大”，而是“有纪律地在可承受风险内使用杠杆”。

## 3) 统一的 Archetype 口径（建议落地版本）

### 保留四类：BPC / ME / FER / LV

- 不新增独立 `trend-continue-shallow` archetype。
- 不把 `pullback` 单独拉成与 BPC 平级 archetype。
- `Trend` 视为状态，不是独立 archetype（常是 Inventory + Energy 同时中高）。

### 四类定义（避免互相污染）

- **BPC（结构确认）**
  - 关键词：breakout 后回踩确认再延续。
  - 核心因果：库存转移与结构重定价。
  - 不是追突破大阳线本身。

- **ME（动量扩散）**
  - 关键词：速度与能量加速。
  - 核心因果：流动性快速注入与订单流推进。
  - 不要求必须有清晰 SR 结构。

- **FER（博弈失败反转）**
  - 关键词：失败 -> 衰竭 -> 控制权翻转。
  - 核心因果：方向控制权由一方切换到另一方。

- **LV（杠杆脆弱触发）**
  - 关键词：OI/FR/清算燃料导致被迫移动。
  - 核心因果：杠杆集中 + 流动性薄弱。

### 三种常见错误（你现在最该避免）

- 用 ME 逻辑去做 breakout 结构确认（ME 被拉回 BPC）。
- 把 `pullback` 作为泛概念，与 BPC/ME 大量重叠。
- entry 已分 archetype，但 exit/holding 还用统一模板。

## 4) 抓肥尾的执行重点：按 Archetype 区分 Holding

这是抓肥尾是否成功的关键。

- **BPC 持仓**
  - 更宽容回撤，允许震荡中延续。
  - 核心目标：保留长尾趋势单。

- **ME 持仓**
  - 更快锁盈，防止动量衰竭后的利润回吐。
  - 核心目标：吃到爆发段，不恋战。

- **FER 持仓**
  - 更短周期兑现，反转段通常短而尖。
  - 核心目标：快进快出，避免“反转再反转”。

- **LV 持仓**
  - 强执行纪律，防止极端波动反噬。
  - 核心目标：利用极端失衡，但不与极端硬扛。

## 5) 回到你的三个问题（最终答案）

### Q1. 抓肥尾是否意味着“BTC 大牛市几十倍 + 高杠杆”？会不会和稳健收益冲突？

- 不意味着。
- 抓肥尾是收益分布管理问题，不是杠杆豪赌问题。
- 与稳健收益不冲突；真正冲突的是无纪律高杠杆和高回撤路径。

### Q2. 合约可多可空，如何“无风险”充分利用杠杆？

- 方向盘上没有无风险杠杆。
- 只能做“风险可控的杠杆使用”：固定风险、波动率缩放、浮盈加仓、触发降杠杆、严格止损与限损。
- 若追求更低风险，需要做对冲/相对价值结构，但依然不是零风险。

### Q3. 能否合并成一个文档？

- 可以，这份就是统一版。
- 后续建议只维护这一份作为总口径，其他文档做补充案例，不再重复定义。

## 6) 落地版职责分工（已采用）

### 谁负责牛熊肥尾

- **BPC-long / BPC-short**
  - 牛市/熊市肥尾主仓。
  - 目标不是高胜率，而是把真正的大趋势拿住。
  - 可以加仓，但前提必须是浮盈已锁定。

- **ME-long**
  - 负责趋势中的动量扩散加速段。
  - 不是整段趋势主仓，但可以作为“第二发动机”补仓。
  - 允许 2 次加仓，目的是把扩散段的右尾拉长。

- **ME-short**
  - 仍视为战术空头，不承担熊市尾部主仓职责。
  - 原因是下跌动量常常更快衰竭、反抽更暴力，路径风险更大。
  - 当前口径：允许 2 次加仓，但 holding 仍短、风控更紧。

- **FER**
  - 只做失败反转后的短促兑现，不做长尾主仓。
  - 不加仓，优先快速兑现。

- **LV**
  - 只做杠杆脆弱触发后的极端失衡修复，不做长持有。
  - 不加仓，且风险预算最低。

### 风险预算（首枪，当前实现）

- `BPC-long / BPC-short`: `2.00%` 净值风险
- `ME-long`: `0.35%`
- `ME-short`: `0.30%`
- `FER-long / FER-short`: `0.35%`
- `LV`: `0.25%`

> 说明：BPC 已切到 2.00% 首枪风险，其余 archetype 保持轻仓预算。

## 7) Holding 层统一口径（已采用）

### BPC

- `initial_r` 宽，允许回撤中的趋势延续。
- `trailing` 激活较晚、`trail_r` 极宽，只做灾难保护。
- `time_stop_bars = 0`，默认无限持有。
- 主退出优先看结构破坏（例如 `ema200`），不是固定 TP。

### ME-long

- 比 BPC 更快激活 trailing，防止动量衰竭后利润回吐。
- 不再无限持有，默认 `time_stop_bars = 72`（60T 下约 3 天）。
- 允许 2 次加仓，但加的是扩散段，不是整段趋势。

### ME-short

- 与 ME-long 同样使用更快的 trailing，但 holding 更短。
- 默认 `time_stop_bars = 48`（60T）/ `36`（120T）。
- 当前允许 2 次加仓（ATR 阶梯触发），但仍按战术短打管理。

### FER

- 保留固定 `TP + time_stop` 的兑现逻辑。
- `time_stop_bars` 维持短周期，避免“反转再反转”。
- 目标是拿失败翻转后的那一段，不把它误当趋势单。

### LV

- 最紧的 holding 和最短的 time stop。
- 只利用极端失衡，不与极端反噬硬扛。

## 8) 加仓层统一口径（已采用）

### 哪些单可以加

- 允许加仓：**BPC-long / BPC-short / ME-long / ME-short / ME-short-120T**。
- 不允许加仓：**FER / LV**。

### 加仓前提

- 必须是**顺势加仓**，禁止逆势摊薄、禁止亏损加仓。
- 父仓必须已经 `breakeven locked`。
- 当前实现的利润锁定阈值为：`current_r >= 1.0R`。
- 只有同 `symbol + direction` 的活跃仓位都已锁盈，才允许开 add-on。

### 触发规则

- **BPC**：`bpc_follow_signal`（跟随同向结构信号）+ 前仓全锁盈。
- **ME**：`me_atr_step_add`（`atr_step=0.5`）+ 动量同号校验 + 新高/新低邻近约束。

### 仓位梯度（fixed 模式默认）

- **BPC 5 次**：`[0.5, 0.35, 0.25, 0.20, 0.15]`，`min_current_r_by_add=[1.0,1.75,2.5,3.5,4.5]`。
- **ME 2 次**：`[0.4, 0.25]`，`min_current_r_by_add=[1.0,2.0]`。
- execution 层支持 `sizing_mode: target_leverage_gap`（方案 C），可按杠杆缺口补仓并受上限裁剪。

### 止损与仓位同源（关键约束）

- 统一使用：`effective_stop_pct = clip(atr_stop_pct, min_stop_pct, max_stop_pct)`。
- **仓位 sizing 与初始止损都用同一个 `effective_stop_pct`**。
- 禁止“按护栏算仓位、按 ATR 下止损”的双口径实现。

### 组合风险约束

- 加仓必须使用“已释放”的风险预算。
- 老仓 stop 至少已移到 breakeven，新仓才允许占用那一部分释放出来的风险。
- 所以系统追求的是：
  - **名义仓位上去**
  - **净最坏风险不上去**

### 冻结条件（操作口径）

- funding / OI 已进入极端拥挤区间时，不继续加。
- 盘口明显变薄、滑点恶化时，不继续加。
- 同方向相关仓位已经过多时，不继续加。
- 出现连续假突破 / 假跌破时，不继续加。

> 一句话：能加仓，不代表应该加仓。只有“利润已锁定 + 新一段优势再次出现”时才加。

## 9) 抓肥尾 KPI 与上线门槛（已采用）

后续不再只看 Sharpe，还要固定看这几项：

1. **尾部贡献率**
   - 定义：`top 10% winning trades profit / total winning profit`
   - 用来判断系统是否真的靠少数大单赚钱。

2. **加仓 expectancy**
   - 定义：`mean(realized_rr of add-on trades)`
   - 用来判断加仓是不是在放大利润，而不是在高位补风险。

3. **MFE 捕获率**
   - 定义：`realized_rr / max_favorable_rr`
   - 只对 `max_favorable_rr > 0` 的交易统计。
   - 用来判断 holding/trailing 是否过早把肥尾卖掉。

4. **加仓后回撤恶化率**
   - 看加仓后组合表现是否显著增大回撤或降低 capture。

5. **杠杆达标率**
   - 定义：达到目标杠杆（如 5x）的交易占比。
   - 用来评估方案 C（target leverage gap）是否真正提高“达标能力”。

6. **止损同源诊断**
   - 字段：`atr_stop_pct`、`effective_stop_pct`、`sizing_stop_source`。
   - 用来审计是否存在口径漂移或护栏异常裁剪。

上线门槛口径（A/B）：

- `tail_contribution_rate` 不下降；
- `add_expectancy` 不下降；
- `MFE_capture` 不下降；
- `max_dd` 增幅在可控阈值内；
- `leverage_hit_rate` 提升。

若后续报告里这些指标变差，即使 Sharpe 还行，也不应视为“抓肥尾做对了”。

## 10) 多 symbol slot 融合方案（当前口径）

- 总风险硬上限固定：`total_risk_cap = 10%`。
- 固定槽位：`FER=1`、`ME=1`。
- BPC 软扩容：`1 -> 2 -> 3`，扩容条件包含：
  - 回撤/日损在阈值内；
  - BPC 活跃仓位数量达到阈值；
  - 风险余量足够（不突破 total risk cap）。
- 回落机制：条件不满足时，BPC 可用软槽位自动降级，优先保证组合生存性。

## 11) `hard_max_total_notional_pct` 到底限制了什么（公式 + 例子）

### 定义

- `hard_max_total_notional_pct` 是**组合总名义敞口硬上限**，不是“保证亏损不超过 X%”。
- 设：
  - 账户权益：`E`
  - 硬上限比例：`H = hard_max_total_notional_pct`
  - 组合总名义敞口：`N_total`
- 则硬约束是：
  - `N_total <= H * E`

当触达硬上限时：

- 新开仓拒绝；
- 普通加仓拒绝；
- 即使是 winner priority 加仓，也不能突破硬上限。

### 它如何影响“最大可能亏损”

`hard_max_total_notional_pct` 本身不直接等于最大回撤，但会把“极端行情的损失斜率”压住：

- 极端单边跳变近似损失上界（忽略资金费和手续费）：
  - `Loss_pct_equity ~= H * Shock_pct`
  - 其中 `Shock_pct` 是持仓方向上的瞬时不利价格变化比例（例如 10%）。

这条近似式的意义是：硬上限把“价格冲击 -> 权益损失”的放大系数固定为 `H`。

### 三个直观例子

1. **`H=6.0`，遇到 5% 极端不利跳变**
   - `Loss_pct ~= 6.0 * 5% = 30%`
   - 含义：如果组合满到 6x 名义敞口，5% 的瞬时不利冲击可造成约 30% 权益回撤。

2. **`H=3.0`，同样 5% 跳变**
   - `Loss_pct ~= 3.0 * 5% = 15%`
   - 含义：把硬上限减半，极端冲击下损失斜率也减半。

3. **`H=6.0`，但当前只用了 `N_total=3.8E`**
   - 有效放大系数是 `3.8`（不是 6.0）
   - 10% 跳变时近似 `Loss_pct ~= 3.8 * 10% = 38%`
   - 含义：真实风险由“当前占用敞口”决定，`H` 只是天花板。

### 为什么它还能兼顾 fat-tail

- 你要抓肥尾，冠军仓（例如 BPC）需要在盈利链条里走到高杠杆区（接近 5x）。
- 若 `H` 太低（如 3.0），冠军仓很容易在中途被敞口上限截断。
- 若 `H` 适中（如 5.5~6.0），配合“软上限 + winner priority”：
  - 普通新开仓先被挡住；
  - 但已盈利且满足 `breakeven_locked + current_r` 的主仓仍可继续加到硬上限附近；
  - 从而提升“冠军仓走满 5x”的概率。

### 实操口径（与当前讨论一致）

- 软上限：`soft_max_total_notional_pct = 4.5`
- 硬上限：`hard_max_total_notional_pct = 6.0`
- 策略：
  - `< 4.5`：正常开仓/加仓
  - `4.5 ~ 6.0`：禁止普通新 slot，优先放行盈利主仓加仓（winner priority）
  - `>= 6.0`：全部拒绝（硬停）

### 相对“初始本金”的损失怎么算

上面的 `Loss_pct_equity` 是对**当前权益 E** 的损失比例。若要看“是否伤本金”，需换成初始本金 `P0` 口径：

- 设当前累计收益率为 `g`，则 `E = P0 * (1 + g)`。
- 冲击后权益：`E_after = E * (1 - Loss_pct_equity)`。
- 相对初始本金的回撤：
  - `PrincipalLoss_vs_P0 = max(0, 1 - E_after / P0)`
  - `= max(0, 1 - (1 + g) * (1 - Loss_pct_equity))`

例子（`H=6.0`, `Shock=5%` => `Loss_pct_equity=30%`）：

- `g = 0%`：`E_after = 0.70 P0`，本金回撤 `30%`
- `g = +20%`：`E_after = 0.84 P0`，本金回撤 `16%`
- `g = +50%`：`E_after = 1.05 P0`，不伤本金，但会明显回吐浮盈

结论：`breakeven` 是必要条件，但不是充分条件；是否伤本金取决于冲击前的利润垫与当时总名义敞口。

### 5分钟暴跌10%并快速拉回：如何防本金损失/爆仓离场

Crypto 常见“先砸后拉”的路径风险，核心不是信号对错，而是**是否先活下来**。建议分层防护：

1. **仓位层（第一道，最有效）**
   - 强制 `hard_max_total_notional_pct`，把损失斜率锁死。
   - 建议同时加：
     - 单 symbol 名义上限：`max_symbol_notional_pct`
     - 单家族名义上限：`max_family_notional_pct`
   - 目标：避免单点暴跌时组合整体被同向敞口拖穿。

2. **加仓层（第二道）**
   - 保留 `breakeven_locked`，并要求更严格的 `current_r` 阶梯（越后层越高）。
   - 在高波动拥挤区（funding/OI 异常）冻结后续加仓。
   - 软上限区（如 `4.5~6.0`）只允许 winner priority，不再开普通新 slot。

3. **执行层（第三道）**
   - 所有退出挂单必须交易所托管（reduce-only + close-position）。
   - 采用 OCO/保护单组合，避免本地进程异常导致“无保护裸仓”。
   - 对于暴跌场景，优先“可成交”而非“理想价格”，允许退出滑点但禁止失控扩大。

4. **系统层（第四道）**
   - 新增“快损熔断”：如开仓后 `<=10s` 触发止损进入 `ABNORMAL`（已实现）。

### 12.2 名义敞口风控分层（当前实现口径）

- **Normal 区 (`total_notional < soft`)**: 新 slot / 普通加仓 / winner 加仓都可评估通过。
- **Soft 区 (`soft <= total_notional < hard`)**:
  - 拒绝普通新 slot（给冠军仓让路）。
  - 仅允许满足 `winner_priority` 的加仓（默认 `BPC + breakeven_locked + current_r>=1.0`）。
- **Hard 区 (`total_notional >= hard`)**: 新 slot 与所有加仓一律拒绝。
- **Symbol/Family 子上限**: 任一 symbol 或 family 到达上限时，局部拒绝，防止集中爆仓风险。
