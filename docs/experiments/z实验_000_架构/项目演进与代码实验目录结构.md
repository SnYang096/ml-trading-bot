# 项目演进与代码+实验目录结构

> **核心论点：这个项目的目录结构不是"设计出来的"，而是"长出来的"。**
> 每一次架构变迁都是一次认知升级，最终形成了"代码不变 + 实验不断"的研究范式。

---

## 一、演进时间线 —— 五个阶段

### Stage 0：纯 if/else 规则策略（最早期）

**代表**：`rule_based_strategies/bt_sr_fake_breakout/`

**思路**：写一个简单的突破策略，用 Backtrader 跑回测，手动调参。

```
base_strategy.py  - 380 行 Backtrader 策略
  ↓
订单流确认 + POC 检测 + 结构止损
  ↓
5 个权重 × 3 个动态阈值 = 8 个旋钮
  ↓
人工拍脑袋调参 → 回测赚钱 → 上线
```

**认知水平**：
- 相信"我能找到一组参数赚钱"
- 不知道为什么赚钱，不知道什么时候会失效
- 代码和参数高度耦合，改一行代码就要重新调参

**结果**：能跑起来，但 Sharpe 不稳定。好的时候 1.5，差的时候 -0.3。不知道原因。

---

### Stage 1：if/else + 参数搜索（Optuna 时代）

**代表**：`rule_based_strategies/intraday_sniper/`

**思路**：规则策略参数太多 → 用 Optuna 自动搜索。

```
strategy.py   - 1229 行主策略
config.yaml   - 83 行参数配置
optimize.py   - 501 行 Optuna 搜索
  ↓
18 个参数空间 → Optuna 搜索
  ↓
"最优参数" → 上线
```

**认知升级**：
- 意识到"人工调参效率太低" → 引入自动化搜索
- 开始接触"特征工程"（压缩检测、CVD 评分器、突破质量评分）
- 评分函数 `score = profit_score + trade_score + efficiency_score` 本身也是拍脑袋

**新问题**：
- 18 个参数超出 Optuna 有效范围（>10 维基本是随机搜索）
- "最优参数" = 过拟合，实盘表现和回测差距巨大
- 参数之间强耦合，改一个影响所有
- **不知道赚的是哪种钱，也不知道什么时候会停止赚钱**

---

### Stage 2：上先进技术 —— NN 多头 + Router + 黑盒 ML（追求"最强"）

**代表**：`nnmultihead/`（已归档）、大量 `docs/archive/` 文档

**思路**：传统规则不够 → 上 NN、上 Transformer、上 Mamba、上 Router、上 RL。

```
NN 多头路径模型:
  dir 头  → 方向预测
  mfe 头  → 最大有利空间
  mae 头  → 最大不利空间
  mtt 头  → 持仓时间预测
    ↓
Router（概率分配）:
  ME: 0.4 / BPC: 0.35 / Fade: 0.25
    ↓
Execution（执行参数）
    ↓
实盘
```

**认知水平**：
- 相信"更先进的技术 = 更好的结果"
- 投入大量时间在模型训练、GPU 配置、特征工程上
- 建立了完整的特征管线（100+ 特征）、标签系统、回测框架

**实际遭遇的问题**：
1. **NN dir 头和 Gate dir 冲突** → 不知道信谁
2. **Router 概率加权 ≈ 平均输钱** → 市场状态不可线性混合
3. **Mamba/Transformer 的 IC 不稳定** → 跨 regime 漂移严重
4. **RL 和"限制自由度换生存"的哲学正面冲突** → 越优化越不稳
5. **实盘 Sharpe 远低于回测** → 不知道是哪个模块出了问题
6. **debug 极其困难** → NN 输出一个数字，无法解释来源

**关键教训**：
> **不是技术不够先进，而是"先进的技术"在非平稳、对抗性的金融市场中反而更脆弱。**

这个阶段留下了大量文档（现在在 `docs/archive/`），但也留下了最宝贵的东西——**特征管线、标签系统、回测框架**。这些基础设施在后续阶段被复用。

---

### Stage 3：退回规则 + ML 配合（开始理解市场）

**代表**：`rule_based_strategies/sr_reversal_rule_strategy.py`

**思路**：NN 不靠谱 → 退回规则，但保留 ML 的特征管线和标签系统。

```
特征管线（从 Stage 2 继承）
  ↓
树模型训练 → 导出规则 → 人工审核语义
  ↓
规则系统（可解释、可审计）
  ↓
回测验证
```

**认知升级（关键转折）**：
1. **树模型 = 望远镜，不是方向盘** → 用树发现规律，不用树直接决策
2. **规则 > 概率** → Gate 的 hard veto 比 Router 的概率加权更稳
3. **可解释性 > 精度** → 宁可粗糙可解释稳定，不要精细不稳漂移
4. **Sharpe 下降时能精确定位** → 逐条规则检查贡献

**`sr_reversal_rule_strategy.py` 是过渡产物**：
- ✅ 使用了特征管线
- ✅ 使用了标签系统
- ✅ 使用了 RR 标签计算
- ❌ 仍然没有"假设"的概念
- ❌ 没有 plateau 验证
- ❌ 没有逐条规则 IC 监控
- ❌ 没有降级方案

---

### Stage 4：研究框架成型 —— BPC + 实验目录（当前阶段）

**代表**：`docs/z实验_001_bpc/`、`config/strategies/bpc/`、全套研究脚本

**思路**：研究过程本身就是对数据、市场、系统的理解。每个发现都要被验证、被记录、被监控。

```
假设冻结（7 条 BPC 假设）
  ↓
树模型发现规则 → 人工审核语义 → 写入 YAML
  ↓
Plateau 验证（阈值鲁棒性）→ snotio KPI（多维质量）
  ↓
配置驱动（gate.yaml / evidence.yaml / execution.yaml / entry_filters.yaml）
  ↓
回测验证（逐bar模拟、per-symbol、grid search）
  ↓
实盘部署（BPCLiveStrategy → OrderFlowListener → run_live.py）
  ↓
监控（逐条规则 IC / 否决率 / 通过率）
```

**认知终态**：
> **不是在"找一个赚钱的策略"，而是在"建立一个允许 alpha 死亡的研究框架"。**

## 二、为什么最终形成"代码 + 实验"的目录结构

### 2.1 核心洞察

```
代码 = 基础设施（不变的部分）
实验 = 验证结果（变化的部分）
```

在 Stage 0-2 的痛苦经历后，产生了一个深刻认识：

> **代码应该稳定，变化的只应该是配置和验证结果。**

| 维度 | 代码（src/） | 实验（z实验_NNN/） |
|------|------------|-----------------|
| **变化频率** | 低（基础设施级） | 高（每次调参/验证） |
| **内容** | 特征管线、回测引擎、研究脚本 | 配置快照、验证结果、决策记录 |
| **版本管理** | git diff 审计 | 文档记录 + 数据快照 |
| **复用性** | 跨实验复用 | 每次实验独立 |

### 2.2 为什么需要"固定实验目录"

Stage 2 最大的痛点：**不知道上次是什么配置、什么结果、为什么改了**。

```
没有实验目录时：
  改了 gate.yaml → 跑回测 → Sharpe 变了
  → 不记得改了什么
  → 不记得上次 Sharpe 是多少
  → 不记得为什么要改
  → 再改一次 → 更乱了
  → 循环 ×100

有实验目录后：
  docs/z实验_001_bpc/实验_001.md         ← 命令速查（可复现）
  docs/z实验_001_bpc/实验_001详细记录.md   ← 完整决策过程
  docs/z实验_001_bpc/entry_filter_design.md ← Entry Filter 正交设计
  docs/z实验_001_bpc/因果性思考.md        ← 特征因果性审计
  ↓
  每次变更都有：原因 → 操作 → 结果 → 结论
  ↓
  知识不丢失，可回溯，可审计
```

### 2.3 命名规范：z实验_NNN_主题

```
docs/z实验_000_架构/    ← 架构级文档（跨实验的设计原理）
docs/z实验_001_bpc/     ← BPC 策略的所有实验记录
z实验_002_xxx/     ← 未来新策略/新 archetype 的实验
```

**为什么以 `z` 开头**：在文件排序中自然排到最后，不干扰代码目录。
**为什么 `_000`/`_001`**：数字编号，按创建顺序排列。
**为什么带主题**：一看就知道这个实验是关于什么的。

## 四、实验工作流 —— 每次升级怎么做

### 4.1 标准实验流程

```
Step 1: 提出假设
  "BB Width < 某阈值时，突破后延续概率更高"
    ↓
Step 2: 树模型验证
  训练 Return Tree → 检查 bb_width 是否高频分裂 → 导出候选规则
    ↓
Step 3: Plateau 验证
  python scripts/optimize_entry_filter_plateau.py --filter bb
  → 确认阈值存在"对参数不敏感的稳定区间"
    ↓
Step 4: snotio KPI 验证
  python scripts/optimize_entry_filter_snotio.py --all
  → 多维 KPI（Sharpe × (1-Noise) × (1-Timeout) × IR × (1-Overlap)）
    ↓
Step 5: 回测验证
  python scripts/backtest_execution_layer.py --tiers --noise-penalty --entry-filter
  → 全管线回测（Gate → Entry → Evidence → Tier → NP → 逐bar模拟）
    ↓
Step 6: 写入配置
  更新 entry_filters.yaml（enabled: true, threshold: plateau 推荐值）
    ↓
Step 7: 记录实验
  更新 docs/z实验_001_bpc/实验_001.md（命令）
  更新 docs/z实验_001_bpc/实验_001详细记录.md（完整记录）
    ↓
Step 8: 实盘部署
  代码已在 BPCLiveStrategy 中自动读取 YAML → 零代码上线
```

### 4.2 为什么每次升级都要增加实验记录

```
没有实验记录：
  "上次改了什么来着？" → 不知道
  "为什么改的？" → 忘了
  "改完效果好吗？" → 没记录
  "能回滚吗？" → 不确定回滚到哪里

有实验记录：
  实验_001详细记录.md 第 X 节：
    日期: 2026-02-05
    操作: 启用 liquidity_silence filter
    原因: 正交维度发现（与 bb_compression 不重叠）
    结果: Sharpe 0.28 → 0.31, snotio 0.032 → 0.035
    结论: 保留，设为 OR 组合
    回滚: 在 entry_filters.yaml 设 enabled: false 即可
```

### 4.3 "相同代码基础 + 不同数据验证"的价值

```
同一套代码（backtest_execution_layer.py）：
  实验 A: BTCUSDT 2024-01~2024-06 → Sharpe 0.35
  实验 B: BTCUSDT 2024-07~2024-12 → Sharpe 0.28
  实验 C: ETHUSDT 2024-01~2024-12 → Sharpe 0.22
  实验 D: 修改 evidence 后 → Sharpe 0.38

所有实验用同一套代码：
  → 差异 100% 来自数据/配置/市场
  → 可以精确归因"是规则问题还是市场问题"
  → 不存在"我改了代码还是改了参数"的困惑
```

## 五、从 Stage 0 到 Stage 4 的认知升级总结

| 阶段 | 信念 | 实际经历 | 最终认知 |
|------|------|---------|---------|
| **Stage 0** | "找到好参数就能赚钱" | 参数不稳定，Sharpe 波动大 | 参数是表面，背后需要假设 |
| **Stage 1** | "自动搜索能找到最优" | 18 维 Optuna ≈ 随机搜索 | >10 维不可优化，需要降维 |
| **Stage 2** | "更先进的模型 = 更好的结果" | NN 漂移、Router 平均输钱、无法 debug | 复杂度不是能力，是噪声 |
| **Stage 3** | "规则 + ML 配合更好" | 初步可解释，但缺少假设概念 | 规则需要假设支撑 |
| **Stage 4** | "研究过程 = 对市场的理解" | 每条规则可追溯、可降级、可监控 | **框架不变，假设迭代** |

### 每个阶段留下的遗产

| 阶段 | 被抛弃的 | 被保留的 |
|------|---------|---------|
| **Stage 0** | Backtrader、人工调参 | "结构特征有因果性"的直觉 |
| **Stage 1** | Optuna 作为主优化工具 | "参数搜索"的思想 → 演化为 Plateau |
| **Stage 2** | NN 多头、Router、RL | 特征管线、标签系统、回测框架 |
| **Stage 3** | 树模型直接做策略 | 树模型做"假设发现器" |
| **Stage 4** | 无 | 全部保留 → 当前系统 |

## 六、为什么实验目录指导实盘执行

### 6.1 实验 → 配置 → 实盘的闭环

```
研究环境:
  docs/z实验_001_bpc/ 的实验记录
    ↓ 验证通过
  config/strategies/bpc/ 的 YAML 配置
    ↓ 自动加载
  src/.../live/bpc_live_strategy.py
    ↓ 零代码部署
  python src/.../live/run_live.py --mode bpc --symbol BTCUSDT
```

**实验记录是实盘的"法律依据"**：
- 每个 YAML 参数都有实验来源
- 每个阈值都有 plateau 验证
- 每个规则都有因果性审计

### 6.2 实盘出问题时的回溯链

```
实盘 Sharpe 下降 → 触发诊断
  ↓
Step 1: 检查 docs/z实验_001_bpc/实验_001详细记录.md
  → 找到当前配置的验证记录
  ↓
Step 2: 检查每条规则的 IC / 否决率
  → 定位到具体失效的规则
  ↓
Step 3: 在 YAML 中禁用该规则
  → 系统降级但不死
  ↓
Step 4: 记录新实验
  → 更新实验文档，等待下次 plateau 验证
```

## 七、总结

### 一句话版本

> **这个项目不是"设计出一个好架构"，而是"在不断犯错中长出了一个好架构"。**

### 目录结构的哲学

```
docs/z实验_000_架构/  = 为什么这样做（设计原理，跨实验）
docs/z实验_001_bpc/   = 怎么做的（实验记录，可复现）
src/             = 做了什么（代码实现，稳定）
config/          = 用什么配置（YAML 驱动，可审计）
```

### 实操原则

1. **代码尽量不变** — 变化集中在 YAML 配置
2. **每次变更必须记录** — 原因 → 操作 → 结果 → 结论
3. **相同代码 + 不同数据** — 差异 100% 可归因
4. **实验指导实盘** — YAML 参数都有实验来源
5. **可回溯、可回滚** — 每个配置变更都可一键撤销

### 演进的终极教训

> **研究过程不是"开发策略"的手段，研究过程本身就是对数据、市场、系统的理解。**
> **当你的代码库稳定到只需要改配置，你的实验记录详细到可以复现任何一个决策——**
> **你就不再是"调参工程师"，而是"假设审计员"。**
