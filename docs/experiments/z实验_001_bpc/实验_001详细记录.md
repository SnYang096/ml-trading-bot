# 树模型导出规则到archetype

| 层            | 本质问题 | 输出        | 优化目标            |
| ------------ | ---- | --------- | --------------- |
| Gate         | 能不能做 | veto / 降权 | Lift            |
| Evidence     | 下多少  | 连续强度      | bad suppression |
| Entry Filter | 现在吗  | 等 / 不等    | snotio          |

正确的因果链是这样的（请记住这条）
Gate 优化 Lift
   ↓
改变的是：哪些环境被允许进入系统
   ↓
Entry Filter 优化 snotio
   ↓
改变的是：单笔 trade 的性价比
   ↓
Evidence 优化 bad suppression
   ↓
减少“本来就要失败”的交易权重
   ↓
Execution 控制 MaxDD / Tail risk


👉 MaxDD 只能在链条末端被控制，不能在前端被直接“学出来”。
Gate 决定“这个世界值不值得参与”，
Entry 决定“这一次值不值得出手”，
Execution 决定“就算错了我会不会死”。

##  训练 Return Tree，kpi比较复杂，主要有语义是否可用

> 📖 **详细评估框架**: 参见 [Return Tree KPI 量化评估文档](./return_tree_kpi_framework.md)

如果前面gate已经过滤了一些差得机会，我们可以训练 Return Tree
目的：让 Return Tree 在"已被 Gate 过滤后的样本"上学习，看它能否进一步区分好坏。
如果 Return Tree 的 特征重要性 主要来自：
  vol_regime_*（波动regime）
  terminal_risk_score（追末端）
  volume_participation_score（execution时机）
  exhaustion_*（节奏错位）
→ ✅ 证明剩余失败确实来自 Evidence/Execution 层面
如果特征重要性主要来自：
  bpc_dir_*（方向）
  trend_r2_*（趋势结构）
  jump_risk_*（极端风险）
→ ❌ 说明 Gate 还不够，需要加强

判断标准：
Return Tree 的 Top 10 重要特征中：
如果 ≥7个 是 Evidence 特征（vol/terminal/exhaustion/execution）
→ ✅ Gate 已经足够
如果 ≥5个 是 Gate 特征（dir/structure/trend/jump）
→ ❌ Gate 需要加强
训练完后，查看 feature_importance 和 tree_rules.md 就能验证您的假设！

## 2.4 Gate 平坦高原优化 (lift 目标)

### 原理说明
**为什么用 Lift？**
- Gate 的目标是过滤坏机会，不是单纯提高通过率
- Lift 衡量 "Gate 是否让好样本比例提升"

**Good/Bad 定义** (基于 rr_extreme 标签):
- Good: `bpc_impulse_return_atr >= -0.8` (不踩大坑)
- Bad: `bpc_impulse_return_atr < -0.8` (踩大坑)

```bash
# 统一 Gate 优化（Hard + Soft 一次性优化）
python scripts/optimize_gate_unified.py \
  --strategy bpc \
  --logs results/train_final_<timestamp>_rr_extreme/bpc/logs_gated.parquet \
  --output results/train_final_<timestamp>_rr_extreme/bpc/gate_unified_optimization_v2.json

# 实际使用示例 (2026-02-08):
python scripts/optimize_gate_unified.py \
  --strategy bpc \
  --logs results/train_final_20260208_220616_return_tree/bpc/logs_gated.parquet \
  --output results/train_final_20260208_220616_return_tree/bpc/gate_unified_optimization_v2.json
```

**输出结构**：
- `gate_unified_optimization_v2.json`: 每条规则的 Lift、PassRate、Robustness、阈值
- 优化后手动审核，将结果更新到 `gate.yaml`

**Gate 四层分权架构**：
```
System Safety     → 市场级保护（反身性拥挤），不参与优化
    ↓
Hard Gates (3)    → 必要条件：方向过拥挤、WPT点火过强、WPT未力竭
    ↓
Guardrails (2)    → 策略语义约束：价格位置/缩量程度（固定阈值，不参与优化）
    ↓
Soft Filters (9)  → 增强条件：降权但不否决
```


## 2.5 Evidence 平坦高原优化 (bad_suppression 目标)

### 原理说明
**为什么用 bad_suppression？**
- Evidence 是软调整，不做硬 veto，而是影响置信度/仓位
- **bad_suppression** (主 KPI) = P(score<0.3|bad) - P(score<0.3|good)，衡量 bad 被选择性压制的程度
- **good_amplification** (辅 KPI) = P(score>0.7|good) - P(score>0.7|bad)
- 双重约束：bad_suppression > 0.05 AND good_amplification > 0.05

**bins 参数说明**：
- `bins: [0.25, 0.45, 0.65, 0.85]` 是分位数切分点，把特征值分布切成 5 档
- 每档对应语义标签：suppress(0.0) / downweight(0.25) / neutral(0.5) / favor(0.75) / amplify(1.0)
- bins 既用于优化时计算 KPI，也用于生产时实盘评分（不是中间参数）
- 优化脚本会遍历所有候选 bins 组合，找到 bad_suppression 最大且 plateau 稳定的那组

**Gate-conditioned 分位数阈值模式**：
- 用 good 样本（gate 放行）的分布计算分位数阈值
- 用这些阈值对 **所有样本**（包括 bad）赋分
- bad 样本如果特征值偏低，就会被映射到 suppress 区间

```bash
# 优化 Evidence 分位数划分
python scripts/optimize_evidence_plateau.py \
  --strategy bpc \
  --logs results/train_final_<timestamp>_return_tree/bpc/logs_gated.parquet \
  --output results/train_final_<timestamp>_return_tree/bpc/evidence_optimization.json \
  --min-sharpness 0.05

# 实际使用示例 (2026-02-08):
python scripts/optimize_evidence_plateau.py \
  --strategy bpc \
  --logs results/train_final_20260208_220616_return_tree/bpc/logs_gated.parquet \
  --output results/train_final_20260208_220616_return_tree/bpc/evidence_optimization.json \
  --min-sharpness 0.05
```

**输出**：
- `evidence_optimization.json`: 每条特征的 bins、bad_suppression、sharpness、plateau 判断
- `evidence_optimization.html`: 可视化报告
- 优化后将通过的特征 bins 更新到 `evidence.yaml`

**Plateau 稳定性判断** (三重条件)：
1. CV < 0.5（邻域 bad_suppression 变异系数低）
2. bins 连续（相邻 bins 步长 ≤ 0.15）
3. 语义邻居 ≥ 3 且 semantic_drift < 0.10


### 前后对比

| Filter | 旧 Sharpe | 新 Sharpe | 旧 Trades | 新 Trades | 变化 |
|--------|----------|----------|----------|----------|------|
| deep_pullback_vol | 0.474 | 0.443 | 224 | 285 | trades +27%, Sharpe -7% |
| deep_pullback_wick | 0.414 | **0.422** | 150 | **343** | **trades +129%, Sharpe +2%** ⭐ |
| deep_pullback_bb | 0.395 | 0.392 | 356 | 288 | 接近不变 |
| deep_pullback_momentum | 0.413 | 0.360 | 177 | 380 | trades +115%, Sharpe -13% |
| deep_pullback_cvd | 0.332 | **0.347** | 527 | **650** | trades +23%, Sharpe +5% |
| deep_pullback | 0.326 | 0.329 | 962 | 1202 | trades +25%, Sharpe 持平 |
| deep_pullback_liq_void | 0.455 | 0.322 | 193 | 408 | trades +111%, Sharpe 下降 |

### 结论

1. **偏宽容阈值的效果符合预期**: trades 普遍增加 23%~129%，Sharpe 基本持平或略降
2. **deep_pullback_wick 最突出**: trades 从 150 翻到 343，Sharpe 反而上升，证明原来 wick_absorption > 0.3 太严格，错过了许多好机会
3. **deep_pullback_cvd 也提升了**: Sharpe +5% 且 trades +23%，说明 pullback_depth 0.6→0.55 放宽是合理的
4. **deep_pullback_liq_void 下降明显**: pullback_depth 0.6→0.45 可能太激进，这个 filter 的 plateau 实际不够宽
5. **核心结论**: `pullback_depth ≥ 0.6` 对多数 filter 偏严格，下调至 0.5~0.55 是合理的

### 注意点

1. **阈值更新后必须同步更新 description**: `entry_filters.yaml` 中每个 filter 的 `description` 字段要和新阈值一致
2. **Trades CV 警告不能忽视**: `deep_pullback_wick.bpc_pullback_depth` 有 ⚠️ Trades CV 警告，意味着 Sharpe 稳定但 trades 数在高原内波动大，实盘节奏可能不稳
3. **Plateau 扫描是语义保持的**: 每次只动一个阈值，其他条件保持不变，不是网格搜索
4. **20% 偏宽容侧原理**: `start + 0.2 * width`（>=条件）/ `end - 0.2 * width`（<=条件）——因为 Entry Filter 的错误成本是「错过」而非「多等一次确认」
5. **置信度等级**: width ≥ 0.3 = HIGH, ≥ 0.15 = MEDIUM, < 0.15 = LOW
6. **deep_pullback_liq_void 可能需要回滚**: 从 0.6 降到 0.45 后 Sharpe 从 0.455 降到 0.322，考虑回滚到 0.55
7. **不要在 Evidence/Execution 层用 Plateau 扫描**: Plateau 扫描专门为 Entry Filter 设计（语义完成态特征），其他层有各自优化脚本

### BPC 决策管线详细流程

```
features (from IncrementalFeatureComputer)
    │
    ├─ 1. bpc_breakout_direction → 方向 (+1=LONG, -1=SHORT, 0=不入场)
    │
    ├─ 2. Gate 检查
    │     archetype.apply_gate(features)
    │     → hard deny: 拒绝入场
    │     → soft filter: 降权 gate_weight (0.0~1.0)
    │
    ├─ 3. Entry Filter (OR 组合)
    │     DerivedEntryFeatureState().update(features)
    │     check_entry_filters_or_single()
    │     当前启用: bb (波动率压缩) OR liq_silence (流动性枯竭)
    │
    ├─ 4. Evidence Score
    │     archetype.compute_evidence_score(features)
    │     → composite_score × gate_weight
    │
    ├─ 5. Tier 选择 → 执行参数
    │     select_tier(score, tiers_cfg, exec_config)
    │     → initial_r / activation_r / trail_r / size_multiplier / time_stop_bars
    │
    └─ 6. 输出 TradeIntent
          action=LONG/SHORT, execution_profile={SL/TP/trailing/timeout}
```

# 3.3 设计决策：NN 多头与 Router 的替代

### 为什么去掉 NN 多头路径模型

在 Gate → Evidence → Tier → Noise Penalty 已成型的系统里，NN 四个头已被更稳定的模块完全覆盖：

| NN 头 | 原职责 | 替代者 | 替代方式 |
|-------|--------|--------|----------|
| dir | 方向预测 | **Gate** | `bpc_breakout_direction` 结构性特征 |
| mfe | 空间预测→仓位 | **Evidence + Tier** | `evidence_score → tier → size_multiplier` |
| mae | 路径风险→止损 | **Noise Penalty + SL** | 数学特征压缩 + execution.yaml 宪法 |
| mtt | 时间预测→持仓 | **Execution Budget** | 规则 `time_stop_bars`（FAST/NORMAL/SLOW） |

核心判断：NN 在 execution 层是"非法公民"——宁可粗糙/可解释/稳定，也不要精细/不稳/漂移。
详见: `nn多头路径原语模型不要了.md`

### 为什么去掉 Router 模型

Router 输出概率只是在"分摊错误"——当市场"语义不唯一"时，加权平均 ≈ 平均输钱。

替代方案：**Gate 优先级 + Archetype 互斥规则（先验规则 Router）**

```
规则 Router 逻辑:
  趋势族（ME + BPC）可共存 → 优先 ME
  趋势 vs 反转 不可共存 → 不交易
  单 archetype 通过 Gate → 直接交易
  多 archetype 冲突 → 降仓 or 不交易
```

Gate 是 hard veto（反证法），不是 soft score。在噪声高、非平稳的市场里，强 veto 比加权平均更稳。
详见: `没有router原因.md`

### 什么时候才需要重新引入？

| 条件 | Router | NN mtt |
|------|--------|--------|
| 同族 archetype 内部 RR 分布显著分化 | ✅ 需要 | - |
| trades > 100k 且 Evidence 无法解释 realized path | - | ✅ 需要 |
| 跨市场复用，优先级不再稳定 | ✅ 需要 | - |

在这些条件**同时成立**之前，当前纯规则架构是最优选择。

### 多 Archetype 扩展方式

当前 BPC 是单 archetype。未来如果有其他纯规则 archetype（如 failure_reversal、liquidity_sweep），通过 `decision_handler` 接口扩展：

```python
# Composite Handler: 多个纯规则 strategy + 先验优先级仲裁
class CompositeDecisionHandler:
    def __init__(self, strategies: List, priority_order: List[str]):
        self.strategies = strategies
        self.priority = priority_order  # e.g. ["ME", "BPC", "FBF", "LSR"]
    
    def decide(self, *, features, symbol, bars=None):
        all_intents = []
        for s in self.strategies:
            all_intents.extend(s.decide(features=features, symbol=symbol, bars=bars))
        return self._resolve_conflicts(all_intents)  # 按优先级 + 互斥规则仲裁
```
