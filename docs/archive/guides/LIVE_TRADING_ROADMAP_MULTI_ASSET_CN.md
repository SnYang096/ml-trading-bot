## 多资产合约实盘落地路线图（CN）

你的当前状态非常典型：**特征很多（1000+）、模型很多（树/NN/RL）、技术调研很多（Mamba/WPT/Hilbert/订单流）**，但“怎么把系统跑到实盘并稳定迭代”反而最难。本文的目标是：把“技术栈”转换成一个**低维护、可验收、可逐步加仓**的实盘路线。

> 重要现实：任何“高置信度 1w→10w”都无法被工程保证。我们能做的是：用**强风险控制 + 严格 OOS 验收 + 组合分散 + 逐步加仓**，把“爆仓/回撤失控/线上漂移”的概率压到足够低，并把改进方向变得明确。

---

## 0) 你真正要的是什么？（把目标可操作化）

你想要的是：
- **可扩展到多资产**（10+ coins，4H/更高频也可）
- **维护成本低**（不想每次加一个新模式就重写一堆策略）
- **稳健可控**（不是单一币/单一 regime 的一次性 lucky）
- **可把资金从 1w 做到 10w 的证据链**

把“高置信度”变成可执行的工程指标：
- **回撤/爆仓风险约束**：最大回撤（DD）、最大连续亏损、仓位上限、资金曲线稳定性
- **组合层稳健性**：TopN 组合 Sharpe、相关性、换手、成本敏感性
- **OOS 稳定性**：不同窗口/不同币/不同成本假设下的表现不崩
- **线上漂移监控**：数据缺失、特征分布漂移、模型输出漂移、交易分布漂移

### 0.1 “证据链”是什么意思？（举例）

这里的“证据链”不是口号，而是一组**可量化的门槛 + 可回滚的行动规则**：当系统持续达标，你才允许它“承担更大的风险预算/更高的仓位上限”，从而有机会把资金从 1w 做到 10w；一旦不达标，必须降级/停机。

一个可执行的例子（仅示例，你可以更保守）：
- **阶段 A（1w）**：仅开最小仓位（例如单笔风险 ≤ 0.25% 账户），目标是验证“系统跑得稳”
  - 连续 4 周满足：Top10 OOS `rule_sharpe_mean > 0`，且 `DD_mean` 不超过阈值
  - 成本敏感性：把 cost/slippage 提高一档仍不崩（Sharpe 仍为正或接近 0）
  - 单币一致性：不允许收益主要由单一币贡献（组合不靠“一个币撑起来”）
- **阶段 B（2w~5w）**：允许提高风险预算（例如单笔风险 ≤ 0.5%），但必须加更严格的停机/冷却规则
- **阶段 C（5w~10w）**：只有当多窗口 walk-forward 仍稳定为正，才允许进一步提升上限；否则保持或回退

> 总结：证据链 = “达标→允许扩大风险预算；不达标→立刻回滚”，用它避免在不可控阶段过早放大仓位导致归零。

### 0.2 “漂移”是指失效吗？为什么用漂移这个词？

不是。**漂移（drift）是“输入/输出/行为分布发生变化”的现象**，它是失效的常见前兆，但不等于“已经失效”。

为什么不用“失效”而用“漂移”：
- “失效”是结果（PnL/Sharpe 明显变差），往往发生得比较晚
- “漂移”是原因层的早期信号（数据、特征、模型输出、交易行为），更早、更可控

几个实盘中常见的漂移例子（对应你文档里那 4 类）：
- **数据缺失漂移**：某些 symbol 某些月的 ticks/orderflow 缺失 → 相关特征大量 NaN
- **特征分布漂移**：同一特征的均值/方差/分位数显著变化（例如波动结构变了）
- **模型输出漂移**：`pred_mfe_atr/pred_mae_atr/dir_prob` 的分布整体偏移或极值频率激增
- **交易分布漂移**：mode 分布从“有交易”漂到“全 NO_TRADE”或突然高换手（通常意味着阈值/数据/执行假设发生变化）

> 实盘治理思路：把“漂移”当作报警与降级触发器（降仓/NO_TRADE/fallback），把“失效”留给事后复盘。

## 1) 先选“一个主干架构”，其它技术全部降级为“辅助工具”

你现在最容易“卡住”的原因不是缺技术，而是**主干架构不唯一**，导致每个新想法都能把你拉去重构。

### 推荐的主干（短期实盘最优、维护最低）

**NN multihead（路径原语） + Rule Router（3-action） + Execution 模板（RR/成本/延迟）**

原因：
- **多资产可复用**：路径原语本质是“市场形态结果的统计预测”（dir/mfe/mae/ttm），天然跨策略/跨币共享。
- **维护成本低**：Router action 固定（NO/MEAN/TREND），Execution 复用模板；新增“模式”优先加 detector/gating/阈值，而不是新增模型头/新增策略目录。
- **可迭代**：你已经证明 Router 阈值调参能把 Top9 OOS 的 Sharpe 从负变正——这更准确地说明的是 **“系统可控（controllable）”**，而不等于 **“可泛化/可预测（generalizable）”**。
  - **可控指什么**：存在少量、可解释的旋钮（阈值）能显著改变 **mode 分布/交易频率/风险暴露**，并能通过报告定位问题出在 Router 阈值还是执行假设。
  - **你担心的点成立**：如果自由度足够大，“总能调到好参数”。所以“调到正 Sharpe”本身不能当作长期有效证据。
  - **为什么这仍然是强信号**：在固定数据、固定执行假设、固定 action space 的前提下，用少量阈值就能把“塌缩/亏损”拉回到“有交易且改善”，说明系统不是完全不可控/不可调试的黑箱。
  - **必须补上的泛化验证**：walk-forward 多窗口、成本/滑点/延迟敏感性、多币一致性、以及限制阈值自由度（防止过拟合）。只有这些通过，才说明“可泛化”。

### 其它技术在这个主干下怎么定位（避免走偏）

- **树模型策略**：不再作为“上线主系统”的核心，而作为：
  - **特征筛选/灵感源**（找 PoolB / 发现有效语义特征）
  - **detector 的快速原型**（wick/absorption/failed breakout 等）
    - **这里的“原型”是什么意思**：不是让 Tree 去“直接开仓”，而是把它当作一个**快速的模式识别器训练工具**——训练一个“事件是否发生/有多强”的评分器（`detector_score`），再把这个 score 下沉到 Router/Gating 里做过滤/禁用/降仓。
    - **为什么用 Tree 来做原型**：
      - **迭代快**：训练成本低，改 label/特征/阈值后能很快得到结果。
      - **可解释**：可以从特征重要性/分裂条件里提炼出稳定的规则（比直接上 NN 更容易 debug）。
      - **对稀疏事件友好**：wick 反转/失败突破/吸收这类事件往往样本稀疏，Tree 更容易先跑出 baseline detector。
    - **怎么操作（最小步骤）**：
      1. **定义事件 label**（允许看未来窗口，但特征只能用 t 时刻可得数据，避免未来泄露）
      2. **训练一个轻量 Tree detector**：输入用现成特征（例如 `wick_upper_ratio/wick_lower_ratio/volume_anomaly/cvd_* / vpin_*`），输出为 `P(event|features)`
      3. **把 Tree 输出当作 detector_score**（或从树里抽几条核心规则固化为规则 detector）
      4. **接入 Router/Gating**：例如 `if detector_score > th: 禁止 TREND / 只允许 MEAN / 直接 NO_TRADE / 降仓`
      5. **用 OOS + 成本假设复验**：确认它减少尾部亏损/减少错误参与，而不是过拟合。
    - **一个具体例子（failed breakout detector）**：
      - **label（未来 H 根）**：`failed_breakout=1` 当“发生破位后在 H 根内回到区间且出现显著不利 excursion（例如 `mae_atr` 高）”，否则 0。
      - **特征（t 时刻可得）**：`wick_*`、`volume_anomaly`、`vpin_*`、`cvd_*`、`range_ratio_*` 等。
      - **上线用法（不直接交易）**：`if failed_breakout_score > th: cooldown N bars 或 NO_TRADE`（先防守，再谈进攻）。
  - **对照组**（检验 NN+Router 是否真的更稳健）
- **RL/BC**：放到后面。只有当 Rule Router 在 OOS 已经稳定正 Sharpe，才值得做 BC/RL（否则 RL 只会学到坏分布）。
- **Mamba / WPT / Hilbert**：一律当作“可选 feature blocks”，满足三个条件才纳入：
  - 有严格 no-lookahead 测试
  - 多资产归一化语义清晰（单位/量纲不漂移）
  - 在 OOS/成本/延迟下可带来可复验增益
- **订单流**：优先用于 **gating/风险控制/过滤**（例如极端不利流动性时 NO_TRADE），而不是直接喂给模型让其学“尺度捷径”。

## 2) 落地路线（从今天开始，最快逼近实盘）

下面路线的核心原则是：**先做能上线的最小闭环**，并把每一步都变成可验收的 gate。你已经有大部分基础设施（FeatureStore、报告、E2E），所以关键是收敛目标与迭代节奏。

### Phase 1（1–2 周）：固定一个可跑通的“上线 MVP 闭环”

目标：让系统具备“可每天自动跑 + 输出报告 + 有交易但风险可控”的能力。

做法（只允许动很少的旋钮）：
- **固定数据与评估**：
  - Train：例如 `2023-01-01 ~ 2025-04-30`
  - OOS：例如 `2025-05-01 ~ 2025-10-31`
  - symbols：Top9/Top10
- **固定 returns_source 与成本假设**：
  - 先用 `rr_execution`（你已经在用）
  - 先固定一套“保守成本/滑点/延迟”，后面只在这个假设上优化
- **固定 Router action space**：3-action
- **只调两类东西**：
  1) Router thresholds（你已经成功）
  2) Execution 模板参数（RR/持仓上限/成本/延迟），以 Sharpe+DD 为目标

Phase 1 的验收门槛（示例，你可以更保守）：
- Top9 组合：**OOS rule_sharpe_mean > 0** 且 DD 不超过阈值
- 单币：不允许“靠某 1 个币撑起来”
- 模式分布：不是全 NO_TRADE，也不是超高换手

产物必须齐全：
- `preds_oos/`、`mode.parquet`、`logs.parquet`、`e2e/report.html + metrics.json`
- 一份复盘文档（例如 `docs/experiments/EXP_006...md` 这种格式）

### Phase 2（2–6 周）：从“能盈利”转到“能加仓”

目标：把策略从“可能赚钱”变成“在多窗口/多成本假设下仍然不崩”。

最划算的提升方向（按优先级）：
1) **阈值稳健性**：把阈值调参变成：
   - 多窗口（walk-forward） + 多成本假设 的联合目标
   - 限制自由度（只调少数阈值），避免过拟合
2) **gating / risk filters**：用最少的规则显著降低尾部风险：
   - 极端波动/流动性恶化时强制 NO_TRADE
   - 极端 drawdown/cooldown
   - “模型输出不可信”时 fallback 到更保守模式
3) **组合仓位分配**（维护成本低但收益巨大）：
   - 等权 vs 波动率/ATR 风险平价
   - 限制单币最大暴露
4) **特征收敛**：从 1000 特征收敛到：
   - “可解释、归一化语义正确、稳定”的一小套底座
   - 其它都作为 optional blocks，且必须通过 OOS gate 才保留

Phase 2 的验收门槛建议：
- Top10 组合：多个 OOS 窗口下 Sharpe 为正（不是一次性）
- 成本敏感性：成本翻倍仍不崩
- 线上监控：缺失/漂移能自动报警，并且能 fallback

### Phase 3（6–12 周）：再考虑 BC/RL（在正确的地基上）

只有当 Phase 2 达标，才做 BC/RL：
- 目标不是“RL 神奇赚钱”，而是：
  - 更平滑的 mode 切换
  - 更少人工阈值
  - 更好的 regime conditioning
- 仍然必须保留 hard gating（风险控制永远不交给网络）

## 3) 如何从 1000 特征里走出来（低维护策略）

你不需要“更多特征”，你需要“更好的特征治理”：

### 3.1 让特征成为“可管理资产”

每个特征进入主干必须满足：
- 有 no-lookahead 测试
- 有 normalization contract（raw/unitless/price_unit/usd 等语义清楚）
- 有多资产一致性检查

### 3.2 把特征分成三层（降低维护成本）

- **Layer Base（长期底座）**：少量（几十）稳定特征 + 必要尺度列（atr 但可 exclude_columns，不喂给模型）
- **Layer Blocks（可插拔）**：订单流/序列模型/高级变换，必须可单独开关、可缺失
- **Layer Detectors（Router 用）**：结构识别/风险过滤优先，不直接当作模型输入的“语义动作”

你想的那些模式（插针反转、阻力上移突破、失败突破回归）：
优先以 **detector + gating + execution 模板选择**落地，而不是立刻发明新 action 或新增一堆策略目录。

## 4) 关于“阈值调参是不是炼丹、对未来没意义？”

它确实可能过拟合，但这不是否定它的理由。正确做法是把它工程化：
- **时间隔离**：tune window 与 holdout window 彻底分离
- **多假设**：成本/滑点/延迟多场景同时过关
- **限制自由度**：阈值数量少、范围合理
- **稳定性优先**：先让 mode 分布与交易频率稳定，再追求 Sharpe 极致

当你按上述方式做时，“调参”就不再是炼丹，而是**可复盘的工程优化**。

## 5) 推荐你下一步立即做的 5 件事（按收益/维护比排序）

1) 固定一套“保守成本/滑点/延迟”口径，所有优化都在这套口径下进行（避免自欺）
2) 把 Router tuning 做成 walk-forward（至少 2–3 个窗口），并输出同一份报告格式
3) 做 Top10 组合的仓位分配（等权 vs 风险平价）与单币诊断报告（找异常币）
4) 加 2–3 条硬 gating（尾部风险过滤），优先用订单流/波动/流动性信号
5) 收敛特征底座：只保留能在多窗口 OOS 上稳定增益的 blocks，其它默认关闭

## 6) 你可以如何衡量“离 1w→10w 更近了”？

不要用一次回测的 Sharpe 判断。用“可加仓证据链”判断：
- 同一套系统在多个 OOS 窗口、多个成本假设下为正
- 单币不过度集中（组合分散有效）
- 回撤、尾部风险可控
- 线上运行稳定（数据/特征/预测/交易都有监控与 fallback）

当这些成立时，你才有资格谈“加仓曲线”（从 1w 到 10w 的路径）。

## 7) 对你贴的“外部建议”的评估：对了一半，但需要改成可落地的版本

你贴的观点大意是：
> 0 规则实盘主力；1 树模型做增强；2 NN 多头 & 3 横截面只留模拟/小仓位。

### 7.1 哪些地方是对的（值得采纳）

- **这是“实盘生存策略”问题，不是单纯技术选型**：对。你现在最稀缺的是“可控 + 可复盘 + 可逐步加仓”，不是表达能力。
- **不要把复杂系统直接一把梭上实盘**：对。RL/复杂 Router/多层黑箱在小资金阶段极易把你拖进 debug 地狱。
- **树模型更适合做过滤/仓位缩放/确认器，而不是直接 Trade/NoTrade**：对。这能显著降低尾部风险和维护成本。

### 7.2 哪些地方是“容易误导你”的（需要纠偏）

- **“规则 + 极简执行”作为唯一主力**：这在“先活下来”的角度可以成立，但它忽略了你已经在本 repo 里跑通的更优结构：
  - 你现在的体系已经证明：仅调整 `mlbot rule mode-3action` 阈值，就能把 Top9 OOS `rule_sharpe_mean` 从负值调到正值（并且全链路 e2e 对齐）。
  - 这意味着：你不需要退回“纯规则”，而是应该采用 **Rule Router（固定 action space）+ 可控执行假设** 的版本，仍然保持低维护和可解释。
- **“NN 多头现在别上实盘”**：如果指“让 NN 直接决定交易动作/策略选择”，这条建议基本正确；但如果指“NN 完全不能参与实盘”，那就不对。
  - 你当前的 nnmultihead（dir/mfe/mae/ttm）更像 **Execution Head / 风险收益形态预测**，它可以被限制在“只参与执行与风控阈值”，并且 Router action 仍然由规则/阈值控制。
  - 这类“NN 参与但不掌权”的用法，通常比“纯规则”更接近你要的低维护复利路径。
- **“横截面不适合小资金”**：结论在“高杠杆快速增值”语境下大体成立（会稀释 edge、受成本和约束影响大），但它不应该被永久否定：
  - 更合理的定位是：横截面属于 **后期的风控/组合优化层**（如风险平价、暴露约束、相关性管理），而不是你早期的“主要 alpha 引擎”。

### 7.3 把外部建议改成“适配你当前仓库的正确落地版”

结合你当前已经跑通的 pipeline，推荐你采用这个版本（从低维护到可扩展）：

1) **Rule Router（3-action）作为主控**（固定 action space，强风控，易复盘）
   - 你的主旋钮是：阈值 + gating + execution 假设（成本/延迟/持仓）。
2) **Tree 作为增强器**（可选，逐步加）
   - 用法：过滤低质量交易 / 仓位缩放 / 风险开关（不要直接决定方向/开仓）。
3) **NN 多头作为“执行与风险形态”模块**（可用，但不要让它变成“策略选择器”）
   - 用法：把 `pred_mfe_atr/pred_mae_atr/pred_t_to_mfe` 用于 RR/时间止损/仓位上限/交易频率约束；并让 Router 仍可随时 NO_TRADE。
4) **横截面（Cross-section）与 RL/BC**：放到后期
   - 当 Top10 组合在多个 OOS 窗口、多个成本假设下都稳定为正，再考虑引入。

> 如果你只能记一句：**“主控用规则（可控），增强用 Tree（连续评分），执行用 NN（风险收益形态），后期再上 RL/横截面。”**

## 8) Regime Heads vs Path Primitives：哪个“更高级”？要不要都训练？头越多越强吗？

### 8.1 结论（先给）

- **没有谁更高级**：它们解决的是不同层级的问题。
- **不需要一开始都训练**：在你当前阶段（要快速落地、低维护），优先保留 **Path Primitives**，把 Regime 先用 **detector/gating** 做出来。
- **头越多不一定越强**：头越多 = 标签越多/噪声越多/调参空间越大/过拟合风险更高/线上维护更复杂。

### 8.2 两类 head 的职责边界（非常关键）

#### A) Path Primitives（路径原语 heads）

典型输出：`dir_prob / mfe_atr / mae_atr / t_to_mfe`

- **回答的问题**：如果参与交易，这一段行情的“风险-收益形态”更像什么？（执行层怎么做）
- **优点**：
  - 跨策略复用（同一套原语可服务多种执行模板）
  - 与仓位/止损/止盈/时间止损天然匹配
  - 训练目标更接近可度量的连续量（相对更稳定）
- **缺点**：
  - 不能直接告诉你“这是 compression 还是 SR reversal”，需要上游 Router/规则来做结构选择

#### B) Regime Heads（状态/阶段/风险类型 heads）

典型输出：`compression→expansion / trend persistence / fake-break risk / mean-revert pressure`

- **回答的问题**：现在是否值得参与？适合哪种结构性策略？是否应该 NO_TRADE？（上游决策层怎么做）
- **优点**：
  - 更贴近你原来的策略直觉（compression、SR、trend 等）
  - 如果标签定义得好，可以直接提升 Router 的可解释性
- **缺点（决定了它“更贵”）**：
  - 标签构造更难（很多 regime 的定义是“弱监督/主观阈值”）
  - 更容易漂移（不同市场/不同年份/不同币的 regime 口径不稳定）
  - 更容易把你拉回“规则爆炸/标签爆炸”的维护地狱

### 8.3 你要不要都训练？

**建议路径（低维护、最快逼近实盘）：**

1) **先只训练 Path Primitives**（你现在已经在做，并且已证明能通过 Router 阈值把 OOS Sharpe 拉正）
2) Regime 先用 **detector/gating** 表达（规则或小树）：
   - compression 检测、fake-break 风险过滤、极端波动 NO_TRADE、cooldown 等
3) 只有当你遇到下面“硬信号”时，再加 Regime Heads：
   - 同一套阈值在不同窗口/不同币上不稳定
   - 需要更系统地做 regime conditioning（而不是堆更多规则）
   - 你能给 regime 提供可复盘的标签定义与严格 OOS 验收

> 换句话说：**Regime Heads 是“下一阶段升级”，不是你现在必须的前置条件。**

### 8.4 “头越多越强大”吗？

不一定。增加 heads 会带来：
- **训练成本**：多任务损失权重、样本有效性、label 缺失处理更复杂
- **工程成本**：线上监控维度更多、漂移排查更难
- **过拟合风险**：每个 head 都是一个新的“可被拟合的噪声通道”

更好的原则是：
- **少而稳定**（优先让现有 4 个 primitives heads + Router 阈值在多窗口/多成本假设下稳定）
- 需要新增 head 时，先写清楚：
  - label 如何定义（可复现）
  - 预期提升的哪项指标（Sharpe/DD/换手/稳定性）
  - 失败了如何回滚（不影响主系统）

## 9) 长期有效性：Regime-only vs Primitives-only vs Hybrid（哪个更适合“长期工作”？）

你担心的点是对的：**树模型策略往往等价于“某个 regime 下的一组策略函数”**，如果 regime 漂移或样本结构变了，它就会逐步失效。那是否意味着必须上 Regime Heads？不一定。

### 9.1 先把“长期有效”的含义说清楚

在交易系统里，“长期有效”通常不是指“永远赚钱”，而是：
- **失效是可检测的**（分布漂移、mode 分布漂移、交易质量下降能报警）
- **失效是可控的**（能降仓/停机/回退到更保守策略）
- **修复成本低**（不需要大规模重训/重写策略）

### 9.2 三种方案的对比（高层裁决）

| 方案 | 核心思路 | 长期有效性（可控性/可修复） | 维护成本 | 典型失败模式 |
|---|---|---|---|---|
| **1) Regime-only**（主要靠 Regime Heads） | 先判断市场状态/阶段，再决定策略/执行 | **中等**：解释性强，但 label 漂移风险高 | **高**：标签定义、漂移、重训成本高 | regime 标签口径在不同年份/不同币上不一致；上线后“看似识别了 regime，但交易并不挣钱” |
| **2) Primitives-only**（只用 path primitives） | 只预测执行形态（dir/mfe/mae/ttm），用阈值/规则做 Router | **高（在你当前阶段）**：目标更连续、更易稳定；失效可通过 OOS gate/成本敏感性检测 | **低~中**：少量 head + 阈值 + gating | 缺少“结构选择”，导致在不适合交易的环境仍然参与；需要更多 gating 约束 |
| **3) Hybrid（推荐的长期形态）** | Primitives 做执行形态；Regime（先 detector/gating，后可升级为 head）做参与/策略选择 | **最高**：上游负责“是否参与/过滤”，下游负责“怎么做” | **中**：先规则/小树，后再考虑 head | 如果不限制自由度，容易变成“头爆炸/规则爆炸”；需要严格 gate 与回滚机制 |

### 9.3 我的推荐：分阶段走向 Hybrid，但不要一开始就“训练很多 Regime Heads”

结合你当前 repo 已经跑通的事实（Top9 OOS Router 阈值调参能把 Sharpe 拉正），推荐路线是：

1) **短期（现在）**：以 **Primitives-only 为主**  
   - 把“是否参与”的问题先用 **detector/gating + Router 阈值稳健性**解决  
   - 先把多窗口/多成本假设的稳定性做出来（这才是“长期工作”的基础）
2) **中期（当你确实需要更系统的 regime conditioning）**：走向 **Hybrid**  
   - 先把 Regime 作为 **可插拔 detector/gating**（规则或小树）  
   - 只有当 detector 证明稳定、且你能给出可复盘 label 口径时，再把它升级为 Regime Head
3) **长期（规模化/策略增多）**：少量 Regime Heads + 固定 action space + 强风控  
   - heads 数量受控（宁可 2–4 个稳健 head，也不要 10+ 个脆弱 head）

> 关键点：**长期有效来自“可控的失效与回滚机制”，而不是来自“更像策略直觉的标签”。**

## 10) 你贴的“path 更赚钱、regime 更稳态；先加一个零重训刹车”的说法对吗？

总体结论：**方向基本正确，但需要两点“工程化约束”才能避免误用。**

### 10.1 哪些点是对的（可以直接采纳）

- **Path heads 与 Regime heads 不互相替代**：对。它们分别对应“执行形态”和“参与/环境选择”。
- **短期（快速验证/快速增值）更依赖 path heads；中长期更依赖 regime/gating 来抗退化**：对。
- **先做一个零重训、极轻量的“是否值得参与”的刹车踏板（regime proxy gate）**：对，而且非常符合你“低维护”目标。

### 10.2 需要纠偏的关键点（避免踩坑）

#### (A) “用现有 head 组合 tradeability_proxy”——可以做，但要明确它是什么

你贴的这种形式：

```
tradeability_proxy = sigmoid(a*mfe - b*mae + c*dir_prob - d*ttm)
```

工程上可行，但必须明确：
- 它更像 **execution quality / risk-reward attractiveness**（“参与以后大概长什么样”）
- 它不是严格意义的 **market tradeability/regime**（“现在是否值得参与”）

> 换句话说：它可以当作“仓位缩放/过滤”，但不应该被当作唯一 gate，
> 否则很容易把你带回“阈值过拟合/自我确认”的陷阱。

#### (B) “path heads 会害你”——这句话要翻译成可检测的工程信号

这点的直觉是对的：在某些环境下你会出现“模型看起来很美，但实际频繁止损”的情况。
但落地上应该写成**可监控、可报警**的信号，例如：
- **高置信度却频繁触发止损**（例如 `pred_dir_prob` 高、`pred_mfe_atr/ pred_mae_atr` 看起来好，但 realized pnl 连续为负）
- **分环境差异**（在趋势窗口好、震荡窗口 bleed）
- **同一套阈值跨窗口崩**（walk-forward 不稳定）

这些信号出现时，不是立刻“重训很多 regime heads”，而是优先：
- 加强 gating（减少不该参与的段）
- 做阈值稳健性（多窗口/多成本假设）

### 10.3 我推荐的“零重训定型方案”（真正可落地）

你现在最划算的版本是：

1) **保持 path primitives heads 不变**（继续作为执行形态与风险收益模块）
2) 在 Router 前加一个 **极轻量的 regime proxy gate（规则/小树/固定指标）**
   - 例：chop 高 + vol 低 → NO_TRADE
   - 例：极端波动/流动性恶化 → NO_TRADE
   - 例：连续亏损/回撤触发 cooldown → NO_TRADE
3) 只有当 gate 经过多窗口 OOS 证明稳定增益，才考虑把某个 proxy 升级为 **单个 Regime Head**

一句话总结：
- **path heads 决定你能跑多快（进攻）**
- **regime/gate 决定你会不会翻车（防守）**
- **你现在需要的是给油门装 ABS：先加刹车，不换发动机。**

## 11) 如果要加 Regime Heads，我会怎么加？（最小增量、可回滚）

先给结论：**我会“只加 1 个 head”作为起步**，并且这个 head 必须满足：
- **label 可复现**（不是主观口径）
- **label 是“预测未来”的**（否则只是特征，不该训练 head）
- **能直接用于 Router/gating**（否则上线收益不明确）
- **随时可回滚**（关掉 head 不影响主系统）

### 11.1 先纠正一个常见误区：哪些“regime”其实不该训练 head？

如果你的 regime 只是由历史窗口可确定的统计量定义（例如过去 20 根的 realized vol 分位数、chop index、过去波动状态），那它本质是 **feature / detector**：
- ✅ 直接做成特征列 + gating 规则（零训练、零漂移口径）
- ❌ 不要训练成 head（训练只会引入噪声与不稳定）

只有当你想预测“未来 H 根会处于什么状态”（例如未来是否会趋势延续/是否会假突破/未来是否 tradeable）时，才值得加 head。

### 11.2 我推荐的第一个 Regime Head：`tradeable_regime`（二分类）

**为什么先做它？**
- 它最贴近你现在的业务闭环：**是否值得参与**（NO_TRADE gate）
- label 可以直接从你已有 primitives 定义出来（复用同一个未来窗口），实现成本最低
- 它能显著降低 “path heads 把错误做得更精致” 的风险

#### Label 定义（基于未来窗口，预测意义明确）

以与你现有 `path_primitives_labels` 同一套 horizon 为准：
- 先计算 `mfe_atr(t), mae_atr(t), t_to_mfe(t)`（你已有）
- 定义未来窗口的 tradeability：
  - `tradeable = 1` 当：
    - `mfe_atr >= mfe_min_for_trade` 且
    - `mfe_atr/(mae_atr+eps) >= eff_min_for_trade` 且
    - （可选）`t_to_mfe <= ttm_max_for_trade`
  - 否则为 0

直觉：这就是“未来窗口里确实有足够非对称的机会”，而不是仅仅“看起来可能”。

#### 如何接入到 Router（最小侵入）

- Router 先看 `pred_tradeable_regime`（或其概率）：
  - 低于阈值 → 强制 `NO_TRADE`
  - 高于阈值 → 再走现有的 `mode-3action` 阈值逻辑（MEAN/TREND）

这样你不会改变 action space，也不会破坏已有 tuned 阈值的主体逻辑，只是加了一个“参与许可”。

如果 `tradeable_regime` 在多窗口 OOS 证明稳定增益，再考虑加第二个（仍然保持少量）：

- `trend_persistence`（二分类或回归）：未来窗口里是否呈“慢趋势/延续”（比如 `t_to_mfe` 较大且净变化显著）
- 或 `fake_break_risk`（二分类）：未来窗口里先突破后回撤并触发较大 MAE（需要非常谨慎定义）

原则：**每次只加一个 head，并要求它的收益/稳健性提升可以在报告中被单独归因**。

### 11.5 你提出的延伸：既然能加 tradeable head，那能不能再加 mean/trend head，“一个 MLP 搞定”？

你的理解**方向是对的**：从“减少反复调阈值”的目标出发，确实可以把 Router 进一步学习化：
- `tradeable`：是否允许参与（NO_TRADE gate）
- `mode`：参与后更像 MEAN 还是 TREND（Router 分流）

但要把这件事做成“更省维护”而不是“换一种调参”，必须满足两个前提：

1) **label 必须清晰且可复现**  
   - `tradeable` 可以用“未来窗口是否存在足够非对称机会”来定义（相对自然）
   - `mean vs trend` 的 label 更难：它往往是“策略语义/执行模板”的产物，口径很容易漂移  
   现实里常用两条路线：
   - **Distillation / Imitation**：用你已经 tuned 的 rule-router（或一套稳定规则）产生 `mode` 作为教师标签，让 MLP 学会近似它（省人工阈值，但本质是“学习规则”）
   - **Counterfactual-based**：用 `ret_mean/ret_trend`（你已有 rr_execution 产物）定义“哪种 mode 更赚钱”，作为 label（更贴 PnL，但对成本/延迟/执行假设非常敏感）

2) **必须证明“更稳”**（而不是“只是在某个窗口更好看”）  
   你担心的点仍然成立：如果自由度变大，总能在某个窗口拟合出好结果。  
   所以 mean/trend head 的验收必须更严格：walk-forward、多成本假设、多币一致性、以及 mode 分布稳定性。

关于“一个 MLP 搞定”：
- 可以：在同一个 trunk 上加多个 heads（tradeable + mode_logits + primitives），工程上可行。
- 但不必一口吃成胖子：多任务会引入 loss 权重、样本不平衡、label 缺失等复杂度；**头越多不一定更省维护**。

**推荐的落地顺序（最低风险）**：
1) 先用规则/阈值把系统跑稳（你已经做到）
2) 如确有维护压力，再加 `tradeable` head（只做 NO_TRADE gate）
3) 只有当 tradeable head 稳定增益后，再考虑 `mode` head（mean/trend），优先用 distillation 方式降低口径漂移风险

## 12) “NN 多头方案一定优于 Tree 直接开仓吗？”（对，但别说绝对）

你贴的结论方向基本正确：**在多资产 + 合约 + regime 频繁变化 + 需要低维护可控的场景里，“分层系统（NN 多头输出 + Router/Gating + Execution）”通常比“Tree 直接开仓”更稳、更可持续。**

但“几乎一定/必胜/一定优于”太绝对，建议改成更安全的工程表述：

### 12.1 什么时候它成立（必须同时满足的前提）

- **你有可控的 Router/Gating**（例如固定 action space、可回滚阈值、风险停机）
- **Execution 假设明确且可审计**（成本/滑点/延迟/持仓上限，并有敏感性分析）
- **OOS 稳定性已验证**（多窗口/多币，不是单次幸运）
- **线上有监控与 fallback**（漂移/缺失/输出异常能触发降仓/NO_TRADE）

在这些前提成立时，分层系统的优势来自“系统可控性”，而不是“NN 更强”：
- 错误可分解（是参与错、执行错、阈值错，还是数据错）
- Execution 可进化（调整 SL/TP/仓位/时间止损不必重训主模型）
- 更容易跨币复用（减少策略目录爆炸）

### 12.2 Tree 直接开仓的典型问题（在你的场景更致命）

- **把多目标揉成一个目标**（方向/收益/风险/时间尺度混在一起，线上很难解释失败原因）
- **regime shift 更脆**（在 4H 多资产里更容易“突然不工作”）
- **回滚与治理成本高**（一旦失效，你往往只能重训/重搜特征/重写规则）

### 12.3 什么时候 Tree 直接开仓仍然可能更好（例外）

存在一些例外象限：
- 单一市场、regime 相对稳定
- 执行极简单（低交易频率/低成本敏感）
- 你只追求一个非常窄的目标（例如单一方向分类），并且有强外部风险约束

> 但就你目前的目标与约束（多资产合约 + 快速迭代 + 低维护 + 可加仓证据链），Tree 直接开仓通常不是最优主控形态。

### 12.4 推荐原则（写进系统设计的“硬规则”）

> **任何模型（Tree/NN/线性/规则）都不允许直接决定“是否开仓”。**  
> 必须先经过 **Router/Gating（是否参与/风险许可）**，再进入 **Execution（仓位/SL/TP/时间止损）**。
