# 多 Archetype 系统中的优先级与资源分配框架

## 一、核心问题

> ME 胜率低但 R 高，是否应降低优先级，让 BPC 更优先？

答案不是简单的 Yes / No。

必须区分两个层面：

```
系统层面优先级 ≠ 单次机会优先级
```

---

# 一、统计结构差异（角色认知）

在典型结构下：

## BPC（结构延续）

* 胜率：高
* R：中等
* 波动：小
* 稳定性：强
* 分布：对称或轻微右偏

**作用：**

* 提供资金曲线“骨架”
* 稳定现金流来源
* 解释性强

---

## ME（动能扩张）

* 胜率：低
* R：高
* 波动：大
* 分布：强右尾
* 稳定性：中等

**作用：**

* 提供收益右尾
* 捕捉爆发行情
* 提高分布非对称性

---

## 结构性结论

* 只做 BPC → 曲线平稳但缺爆发
* 只做 ME → 曲线锯齿状但尾部极强

因此：

> BPC = 稳定因子
> ME = 偏态增强器

---

# 二、系统优先级 vs 机会优先级

## 1️⃣ 系统层面（长期结构）

在多 archetype 系统中：

```
默认优先级：BPC > ME
```

原因：

* BPC 负责稳定
* ME 负责增强
* 系统需要稳定的基准收益源

如果 ME 占系统主导：

* 连续亏损压力大
* 心理干扰强
* 曲线波动放大

---

## 2️⃣ 单次机会层面

在强动能环境下：

```
ME > BPC
```

当满足：

* 波动扩张
* 流动性活跃
* 动能一致性强

此时 ME 可以覆盖 BPC。

---

# 三、成熟系统的正确结构

不是“选谁更高”。

而是定义：

| Archetype | 角色     |
| --------- | ------ |
| BPC       | 稳定现金流  |
| ME        | 右尾增强   |
| FER       | 失败反转捕捉 |
| LV        | 极端事件响应 |

这叫 **角色分工系统**。

# 四、Slot 与资源分配设计

## 推荐结构：三层控制

```
Layer 1：基础资源分配
Layer 2：优先级动态调整
Layer 3：极端信号覆盖
```

---

## 第一层：基础资金结构（静态）

例如：

```
BPC = 45%
ME  = 25%
FER = 20%
LV  = 10%
```

意义：

* BPC 为骨架
* ME 为增强
* FER 为对冲
* LV 为尾部

这是长期结构稳定性。

---

## 第二层：优先级动态（推荐简化做法）

在工程实践中：

> 建议只做优先级动态，不做 budget 动态。

原因：

* 优先级变化 = 决策结构变化
* budget 变化 = 风险结构变化
* 双动态会导致非线性叠加，难以调试

推荐做法：

* 固定每 archetype 最大风险
* 只在 PCM 中调整谁先抢 slot

---

### 示例优先级切换

**常态市场**

```
BPC > ME > FER > LV
```

**高波动扩张期**

```
ME > BPC > FER > LV
```

**高杠杆风险期**

```
LV > FER > ME > BPC
```

但资金比例不变。

---

## 第三层：Override（极端信号覆盖）

当极端事件出现时允许覆盖：

### ME 覆盖 BPC

条件：

```
me_evidence 强
ATR percentile 高
flow strength 高
```

### LV 覆盖所有

条件：

```
liquidation cluster
OI 异常波动
```

LV 属于非线性事件，应有最高抢占权。

---

# 五、是否需要 Regime Detector？

在多 archetype 系统中：

> 不需要独立的 Regime 分类器。

原因：

* 每个 archetype 本身就是一种隐式 regime
* 让 archetype 竞争，本身就是 adaptive 机制
* 显式 regime 容易滞后和过拟合

---

## 真正重要的不是 Regime

而是：

```
是否有绩效反馈机制？
```

例如：

* Rolling Sharpe
* Rolling Expectancy
* 最近胜率 × R
* 回撤斜率

如果 PCM 根据绩效微调优先级：

系统已经在做 regime adaptation。

---

# 六、简化工程建议

当前阶段建议：

✅ 固定 Budget
✅ 动态优先级
❌ 不做双动态
❌ 不引入复杂 Regime 分类器

发展顺序：

1. 先稳定结构
2. 建立 30 日 rolling performance 反馈
3. 再考虑风险权重微调

---

# 七、最终方法论总结

在多 archetype 系统里：

```
先控制“谁上场”
再考虑“给他多少钱”
```

不是：

```
谁 Sharpe 高谁优先
```

而是：

```
谁负责稳定？
谁负责爆发？
谁负责保护？
```

---

# 八、最终优先级哲学

```
系统默认：BPC > ME
强动能机会：ME > BPC
极端事件：LV > ALL
失败信号：FER 覆盖 ME
```
