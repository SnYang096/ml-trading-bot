## ⚡ 使用模式：`--all` vs `--strategy`

> **核心原则**: 日常研究用 `--all`，单策略调试用 `--strategy`

| 模式          | 命令                                     | 用途                                              | PCM 联合回测          |
| ------------- | ---------------------------------------- | ------------------------------------------------- | --------------------- |
| **正式研究**  | `mlbot pipeline run --all`               | 全策略串行训练 → PCM 联合回测 → ADOPT 含 PCM 决策 | ✅ 自动运行 (≥2 策略)  |
| **调试/单看** | `mlbot pipeline run --strategy fer-long` | 单策略训练 → 单策略回测 → 只看交易地图            | ❌ 不运行 (无仲裁意义) |

**为什么 PCM 不需要像 Execution 那样有 Tiers?**

- **Execution Tiers** = 同策略内根据 evidence score 选不同止损/止盈 → 策略内部精细化，需要搜索最优参数
- **PCM** = priority + position_scale → 语义决定的 regime 级全局配置，不是 per-trade 优化
- PCM 是 **validate + snapshot**，不是 optimize — 参数在 `constitution.yaml` 和 `pcm_regime.yaml` 中固定

**多策略 ADOPT 决策矩阵** (仅 `--all` 模式生效):

```
单策略 ADOPT  ∧  PCM PASS   →  最终 ADOPT
单策略 ADOPT  ∧  PCM ALERT  →  ADOPT + WARNING
单策略 ADOPT  ∧  PCM ERROR  →  KEEP (需人工检查)
```

## 重要说明：数据源与常见陷阱

### 陷阱 1: predictions.parquet vs logs_gated.parquet

> **关键提醒**: Execution 优化和回测务必使用 `logs_gated.parquet`，不要用 `predictions.parquet`。
>
> 两个文件**行数相同**（如 FER 都是 3005 行），区别在于：
>
> | 文件 | 产出步骤 | gate 标记列 | 脚本行为 |
> |------|---------|-----------|----------|
> | `predictions.parquet` | Step 6 Evidence apply | **无** | 全部信号参与 → 虚高胜率(96%) |
> | `logs_gated.parquet` | Step 5 Gate apply | **有** `gate_decision` | 自动过滤 → 只有 allow 信号参与 |
>
> **为什么重要**:
> 1. `optimize_execution_grid.py` 会自动检测 `gate_decision` 列并过滤 veto 信号
> 2. `predictions.parquet` 没有该列 → 脚本无法过滤 → 全部信号参与 → 虚高胜率(96.1%)
> 3. 使用错误数据源会产生不合理的参数（如 initial_r=6.0）
> 4. 正确做法：用 `logs_gated.parquet`，回测结果(如54.2%胜率)才与实盘一致

### 陷阱 2: Gate 通过率低导致样本不足

> Gate 过滤后可用信号量可能很少（如 FER: 120/3005=4%），导致 execution 优化只有个位数交易，
> 结果不可靠（如 7 trades + 100% win rate + Sharpe 21）。
>
> **应对**:
> - `auto_research_pipeline` 已内置小样本保护：`gate allows < min_trades` 时自动跳过 Step 8，保留现有 execution.yaml
> - 手动运行时需自行判断交易数是否足够（建议 ≥30 trades）

### 陷阱 3: 目录名 ≠ 内容范围

> `train_final_*_return_tree/` 中的 `return_tree` 仅标识 Step 2 prepare 使用的 label 类型，
> 但 **Step 3–10 全流程产出都写入同一目录**，包括 gate、evidence、execution 等阶段的文件。
> 不要因目录名误以为里面只有 return_tree 相关内容。

### 特征文件生命周期: features.yaml → features_gate_shap.yaml → gate_draft.yaml → gate.yaml

> **核心问题**: `features_gate_shap.yaml` 从哪来？它和 `gate_draft.yaml`、`gate.yaml` 是什么关系？
>
> 下图展示了 gate 侧特征文件的**完整生命周期** (evidence 侧同理):

```
┌──────────────────────────────────────────────────────────────────────────┐
│                     特征文件完整生命周期                                   │
│                                                                          │
│  ① features.yaml               唯一 truth source 全量特征候选池 (永远不动) │
│      (config/strategies/{s}/)   4个archetype共享79节点超集, SHAP自动裁剪 │
│                │                                                          │
│                ▼                                                          │
│  ② Step 2: Prepare              用全量特征构建训练数据                    │
│      features_labeled.parquet   (全量特征列 + gate_label 标签列)           │
│                │                                                          │
│                ▼                                                          │
│  ③ Step 2.5: SHAP Walk-Forward  ★ features_gate_shap.yaml 在这里产出 ★  │
│      输入: features_labeled.parquet + features.yaml                       │
│      算法: 4-fold 时间切分 → 每 fold 训 LightGBM + SHAP                  │
│            → 聚合稳定性 → 只保留 stable_nodes + protected_nodes (atr_f)  │
│      输出: features_gate_shap.yaml (裁剪后的特征子集)                      │
│                │                                                          │
│                ▼                                                          │
│  ④ Step 5: Gate Train            用裁剪后的特征训练 gate 模型             │
│      输入: features_gate_shap.yaml (如果 SHAP 开启, 否则用原始)           │
│      产出: gate 模型 + gate_draft.yaml (从模型树分裂自动提取的规则)        │
│                │                                                          │
│                ▼                                                          │
│  ⑤ Step 5b: Gate Optimize        对 gate_draft 规则做阈值优化            │
│      输入: gate_draft.yaml + logs_gated.parquet                           │
│      产出: archetypes/gate.yaml (最终部署的 gate 规则)                    │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

**各文件职责对比:**

| 文件                      | 位置                     | 产出步骤              | 内容                               | 是否手动编辑 |
| ------------------------- | ------------------------ | --------------------- | ---------------------------------- | ------------ |
| `features.yaml`           | `config/strategies/{s}/` | 手工维护              | 全量特征节点候选池 (79 nodes 超集) | ✅ 可编辑     |
| `features_gate_shap.yaml` | `config/strategies/{s}/` | Step 2.5 SHAP         | SHAP 裁剪后的稳定特征子集          | ❌ 自动生成   |
| `gate_draft.yaml`         | 实验目录下               | Step 5 Gate Train     | 模型树分裂提取的规则 (特征名+阈值) | ❌ 自动生成   |
| `gate.yaml`               | `archetypes/`            | Step 5b Gate Optimize | 最终优化后的 gate 规则 (部署用)    | ❌ 自动生成   |

 **关键理解**:
> - `features_gate_shap.yaml` 和 `gate_draft.yaml` 是**上下游关系**，不是同一层级，不应合并
> - SHAP 裁剪发生在**训练之前** (选哪些特征给模型学)，gate_draft 发生在**训练之后** (模型学到了什么规则)
> - 原始 `features.yaml` 永远不动，SHAP 写入 `_shap.yaml` 后缀文件
> - `--skip-shap` 时管线回退到 `features.yaml` (全量特征)

**流程合理性 — 每层职责清晰且不重叠:**

| 层                      | 回答的问题                       | 决策方式               |
| ----------------------- | -------------------------------- | ---------------------- |
| SHAP (Step 2.5)         | 哪些特征**跨时间稳定重要**？     | 统计 (4-fold 稳定性)   |
| Gate Train (Step 5)     | 用稳定特征**学到什么分裂规则**？ | 模型 (LightGBM 树分裂) |
| Gate Optimize (Step 5b) | 规则的**最优阈值**是多少？       | 搜索 (grid/bayesian)   |

> **设计要点**:
> - SHAP 限制了模型的"特征预算"，模型只能在稳定特征上分裂 → 不同 seed 的分裂点更一致 → gate 规则更稳定
> - 三层分工: 统计筛选 → 模型学习 → 阈值优化，每层只做一件事
> - 配置: `config/research_pipeline.yaml` 的 `shap_feature_selection:` 段

## 四、DEPLOY (config → live)

> 脚本: `scripts/deploy_config_to_live.py`

### 4.0 Deploy 触发条件

**不是每次研究变了就 deploy。** 核心逻辑：“旧的不行了，新的更好” 才 deploy。

规则配置在 `config/research_pipeline.yaml` 的 `deploy_gate:` 中：

**触发条件 (OR, 至少一个满足才值得 deploy):**

| 规则           | 条件                           | 含义                        |
| -------------- | ------------------------------ | --------------------------- |
| sharpe_improve | 提升 >= 5%                     | 新版本显著更好              |
| drift_level    | >= HIGH 且 \|Sharpe变化\| > 3% | 参数大幅漂移 + 性能确实有变 |

> **稳定性保护**: 即使漂移达 HIGH，若 Sharpe 变化 <= 3%，则不触发——防止数据窗口滑动导致的自然漂移误触发。

**安全门禁 (AND, 全部满足才允许 deploy):**

| 规则          | 条件         | 含义             |
| ------------- | ------------ | ---------------- |
| require_adopt | 决策 = ADOPT | 新版不比旧版差   |
| min_trades    | >= 50        | 统计显著性保障   |
| human_confirm | 默认 true    | 部署前需人工确认 |

**决策矩阵:**

| 漂移\\Sharpe变化 | 提升>=5% | \|变化\|>3% | \|变化\|<=3%              |
| ---------------- | -------- | ----------- | ------------------------- |
| HIGH             | ✅ DEPLOY | ✅ DEPLOY    | ❌ SKIP (漂移大但性能没变) |
| < HIGH           | ✅ DEPLOY | ❌ SKIP      | ❌ SKIP                    |

pipeline 跑完会自动显示：
- `🚀 Deploy: ✅ 值得且允许 deploy` + deploy 命令
- `⏭️  Deploy: SKIP` — Sharpe 提升不足 且 漂移较小/性能稳定, 不需要动实盘
- `🚫 Deploy: ❌` — 有触发但安全门禁未通过 + 具体原因

结果保存在 `report.json` 的 `deploy_gate` 字段中。

### 9.2 回测体系设计说明

> **分层架构与数据粒度**:
> - **训练 (Steps 1-8)**: 使用 4H 特征，不涉及 SL 执行精度，正常
> - **Step 9 回测执行层**: 使用 `logs_gated.parquet` + `--use-1min`，评估 **模型本身信号质量**
> - **PCM 联合回测**: 使用 `--from-raw --use-1min`，验证 **实盘可复现性**
> - **事件回测**: 天然用 1min tick 级 SL/TP，最接近实盘
>
> **Step 9 vs PCM 的设计定位**:
> - Step 9 (`--logs`) 回答：「这些模型信号用执行规则交易后赚不赚钱」→ 评估模型质量
> - PCM (`--from-raw`) 回答：「实盘 IFC 实时计算的特征能否复现研究结果」→ 验证实盘可行性
> - 两者回答不同问题，不需要精确一致

#### 对齐目标：同源 100%，跨源容忍偏差

| 对齐层级     | 对比对                                             | 预期偏差  | 说明                                 |
| ------------ | -------------------------------------------------- | --------- | ------------------------------------ |
| **同源对齐** | `--logs` 向量 ↔ `event_backtest --from-logs-gated` | **0%**    | 同一份信号 + 同一套执行规则          |
| **同源对齐** | `--from-raw` 向量 ↔ `event_backtest --from-raw`    | **<1%**   | 同一份 IFC 特征                      |
| **跨源对比** | `--logs` (研究特征) ↔ `--from-raw` (IFC 特征)      | **5-10%** | 浮点差异导致 gate 边界信号翻转，正常 |

> **结论**: 精确对齐 = 同源 100%。`logs_gated` 与 `--from-raw` 是不同信号源，天然有差，不追求精确一致。

**说明:**

- 事件回测模拟实时逐 1min bar 处理，更接近实盘表现
- 两者交易数应在 1% 以内对齐（如 405 vs 407）
- 传统 logs_gated 模式用于快速迭代，但信号源与事件回测不同，不适合做一致性对比

**回测一致性对齐进展 (2026-03-01)**:

| 层              | 状态     | 说明                                                                            |
| --------------- | -------- | ------------------------------------------------------------------------------- |
| 特征计算        | ✅ 已对齐 | 36/36 特征 >=99.9% 匹配                                                         |
| gate 评估       | ✅ 已对齐 | 向量化 gate 支持 all_of/any_of + AND 语义，`--from-raw --recompute-gate` 可重算 |
| entry_filter    | ✅ 已对齐 | 事件侧确认包含 entry_filter                                                     |
| pre-slot 信号数 | ✅ 已对齐 | 1895 (向量) ≈ 1879 (事件)                                                       |
| slot 行为       | ✅ 已对齐 | 向量用 entry_ts_ns 精度 + 多 timeframe span 修复，405 vs 407 = 0.5%             |

**已修复 Bug 清单 (回测一致性)**:
1. gate 向量化不支持 `all_of`/`any_of` 谓词 → FER prefilter 被跳过
2. 多 feature when 条件用 OR 而非 AND → gate 通过率虚高
3. `per_strategy_limits` 耦合于 `add_position_cfg` → slot 分配不正确
4. 事件回测漏斗 `lf.get("gate", True)` 默认值错误 → reject 分类不准
5. PCM 跳过 timeframe 不匹配策略时 `_last_funnel` 保留陈旧数据 → 漏斗统计偏差
6. 向量回测 slot 判定用 bar 级 exit_idx (4H 粒度) → 改为 1min exit_ts_ns 时间戳
7. slot `_bar_close_ns` 前看 bug: 用 `entry_ts + bar_minutes` 判定释放 → 改为 `entry_ts`（不前看）
8. `_estimate_span_years` 多 timeframe 显示 bug: ME 1h 导致 bar 数膨胀 → 改用 timestamp 直接计算

## 10. Archetype 独立调优流程（以 ME 为例）

> **设计原则**：向量回测为主优化层（速度快），事件回测为二次验证层，两者差异不能过大。
> 每步有明确 KPI 验收门槛，不满足即停，不进入下一步。

### 10.1 KPI 验收门槛总表

| 阶段                | 工具                                  | 验收项             | 门槛    | 说明                                          |
| ------------------- | ------------------------------------- | ------------------ | ------- | --------------------------------------------- |
| 阶段1：向量回测优化 | `backtest_execution_layer.py --sym-r` | mean_r             | ≥ 0.18R | 覆盖实盐排耗：手续皅0.08R+滑点0.05R+延迟0.05R |
| 阶段1：向量回测优化 | `backtest_execution_layer.py --sym-r` | sharpe (per-trade) | ≥ 0.05  | 最低信号强度                                  |
| 阶段1：向量回测优化 | `backtest_execution_layer.py --sym-r` | n_trades           | ≥ 100   | 样本量要求                                    |
| 阶段1：向量回测优化 | `backtest_execution_layer.py --sym-r` | at_boundary        | False   | 参数不卡在搜索上限                            |
| 阶段2：事件回测验证 | `optimize_event_execution.py --sym-r` | mean_r             | ≥ 0.15R | 允许事件回测略低                              |
| 阶段2：事件回测验证 | `optimize_event_execution.py --sym-r` | sharpe (per-trade) | ≥ 0.04  | 轻微宽松                                      |
| 阶段2：事件回测验证 | `optimize_event_execution.py --sym-r` | n_trades           | ≥ 50    | 事件回测样本更少                              |
| 阶段3：一致性对比   | `compare_vector_event_consistency.py` | 交易数偏差         | < 10%   |                                               |
| 阶段3：一致性对比   | `compare_vector_event_consistency.py` | Sharpe 偏差        | < 0.3   | SL 简化后应更一致                             |
| 阶段3：一致性对比   | `compare_vector_event_consistency.py` | 胜率偏差           | < 5pp   |                                               |
| 阶段3：一致性对比   | `compare_vector_event_consistency.py` | mean_r 偏差        | < 0.05R | 新增门槛                                      |
