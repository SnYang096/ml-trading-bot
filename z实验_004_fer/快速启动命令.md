## Pipeline 总览

训练是**两轮迭代**流程。Prefilter/Direction 分析只需要 **features + labels**（不需要模型），
通过 `--prepare-only` 直接导出全周期 `features_labeled.parquet`，无需先跑完整训练。

```
═══ Round 0: 数据准备 (--prepare-only, 不训练模型) ═══

Step 1: Feature Store 构建
  ↓
Step 2: 导出 features_labeled.parquet (--prepare-only, 全周期)
  ↓
═══ Round 1: 校准轮 (用 features_labeled.parquet 分析，更新配置) ═══
  ↓
Step 3: Prefilter 语义分层分析 (验证 archetype 特征区分力)
Step 4: Direction 验证 (方向模块质量检查)
  ↓ (人工审核，更新 prefilter.yaml / direction.yaml)
  ↓
═══ Round 2: 正式训练 (带 prefilter + direction) ═══
  ↓
Step 5: Gate 正式训练 + 优化
  ↓
Step 6: Evidence 训练 + 优化
  ↓
Step 7: Entry Filter 优化
  ↓
Step 8: Execution 优化
  ↓
Step 9: PCM 联合回测
```

> **features_labeled.parquet 内容**: 全周期 (train+test) 的特征列 + 标签 (success_no_rr_extreme) + 元数据 (timestamp, symbol, direction 等)。
> 与 predictions.parquet 的区别: 没有模型预测值 pred，但覆盖全周期（不仅是 holdout）。

### Step 3: Prefilter 语义分层分析

> **前置依赖**: Step 2 产出的 `features_labeled.parquet`（全周期数据）

验证 FER archetype 特征的区分力，确认 pre_filter 条件是否合理。
脚本自动读取 `config/strategies/fer/prefilter.yaml` 的 candidates 列表，
并将结果自动回写到该文件的 `last_evaluation` 段。

```bash
PREPARE_DIR="results/train_final_<timestamp>_rr_extreme/fer"

# 基础分析 (全周期)
python scripts/analyze_archetype_feature_stratification.py \
  --logs ${PREPARE_DIR}/features_labeled.parquet \
  --strategy fer

# Mode A (推荐): 近 6 月选特征 + 全周期 rolling 验证
python scripts/analyze_archetype_feature_stratification.py \
  --logs ${PREPARE_DIR}/features_labeled.parquet \
  --strategy fer --select-recent 6

# Mode B: 指定时间范围 + 时间稳定性
python scripts/analyze_archetype_feature_stratification.py \
  --logs ${PREPARE_DIR}/features_labeled.parquet \
  --strategy fer \
  --start-date 2025-07-01 --end-date 2026-01-01 \
  --temporal

# ── 全特征发现: 扫描 ALL 数值特征 ──
# 当启发式候选想不出时，跳过 prefilter.yaml，用数据发现意外的区分模式
python scripts/analyze_archetype_feature_stratification.py \
  --logs ${PREPARE_DIR}/features_labeled.parquet \
  --strategy fer --all-features --temporal

# 全特征 + 近 6 月选特征 + rolling 验证 (推荐组合)
python scripts/analyze_archetype_feature_stratification.py \
  --logs ${PREPARE_DIR}/features_labeled.parquet \
  --strategy fer --all-features --select-recent 6
```

**Prefilter 参数速查**:

| 参数 | 说明 |
|---|---|
| `--logs` | features_labeled.parquet 路径 (必须) |
| `--strategy` | 策略名: bpc / me / fer / lv (必须) |
| `--config` | prefilter.yaml 路径 (默认: `config/strategies/{strategy}/prefilter.yaml`) |
| `--all-features` | 发现模式: 跳过 prefilter.yaml, 扫描 parquet 全部数值特征 |
| `--select-recent N` | Mode A: 用最近 N 个月做特征选择，全周期做 temporal rolling (隐含 `--temporal`) |
| `--temporal` | 时间稳定性分析: rolling bad_rate 曲线 + CV + 多窗口自动选择 |
| `--percentiles` | 百分位列表 (默认: 5,10,20,80,90,95) |
| `--start-date` / `--end-date` | 过滤数据时间范围 |
| `--output` | 输出 JSON 报告路径 (可选) |

**配置文件链路** (配置模式):
```
config/strategies/fer/prefilter.yaml  (候选 _f 分组名)
  → config/feature_dependencies.yaml  (output_columns 解析实际列名)
  → parquet 中匹配存在的列 → 逐列分位数分层
```

**关注指标**：
- 正信号（high=good）: bad rate 差异 > 5% → 可做 pre_filter 正向条件
- 反信号（high=bad）: bad rate 差异 > 5% → 可做反向 hard_gate
- temporal CV < 0.3 稳定，< 0.5 一般，>= 0.5 不稳

**操作**: 审核报告后，更新 `config/strategies/fer/archetypes/prefilter.yaml`。

### Step 4: Direction 验证 + 特征发现

> **前置依赖**: Step 2 产出的 `features_labeled.parquet`

FER 方向由 `direction.yaml` 确定性规则决定。需验证方向模块不是有害的，并可用全特征扫描发现新的方向候选。

```bash
PREPARE_DIR="results/train_final_<timestamp>_rr_extreme/fer"

# ── 基础验证: 覆盖率 + 方向质量 ──
python z实验_005_统一研究/direction_strict_validation.py \
  --logs ${PREPARE_DIR}/features_labeled.parquet \
  --strategy fer

# ── 候选对比: 读 direction.yaml 的候选，按 Median Lift 排名 ──
python z实验_005_统一研究/direction_strict_validation.py \
  --logs ${PREPARE_DIR}/features_labeled.parquet \
  --strategy fer --compare-features

# ── 候选对比 + 时间稳定性 (推荐) ──
python z实验_005_统一研究/direction_strict_validation.py \
  --logs ${PREPARE_DIR}/features_labeled.parquet \
  --strategy fer --compare-features --temporal

# ── 全特征发现: 扫描 ALL 数值特征 ──
python z实验_005_统一研究/direction_strict_validation.py \
  --logs ${PREPARE_DIR}/features_labeled.parquet \
  --strategy fer --compare-features --all-features --temporal
```

**Direction 参数速查**:

| 参数 | 说明 |
|---|---|
| `--logs` | features_labeled.parquet 路径 |
| `--strategy` | 策略名: bpc / me / fer / lv |
| `--compare-features` | 对比模式: 读 `direction.yaml` 候选，按 Median Lift 排名 |
| `--all-features` | 发现模式: 扫描 parquet 里全部数值特征 (需配合 `--compare-features`) |
| `--temporal` | 时间稳定性: rolling 窗口 Median Lift + CV + 衰减检测 |

**Phase 1 通过标准** (覆盖率)：
- 覆盖率 100% (每个 bar 都有方向)
- Short 占比 > 15%

**Phase 2 通过标准** (方向质量)：
- `median(rr_in_direction) > median(forward_rr_long)` → 方向模块优于 always-long
- `mean(rr_in_direction) > 0` → 按方向交易总体不亏
- Bad rate(按方向) < Bad rate(always-long) → 踩坑率降低
- Short 子集样本量 >= 1080 → 统计可信

**操作**: 如果 Phase 2 不通过，检查 `direction.yaml` 规则或暂时禁用 `direction_aware`。

### Step 5: Gate 正式训练 (方向感知 + Prefilter)

> **前置依赖**: Step 3 Prefilter 分析 + Step 4 Direction 验证通过后，带校准配置重新训练。

**方向感知** (`direction_aware: true`):
- `forward_rr = forward_rr_long × direction` → 再判断 < -0.8
- Short 时 MFE/MAE 互换
- 配置: `labels_rr_extreme.yaml`, `labels_return_tree.yaml`

**Archetype Prefilter** (`--archetype-prefilter`):
- 读取 `archetypes/prefilter.yaml` 中的环境前置条件
- 训练前过滤数据：不满足条件的样本不参与训练
- 模型只学习 "好环境下为什么还会失败"

```bash
# Gate 正式训练 (带 archetype prefilter)
mlbot train final --no-docker \
  --config config/strategies/fer \
  --features config/strategies/fer/features_gate.yaml \
  --labels config/strategies/fer/labels_rr_extreme.yaml \
  --archetype-prefilter config/strategies/fer/archetypes/prefilter.yaml \
  --symbol BTCUSDT,ETHUSDT,BNBUSDT,SOLUSDT,XRPUSDT,ADAUSDT \
  --timeframe 240T --data-path data/parquet_data \
  --start-date 2023-01-01 --end-date 2026-01-01 \
  --holdout-start-date 2024-05-01 --holdout-end-date 2026-01-01 --seed 42

# 记录训练输出目录
GATE_RESULT_DIR="results/train_final_<timestamp>_rr_extreme/fer"

# Gate 应用 archetype 规则
mlbot gate apply-archetype \
  --logs ${GATE_RESULT_DIR}/predictions.parquet --strategy fer

# Gate 优化 (plateau 搜索)
python scripts/optimize_gate_unified.py --strategy fer \
  --logs ${GATE_RESULT_DIR}/logs_gated.parquet \
  --output ${GATE_RESULT_DIR}/gate_optimization.json
```

## FER 特定验证

1. **Impulse 识别验证**：
   - 检查特征是否能有效识别"单边impulse"存在
   - 核心特征：impulse_failure_score, time_since_impulse

2. **推进效率验证**：
   - price_delta_efficiency 是否为Top重要特征
   - 效率下降是否与反转成功相关

3. **资金吸收验证**：
   - aggressor_absorption_ratio 是否有效
   - trapped_longs_ratio 是否有信号

4. **PCM 仲裁优先级验证**：
   - 确认 FER = 0（最高优先级）
   - 测试 FER 与 ME/BPC 冲突时的仲裁结果
