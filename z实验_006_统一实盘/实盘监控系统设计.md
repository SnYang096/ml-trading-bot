> 更新日期：2026-02-22
> 设计原则：本地做统计验证（省钱），服务器做风险控制 + 心理安抚（轻量）

---

```
┌─────────────────────────────────────────────────────────┐
│                    监控系统 (两部分)                       │
├───────────────────────────┬─────────────────────────────┤
│   Part A: 本地研究监控      │   Part B: 实盘监控           │

│   频率: 周频/月频          │   频率: 15min / 实时         │

│         + 最新行情重跑      │         只保留当前+近1个月     │
└───────────────────────────┴─────────────────────────────┘
```

# Part A: 本地研究监控（假设验证）

## 设计思路

每次训练 pipeline 的 Step 1-9 都建立了一组假设。
本地监控的核心是：**用最新数据重跑这些验证，确认假设没有漂移**。

频率建议：
- **周频**: 关键层快速检查（Prefilter / Direction / Gate 通过率）
- **月频**: 全层深度验证报告

数据来源：
- **方式 1（推荐）**: 从服务器下载最近 1 个月的实盘特征快照（见 L0），与本地 batch 重算结果对比
- **方式 2**: 本地用最新行情数据重新跑 `--prepare-only` 生成 features_labeled.parquet

---

## 逐层假设与验证对照表

### L0: 线上/线下特征一致性（最优先检查）

> **为什么需要**: 实盘 incremental 计算 vs 本地 batch 计算可能因 warmup 不足、
> 数据源差异、时间对齐 bug 导致同一时间戳的特征值不同。这是最容易出 bug 的地方。

| 训练假设 | 验证方法 | 失效阈值 |
|---------|---------|----------|
| 线上特征计算 = 线下特征计算 | 逐特征对比同一时间戳的值 | 偏差 > 1% 的特征数 > 5 |
| 特征完整性一致 | 对比线上/线下的 NaN 位置 | NaN 不一致的比例 > 3% |

**服务器端**：实盘系统每根 bar 保存完整特征向量到磁盘
```yaml
# 服务器特征快照配置
live_feature_snapshot:
  enabled: true
  storage_path: "data/live_feature_snapshots/"  # 保存路径
  format: parquet                                # 格式
  retention_days: 30                             # 只保留最近 30 天
  # 每天一个文件: features_2026-02-22.parquet
  # 列: timestamp, symbol, strategy, + 所有特征列
  # 估算大小: 6 symbols × 24 bars/day × 200 features × 30 days ≈ 50MB
```

**本地验证流程**:
```bash
# 1. 从服务器下载特征快照
scp server:live/highcap/data/live_feature_snapshots/features_2026-02-*.parquet data/live_snapshots/

# 2. 本地用相同行情数据 batch 重算
mlbot train final --no-docker --prepare-only \
  --config config/strategies/me \
  --features config/strategies/me/features_gate.yaml \
  --labels config/strategies/me/labels_rr_extreme.yaml \
  --symbol BTCUSDT,ETHUSDT,BNBUSDT,SOLUSDT,XRPUSDT,ADAUSDT \
  --timeframe 60T --data-path data/parquet_data \
  --start-date 2026-01-22 --end-date 2026-02-22 \
  --holdout-start-date 2026-01-22 --holdout-end-date 2026-02-22 --seed 42

# 3. 逐特征对比 (待实现: scripts/compare_live_vs_batch_features.py)
python scripts/compare_live_vs_batch_features.py \
  --live data/live_snapshots/ \
  --batch results/train_final_<timestamp>/me/features_labeled.parquet \
  --output reports/feature_consistency_report.html
# 输出: 每个特征的 max_abs_diff, mean_abs_diff, NaN 不一致比例
# 重点关注: diff > 1% 的特征列表 + 时间分布（是持续偏差还是偶发）
```

### L1: Feature Store（特征计算层）

| 训练假设 | 验证方法 | 失效阈值 | 对应 Step |
|---------|---------|---------|----------|
| 特征分布稳定，无数据源异常 | 本地下载最新行情 → 重跑 feature store → 对比新数据 vs 训练期的特征分布 (PSI) | PSI > 0.25 连续 2 周 | Step 1 |
| 零值/NaN 比例正常 | 统计新数据各特征缺失率 vs 训练期缺失率 | 缺失率偏离训练期 > 2x | Step 1 |

**验证方式**: 纯本地操作，不依赖实盘系统。
```bash
# 1. 下载最新行情数据 (或用已有的 data/parquet_data/)

# 4. 对比新导出的 features_labeled.parquet vs 训练期的 features_labeled.parquet

```

### L2: Prefilter（环境前置条件层）

| 训练假设 | 验证方法 | 失效阈值 | 对应 Step |
|---------|---------|---------|----------|
| Archetype 特征区分力存在 (bad_rate 差异 > 5%) | 用最新数据重跑分位数分层 | bad_rate 差异 < 3% | Step 3 |
| 时间稳定性 (temporal CV < 0.5) | 重跑 `--select-recent 6` 的 rolling 验证 | CV 从 < 0.3 恶化到 > 0.7 | Step 3 |
| Prefilter 通过率在合理范围 | 统计最近数据的 prefilter pass rate | 偏离训练期 ±50% | Step 3 |

**验证命令模板**:
```bash
# 用最新数据重跑 prefilter 分层分析
python scripts/analyze_archetype_feature_stratification.py \
  --logs data/live_features_labeled_latest.parquet \
  --strategy me --select-recent 6
```

### L3: Direction（方向判断层）

| 训练假设 | 验证方法 | 失效阈值 | 对应 Step |
|---------|---------|---------|----------|
| 覆盖率 100%，每个 bar 都有方向 | 统计实盘方向分配覆盖率 | < 99% | Step 4 |
| Short 占比 > 15% | 统计方向分布 | short < 10% | Step 4 |
| 方向质量：median(rr_in_direction) > median(rr_always_long) | 重跑 direction validation | Median Lift < 0 连续 2 个月 | Step 4 |
| 方向特征未衰减 | `--compare-features --temporal` | 近 3 窗口 >= 2 个负 Lift | Step 4 |

**验证命令模板**:
```bash
# 重跑 direction 验证（用最新数据）
python z实验_005_统一研究/direction_strict_validation.py \
  --logs data/live_features_labeled_latest.parquet \
  --strategy me --compare-features --temporal
```

### L4: Gate（结构性过滤层）

| 训练假设 | 验证方法 | 失效阈值 | 对应 Step |
|---------|---------|---------|----------|
| Gate 有正向 LIFT（降低 failure rate） | 重算 gate pass 组 vs reject 组的 bad rate | LIFT < 1.0 (gate 无正贡献) | Step 5 |
| Gate 通过率在合理范围 (训练期的 ±30%) | 统计最近 gate pass rate | 偏离训练期 > 30% | Step 5 |
| Bad Rejection Rate 合理 | 统计 gate 拒绝的坏样本比例 | 下降 > 50% (gate 变弱) | Step 5 |
| Good Retention Rate 稳定 | 统计 gate 保留的好样本比例 | 下降 > 20% (gate 过度拦截) | Step 5 |

**关键 KPI 对照（训练基线 vs 最新）**:
```
训练期基线 (来自 gate_optimization.html):
  LIFT: +0.06%
  Pass Rate: 84.6%
  Bad Rejection Rate: 15.5%
  Good Retention Rate: 84.6%

监控: 每月对比，偏离 > 30% 则告警
```

---

### L5: Evidence（信号强度层）

| 训练假设 | 验证方法 | 失效阈值 | 对应 Step |
|---------|---------|---------|----------|
| 高 evidence score → 高 R | Spearman 相关 evidence_score vs actual_return | Spearman < 0.03 | Step 6 |
| Evidence 分位数阈值仍然有效 | 重跑 plateau 分析 | 最优 quantile 偏移 > 15 百分点 | Step 6 |
| Tail loss reduction 正常 | 重算有/无 evidence 的 max DD 差异 | reduction < 5% | Step 6 |

---

### L6: Entry Filter（入场过滤层）

| 训练假设 | 验证方法 | 失效阈值 | 对应 Step |
|---------|---------|---------|----------|
| Entry filter 有正向 lift | 重算 filter pass vs reject 的 avg return | lift < 1.0 | Step 7 |
| Entry filter 不过度拦截 | 统计 filter pass rate | pass rate < 50% 训练期值 | Step 7 |

---

### L7: Execution（执行参数层）

| 训练假设 | 验证方法 | 失效阈值 | 对应 Step |
|---------|---------|---------|----------|
| 高 tier 表现 > 低 tier | 按 tier 分层统计 mean R | T1 ≤ T3 连续 2 周 | Step 8 |
| 止损距离合理 | 统计 SL hit rate by tier | SL hit rate 偏离训练期 > 20% | Step 8 |

---

### L8: PCM（组合仲裁层）

| 训练假设 | 验证方法 | 失效阈值 | 对应 Step |
|---------|---------|---------|----------|
| 被选信号 > 被拒信号 | 反事实对比：selected avg R vs rejected avg R | rejected avg R > selected 持续 2 周 | Step 9 |
| 策略间不冲突 | 统计同时触发的频率和方向 | 冲突率 > 30% | Step 9 |

---

### L9: 宪法层

| 训练假设 | 验证方法 | 失效阈值 |
|---------|---------|---------|
| Kill switch 阈值合理 | 统计触发频率 | 月触发 > 3 次需审查 |
| DD 上限不被频繁触碰 | 统计 drawdown 分位数 | P90 > 0.6 × max_dd (0.20) |

## 本地研究监控：执行流程

### 周频快速检查（~30 分钟）

```bash
# 1. 从服务器下载最近数据
scp server:live/highcap/data/features_recent.parquet data/live_latest.parquet

# 2. 快速三层检查
python scripts/local_monitor_weekly.py \
  --data data/live_latest.parquet \
  --strategy me \
  --checks prefilter,direction,gate_pass_rate \
  --baseline results/train_final_<timestamp>/me/

# 输出: reports/weekly_health_check.html
# 结论: 🟢 假设成立 / 🟡 需关注 / 🔴 需重训
```

### 月频全层报告（~2 小时）

```bash
# 1. 下载完整月度数据
scp server:live/highcap/data/monthly_dump.parquet data/monthly.parquet

# 2. 全层验证 (L1-L9)
python scripts/local_monitor_monthly.py \
  --data data/monthly.parquet \
  --strategy me \
  --baseline-dir results/train_final_<timestamp>/me/ \
  --output reports/monthly_full_report.html

# 输出:
#   reports/monthly_full_report.html   ← 9 层逐项检查
#   reports/drift_heatmap.png          ← 特征漂移热力图
#   reports/kpi_trend.png              ← 各层 KPI 趋势线
```

### 重训决策树

```
任何一层红灯？
  ├─ L1 (特征漂移) → 检查数据源，不需要重训
  ├─ L2 (Prefilter) → 重跑 Step 3，更新 prefilter.yaml
  ├─ L3 (Direction) → 重跑 Step 4，评估 direction.yaml
  ├─ L4 (Gate) → 重训 Step 5 (Gate 模型)
  ├─ L5 (Evidence) → 重训 Step 6 (Evidence 模型)
  ├─ L6 (Entry Filter) → 重跑 Step 7
  ├─ L7 (Execution) → 重跑 Step 8
  ├─ L8 (PCM) → 检查策略间平衡
  └─ L9 (宪法) → 人工审查阈值
```

# Part B: 实盘监控（服务器端）

## 设计原则

- **极轻量**: 纯内存统计 + SQLite 落盘，不依赖外部数据库
- **只保留**: 当前实时状态 + 最近 1 个月的 15min 聚合历史
- **两个职责**: 1) 风控安全 2) 心理安抚

---

## B1: 风险控制监控（实时）

### 核心灯号：系统存活状态

```yaml
# 实时检查，每个 bar 决策前强制执行 (已有实现: constitution_executor.py)
risk_monitor:
  # ── 第一层：资本红线 (宪法层，已实现) ──
  capital_integrity:
    daily_loss: 0.04           # -4% → 日内停止 (constitution.yaml)
    weekly_loss: 0.08          # -8% → 周内停止
    monthly_loss: 0.12         # -12% → 月内停止
    max_dd: 0.20               # -20% → 系统熔断
    cooldown_minutes: 240      # 触发后冷却 4 小时

  # ── 第二层：持仓风险 (新增) ──
  position_risk:
    single_position_dd: 0.03   # 单仓回撤 > 3% → 预警
    unrealized_loss_total: 0.06 # 总浮亏 > 6% → 降杠杆
    
  # ── 第三层：Archetype 连续亏损 (新增) ──
  archetype_health:
    max_consecutive_losses: 5  # 单个 archetype 连续亏损 5 次
    action: pause_archetype    # → 暂停该 archetype (非全局停机)
    review_after_hours: 24     # 24 小时后可人工恢复
    
  # ── 第四层：异常检测 (新增) ──
  anomaly:
    order_execution_fail_rate: 0.3  # 下单失败率 > 30% → 告警
    data_gap_minutes: 5             # 数据中断 > 5 分钟 → 告警
    ws_reconnect_count: 10          # 1 小时内重连 > 10 次 → 告警
```

### 告警通道

```
严重级别:
  🔴 CRITICAL → 立即通知 (Telegram/微信)
     - kill_switch 触发
     - max_dd 突破
     - 数据源完全断开
     
  🟡 WARNING → 延迟通知 (每 15 分钟汇总)
     - archetype 连续亏损
     - 单仓回撤预警
     - 订单执行失败
     
  🟢 INFO → 仅记录日志
     - 正常开平仓
     - 正常 gate veto
```

## 两部分联动机制

```
实盘监控发现异常
    │
    ├─ 🔴 风控类 (B1)
    │   └─ 宪法自动处理 → 停机/降杠杆 → 事后通知人工
    │
    ├─ 🟡 活跃度类 (B3 Activity 黄灯)
    │   └─ 触发本地快速检查 → 下载最近数据 → 跑 L2-L4 验证
    │   └─ 确认: gate 拦截率是否偏高？prefilter 是否过严？
    │
    └─ 🔴 活跃度类 (B3 Activity 红灯)
        └─ 触发本地全层验证 → 跑 L1-L9 完整报告
        └─ 决策: 重训？调参？还是市场本身不适配？
```

## Archetype 级降级控制（与账户级 Kill Switch 共存）

> **设计理念**: LV 连亏 3 笔 → 只下线 LV，不影响 BPC/ME/FER。
> 账户级 kill_switch 是最后防线，archetype 级降级是精细化控制。

### 两层风控共存

```
第一层: Archetype 级降级 (新增)
  ├─ 作用范围: 单个策略/archetype
  ├─ 触发条件: 连续亏损 N 笔 (可按策略配置)
  ├─ 动作: pause_archetype → 该策略不再产生信号
  ├─ 恢复: 人工审核 + 本地验证通过后手动恢复
  └─ 其他策略不受影响

第二层: 账户级 Kill Switch (已有, constitution.yaml)
  ├─ 作用范围: 全账户
  ├─ 触发条件: daily -4% / weekly -8% / monthly -12% / max_dd 20%
  ├─ 动作: 全局停止所有新仓
  └─ 这是不可违反的底线
```

### Archetype 降级配置

```yaml
# 建议加入 constitution.yaml 或独立的 archetype_health.yaml
archetype_degradation:
  enabled: true
  
  # 全局默认
  default:
    max_consecutive_losses: 5    # 连亏 5 笔 → 暂停
    review_after_hours: 24       # 24h 后可人工恢复
    
  # 按策略覆盖 (不同策略风险特征不同)
  per_strategy:
    lv:
      max_consecutive_losses: 3  # LV 波动大，连亏 3 笔就该停
    me:
      max_consecutive_losses: 5  # ME alpha 在盈亏比，容忍更多连亏
    bpc:
      max_consecutive_losses: 5  # BPC 类似 ME
    fer:
      max_consecutive_losses: 4  # FER 中等

  # 降级动作
  actions:
    pause_archetype:             # 暂停该策略
      - 不再产生新信号
      - 已有持仓正常管理（不强平）
      - 发送 WARNING 通知
    
  # 恢复条件
  recovery:
    manual_only: true            # 只能人工恢复，不自动恢复
    require_local_verification: true  # 恢复前建议本地跑一次 L2-L4 检查
```

## 补充建议

### 1. 影子模式统计（Shadow Mode）

在实盘 15min 快照中额外记录 **"如果没有 gate/evidence 会怎样"** 的反事实统计：

```yaml
shadow_stats:
  # gate 拒绝的信号，记录后续 bar 的走势
  gate_rejected_hypothetical_return: -0.003  # 被 gate 拦截信号的假设 PnL
  gate_passed_actual_return: 0.005           # gate 通过信号的实际 PnL
  # → 差值 > 0 说明 gate 在持续发挥作用
```

这个不需要额外资源，只需在 gate reject 时记录当时的特征快照，
15 分钟后回填 forward return 即可。它能直接回答"gate 到底在不在帮忙"。

### 2. 数据导出脚本（服务器 → 本地）

建议在服务器部署一个轻量导出脚本，每周自动打包：

```bash
# live/scripts/export_for_local_monitor.sh
# 导出最近 30 天的 features + labels + decisions
# 压缩后约 50-100MB，方便本地下载分析
sqlite3 data/live_monitor.db ".dump stats_15min" > /tmp/stats_dump.sql
tar czf /tmp/local_monitor_data.tar.gz \
  data/live_features_recent.parquet \
  data/live_decisions_recent.parquet \
  /tmp/stats_dump.sql
```

### 3. 异常日志分层（避免信息过载）

```
日志文件分离:
  logs/live_critical.log   ← 只有 kill_switch、异常、错误
  logs/live_decisions.log  ← 每次决策的简要记录
  logs/live_debug.log      ← 详细特征值（默认关闭，排查时开启）
```

### 4. 数据导出一致性

线上导出的 15min 快照 + 特征快照两类数据，下载后本地脚本需要能自动对齐时间戳。
建议统一用 UTC 时间，parquet 文件名带日期。
